sum / product / average

Definition: atom res = sum(object x, zlr=0)
-- or --
atom res = product(object x, zlr=1)
-- or --
atom res = average(object x, zlr=0)
-- or --
atom res = std_dev(object x, zlr=0)
Description: Add or multiply together all elements of x, however deeply nested,
or calculate their average or standard deviation.

x: atom or sequence of items to add or multiply together
zlr: atom or length 1 sequence, the overrideable zero length result for {}.
pwa/p2js: Supported.
Comments: These functions may be applied to an atom or all elements of a sequence.
Note that unlike most of the other math functions there are no separate sq_sum, sq_product, or sq_average routines.

The zlr (zero length result) parameter is only used when x is {}, an empty sequence, or contains said.

You might, for instance, rely on sum({},1) to award a bonus point for "no faults", or product({},0) to order zero tons of concrete when there is no foundation, instead of the mathematically correct defaults which are the other way around.

If zlr is a sequence containing a single atom, it applies to any (empty) nested subsequences,
whereas if zlr is an atom it is applied at the top-level only, and recursive calls get standard 0/1 defaults.
A fatal error occurs if zlr is not an atom or a length 1 sequence containing a single atom.

Note that average() uses the proper recursive application rules, so average({{1,3},6}) === average({average({1,3}),6}) === 4, it is not the same as sum(x)/length(x) (=== 5) or sum([flatten]x)/length(flatten(x)) (=== 3.33), except when x is a flat sequence. Likewise std_dev() quietly replaces any nested sub-sequences with their (single) average before proceeding, and you are expected to do something else (maybe crash) when x!=flatten(x), should that be needed. What I'm saying is it [only] resorts to recursion when it encounters nested sequence elements, and in contrast sum/product do give the same results [when zlr is the default] for any x that yield the same flattened x.

See also demo\rosetta\Standard_deviation.exw. No attempt is made to apply Bessel's correction.

Aside: the sum of consecutive integers can be calculated as (first+last)*length/2. This works for odd/even length, including length=1 but not {}, the /2 being ok(/always delivering a whole integer result as expected) since either (f+l) or length is even [but not/never both], and ascending or descending sequences, with +/- start/finish, including any that cross the zero boundary. For example sum(tagset(4)) == (1+4)*4/2 == 10, and of course the formula is much faster for very long sequences. Obviously equally applies to average(<flat sequence of consecutive integers>).
For products of consecutive positive integers, the k_perm() routine might be helpful, any that include 0 produce a 0 result, and all-negative lists are the positive result (ensuring k<=n) times -1 when length odd. However there is far less scope for any performance gain over product(), since any length 19/21 (for 32/64 bit) will breach native accuracy limits - that is, when working on a list of strictly consecutive integers.
Example: 
atom s = sum({1,2,3,4}),      -- s is 10
     p = product({1,2,3,4}),  -- p is 24
     a = average({1,2,3,4})   -- a is 2.5
Implementation: See builtins\psum.e (an autoinclude) for details of the actual implementation.
