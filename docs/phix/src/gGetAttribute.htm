gGetAttribute

Definition: include theGUI.e

object res = gGetAttribute(gcx id, string name, object dflt=999_999_999, integer nFrames=1) )
Description: Retrieve an interface element attribute value.

id: Identifier of the interface element.
name: name of the attribute.
dlft: value to return if not found or undefined, with 999_999_999 specifically/randomly meaning "crash".
nFrames: Fine grained control for error handling, intended for internal use.

Returns: the attribute value or dlft if the attribute is not set or does not exist.
pwa/p2js: Supported.
Notes: While gSetAttribute() can be passed settings in either the most natural form or a suitable string representation, this routine returns the internal representation, and there is no way to force it to yield a string. Note that the internal form may be platform or back-end specific, for instance the red colour under GTK is (I think) the atom #FF0000 whereas under JavaScript it might be "#FF0000", or "red", or even "rgb(255,0,0)". In fact, red in WinAPI is actually #0000FF, but theGUI.e expects/returns all colours in rgb form, and quietly maps them to/from the offbeat Windows bgr form for you. Obviously I can and will take pains to address the most common vexing issues, such as the colours just mentioned, and may in time be able to start guaranteeing some consistency between backends, but the more obscure the setting you are retrieving, the more defensive you should probably code.

Also, gSetAttribute() is obviously and for good reason downright belligerent in pointing out unsupported attributes, whereas gGetAttribute() is somewhere between moderately aggressive in that regard and quite wrong in flagging something as an error simply because it hasn't yet been set. There will doubtless be several such cases that should undeniably be treated as a straightforward bug (jlmk). [One specific case that was explicitly fixed for is gGetAttribute(id,"ACTIVE")]
Simply providing a suitable default can sometimes go a long way to alleviating much of that.

When an attribute is marked as inherited (just ACTIVE, BGCLR (container-only), FONT, and VISIBLE), then obviously it is imperative the control is part of the completed layout hierarchy before any attempt is made to read it, otherwise the fetch operation will(/may) crash as it tries to scramble up a tree that isn't there yet.
Aliases: The following aliases can be used for a clearer statement of intent and to make the code a little more self-documenting:

integer res = gGetInt(gcx id, string name, object dflt=999_999_999)
integer {a,b} = gGetIntInt(gcx id, string name, object dflt={0,0})
atom res = gGetDouble(gdx id, string name, object dflt=999_999_999)

With the exceptions of EXPAND, SPACE, one case of VALUE, and SELMODE, for which the return type depends on whether gGetInt() or gGetAttribute() was called, and gGetIntInt() quietly returning {0,0} should there be a typo in name, they don't guarantee or do anything differently, apart from the lightest validation of the result. When (for instance) asked for the "VIEWPORT", gGetIntInt() (only) validates the first two results are integer, and length(res)>=2, before returning all [in that specific case four]. Even something a litte bit strange like gGetInt(id,"TITLE") would be fine as long as to_number() does not object.
Combos: object prev = gGetSetAttribute(gcx id, string name, object v, dflt=999_999_999)

Trivial yet handy routine for combining the two, especially useful when temporarily overriding an attribute.
For boolean attributes use gToggleInt(id,name) instead of gSetInt(id,name,not gGetInt(id,name)).
Text: integer {w,h} = gGetTextExtent(sequence text, object tgt=TG_STD_FONT, bool bSumHeights=true, bNorm=false)
integer {w,h} = gGetTextExtent32(sequence utf32, object tgt=TG_STD_FONT, bool bNorm=true)    <- note diff -^

DEV not yet properly tested since the latest complete redesign...
text:  a single utf8 string, optionally with embedded newlines, or sequence of them, returning max width.
utf32:  one fragment from utf8_to_utf32(), no new/multiple lines, as used in eg demo\theGUI\hText.exw.
tgt:  any gdx control, a gImage, or a FONT string (the private constant TG_STD_FONT is "Helvetica, 9", btw).
For controls and images, uses the current font settings from that, with a gdx including any inheritance
bSumHeights:  determines whether multiple line heights are summed or just the last returned.
bNorm:  by default (eg) "&www" is displayed treated as www but as-is when bNorm is true, eg in theEditor.

When tgt is a FONT string, or a dialog element that has not yet been displayed, these routines may create a private mini-cache of [tiny] gImages to use instead, each with the appropriate FONT setting already applied.
The utf32 version is probably only useful deep in the inner workings of a gText(), but I see no reason to hide it.
See the source code of gDrawText() for an example of using gRotatePolygon() on the results.
Common: The following (read-only) attributes are common to several controls/interface element types.
See gSetAttribute for other (updateable) common attributes.
CLASSNAME (read-only) Returns a human readable string for diagnostic purposes.
Javascript also has elem.className, were you looking for that.
MOUSEPOS (read-only) Returns the cursor position in client coordinates, {0..w-1,0..h-1}, where w,h are as per SIZE.
Standard dflt handling applies should the control not be VISIBLE.
The global version of MOUSEPOS returns absolute screen coordinates.

Returns: {x,y}, a pair of integer pixel offsets.
SIZE Specifies the element user size, and returns the current/final size, in pixels.

Note that technically this is read-only and attempts to set it actually set USER_SIZE, whereas fetching matches reality. In other words the layout manager really owns and calculates this attribute, but it accepts hints.

Most applications should just do everything via SIZE, and pretend that USER_SIZE does not even exist.

Value: {width,height} or "widthxheight", where width and height are integer values corresponding to the horizontal and vertical sizes, respectively.

You can also set only one of the parameters by removing the other one and maintaining the separator "x", which is equivalent to setting the other value to 0, or omit the x to set the width only, ditto an integer or length-1 sequence.
For example: "x40" (height only = "0x40" - ditto {0,40}) or "40x" (width only = "40x0" - ditto "40", 40, or {40}).

When consulted the current/final size of the control is returned, not necessarily what was actually set.
(In some cases a gRedraw() may be required between setting and fetching for updates to filter through.)

A user size of {0,0} or "0x0" can be set, which can also be set using NULL.

The element is NOT immediately repositioned. Call gRedraw() to update the dialog layout.

See Layout Management for more details on sizes.

Note this replaces the RASTERSIZE attribute of pGUI, and theGUI_from_pGUI.e replaces "SIZE" with "SIZE[*charsize]" which needs manual fixup. (In IUP, SIZE is/was in "character units", which theGUI does not support, specifically since it makes no sense on containers, whose children might very well have different fonts set on them.)

Affects: All, except menus (since on them anything you tried to set would be auto-clobbered just before display).

See Also: FONT, USER_SIZE, gNormalise, gRedraw, Layout Management
There is also a size() function in map.e, were you looking for that.
