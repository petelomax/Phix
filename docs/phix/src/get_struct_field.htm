get_struct_field

Definition: include cffi.e
atom v = get_struct_field(integer id, atom pStruct, atom_string field, bool bAsFlt=false)
-- or --
string s = get_struct_string(integer id, atom pStruct, string field, integer len)
Description: get a named field from some previously allocated structure, which (ie pStruct) is most likely to be the result of an earlier call to allocate_struct().

id: a result from define_struct().
pStruct: a result from allocate_struct().
field: a string name or, caveat emptor, a numeric index.
bAsFlt: fudge, see notes
 len: length in bytes, lesser fudge, see notes
pwa/p2js: Not supported.
Comments: The get_struct_string() routine should (only) be used when the field is TCHAR[] or similar. Note that it is your responsibility to pass in the right length, since cffi.e cannot check that for you, and likewise trim trailing zeroes (or '\0') or spaces where needed. Should some unknown C code replace say "some long message" with "error 7\0g message", that's your problem not mine.
Under set_unicode(1), the retrieved widestring is converted to utf8 before being returned. Also, a TCHAR[128] should still get a len of 128 (characters/16-bit unicode points), even though it occupies 256 bytes. (no longer entirely sure about that last statement...)
In truth, there should perhaps be a default for len, say -1, which means "read the lot" [and crop at first '\0', perhaps], then again it is a bit of a stetch to expect get_struct_string to handle the 8/16 bits per char aspects perfectly in all circumstances, plus the vast majority of C based string interfaces explicitly yield the number of bytes or surrogate pairs they have just written to the buffer, and/or have an explict length field in the struct next to each string/tchar, and whenever you have that you should always explicitly pass it along anyway. (There is absolutely no sensible way for cffi to guess which length field corresponds to which string/tchar, of course.)
Far less excusable, by default get_stuct_field treats everything as an integer except for a rare few obvious ones, currently just {double,gdouble} but the rest (when needed, esp single-precision floats) must be explicitly overridden via bAsFlt.
That said, all use of these routines merits thorough testing, and is usually buried away from day-to-day use in a wrappper-lib.

You may need to use say peek_[w]string() or even get_struct_field() itself again, on some atom results, eg an lpszText.
Example: 
include cffi.e
constant idMBP = define_struct(...)
atom pMBP = allocate_struct(idMBP)
.. <set contents of pMBP>
atom pTitle = get_struct_field(idMBP,pMBP,"lpszText")
?peek_string(pTitle)
See Also: allocate_struct, set_struct_field, peek_string
