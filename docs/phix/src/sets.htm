sets.e

The file builtins\sets.e (an autoinclude) contains routines for simple handling of sets.

A set is an ordered1 sequence of items of any type (including mixed types) such that unique() would not alter it and binary_search() can locate/insert/delete items efficiently. It can be a standard hll sequence or string, or a sets.e-managed collection with an atom reference as generated by new_set(), henceforth known on the rest of this page as a set_id. The latter automatically delegates all js-compatibility issues (aka p2js violations) to the sets.e include file, which is one less thing for the programmer to worry about, however it can perhaps make debugging a little bit trickier, or be a shade "less natural" to initially setup. You can freely mix and match set_ids and application-owned sequences/strings (eg add_members(char_set,"aeiou"), and even provide a sequence (not just to new_set) yet get a new/existing set_id result.

(procedure) 
set_management(string how/*="NORMAL"*/, bool bCheckOrder=true)

(optional) in almost all cases the default setting should suffice, ... (expand for more details)
and you would only fine-tune when you think that some fatal crashes might actually help.

One exception is that should you need to create 5000 sets it might be noticeably faster to reclaim them manually, then again in all probability so would not using sets.e since it is a generic solution to make things a bit easier, not necessarily or intentionally faster.

By default the internal type checking hammers down app-owned sequences/strings checking they are in order with no duplicates, but that can be switched off should it become too expensive, once you're confident modest-sized app-owned sets aren't getting mangled.

The how parameter, aka set management options, can be:
"NORMAL": sets can be sequences/strings or set_ids, latter reclaimed as they drop out of scope.
"ATOM": set_ids only, auto-reclaimed (set_id is int idx + 0.5 with delete_routine).
"ATOMSTR": "" and strings.
"ATOMSEQ": "" but sequences.
"INTEGER": set_ids only, not auto-reclaimed.
"INTSTR": "" and strings.
"INTSEQ": "" but sequences.
"NOTAUTO": sets can be sequences/strings or set_ids, latter not auto-reclaimed.
"STR": strings only.
"SEQ": sequences only.
"SEQSTR": sequences/strings only.

The distinction between seq and str exists in the hope it may help, not because it actually matters.

Note this setting is application-wide, individual blocks of code can however be wrapped in `string prev = get_management(); set(for_me)` and `set(prev)` so as to operate as needed without accidentally upsetting any other part of the application. To be crystal clear, the internal set management options default to "NORMAL", but the how parameter on the set routine has no default (why it be tiny) and must always be provided. No attempt has been made to make sets.e thread-safe, so you may also want to wrap/protect similar code blocks in a critical section.
If preferred, get_set_management() is the same as/aliased to get_management().
This is a classic case of over-engineering an insignificant component, in truth I grossly overestimated how helpful validation might be when in practice pretty pointless, but it's done now (though maybe I /could/ scan the include file tree to determine which prior setting best applies to subsequent calls...! A joke btw.).

int/atom res = 
new_set(object items={})

Creates a new sets.e-managed set and returns a set_id.
items can be a sequence/string or some previous set_id.
Note that set_management options do not limit what items can be.
Returns an atom if set management options are one of the first four (inc/default),
an integer if one of the next four (so discard_set() is your responsibility), or
raises an error if one of the last three. It is not uncommon for items to require a deep_copy(), but pointless when an integer and quite wrong when an atom set_id.

(procedure) 
discard_set(object s)

(optional, done automatically by default) Free up slots in the internal tables when no longer needed.
Note that if s is a sequence it is assumed to be a collection of set_ids (and not an app-managed set).

bool res = 
is_member(object s, x) -- returns true if x is a member of s

Symbolically represented with an E shape: ∈ (&isin;) and its opposite as that struckthrough: ∉ (&notin;)

(note) 
<[object res =] >:  In the next five routines, an atom s is returned unaltered and can safely2 be explicitly ignored.
At least, that would be true on desktop/Phix, but such explicit ignores would not work in a browser2.
When with js is not in force, the compiler automatically maps procedural invocations of these next 5(+2) routines to set_ignore_atom(<next_five>(...)) which asserts the implicitly ignored result is indeed an atom.
For app-managed sequences, the result should ideally be assigned back to the same local variable to take full advantage of automatic pbr.

[object res =] 
add_member(object s, x) -- adds x to s if not already present

There is also a trivial add_members(object s, sequence items) to add multiple items in one call.

[object res =] 
remove_member(object s, x) -- removes x from s if present

There is also a trivial remove_members(object s, sequence items) to remove multiple items.

(note) 
<object s1, s2=-1, atom tgt=0>:
In the next two routines, if s2 is not -1 it is assumed there are precisely two sets, otherwise each element of s1 is treated as a set_id or (app-owned) sequence/string. Both can be invoked as fn(s1,s2) or fn({s1,s2}), where the second form permits more than two/any number of sets. The first line is always if s2!=-1 then s1 = {s1,s2}, and it then proceeds by checking whether each element of the resulting s1 is a set_id or an app-owned sequence/string. If tgt is 0 (the default) a sequence/string is returned, if -1 a new set_id is created and returned, otherwise when equal to s1[1] a slightly more efficient in-situ update on that occurs, and (whether or not in-situ happens) that set_id is updated and returned. The note regarding [ignore/]pbr s from just above still applies, but to [a +ve tgt or] (said resulting) s1[1] (not s2). Lastly, set_ignore_atom() also prevents tgt=-1 results being implicitly ignored, when obviously they shouldn't.

[object res =] 
union(object s1, s2=-1, atom tgt=0)

Returns anything that occurs in any set.
Symbolically represented with a U shape: ∪ (&cup;)
The union of empty sets is the empty set, as is the union of an empty set of sets. Otherwise empty sets have no impact on the result when in the presence of other non-empty sets.

[object res =] 
intersection(object s1, s2=-1, atom tgt=0)

Returns anything that occurs in every set. Aliased to intersect() should you prefer that.
Symbolically represented as upsidedown U: ∩ (&cap;)
The intersection of empty sets is the empty set, as is the intersection of an empty set of sets, or a set of just one set. The presence of any empty set means that the result will also be the empty set.

[object res =] 
difference(object s1, s2, bool symmetric=true, atom tgt=0)

Returns items missing from the other set. Unlike intersect/union, always requires precisely two sets.
If symmetric is false only items from s1 not present in s2 are returned, otherwise
the result also contains items in s2 that are missing from s1. Symbolically represented as:
symmetric: a triangle (Δ, sometimes ∇) or even +, but best as a circle with a horizontal line (⊖),
relative: a backslash (∖) (actually &setminus;, and above &ominus;)
difference(s1,s2,true) is the same as union(difference(s1,s2,false),difference(s2,s1,false)),
aka A⊖B = A∖B ∪ B∖A, where A∖B = {x: x ∈ A and x ∉ B}

Internally these routines use a private set_type, but an application should use integer/atom/sequence/string (or even object) as appropriate. (oh yeah, that's <anything> innit!) Should you really want a user defined type to help debugging, then your best bet is probably to carefully validate all the elements of get_members(s,false), though that would (catchably) crash rather than return false should s be an incompatible atom - which might be helped by checking for +ve ints and/or atoms with an exactly .5 fraction that still_has_delete_routine(). Far better yet, store your sets in an explicitly tagged form, such as {"this_really_is_a_set",s} and if needed have your own udt that validates that containing shell. While I could yet be persuaded to grant some form (not actually a type) of improved access to that internal, doing so fundamentally strikes me as a poor strategy in terms of actually being helpful, for instance you could easily check for atoms where expected, which said private type won't, plus you could easily have multiple tags such as "set of vertices", "set of edges", etc. In fact, now that I think of it, you would be much better off validating the items that go in and out of sets rather than the sets themselves.

1 I have seen many people strenuously argue that sets should not be ordered, to which I say this: I may be missing some other performance advantage, but mathematically speaking, changing the order of a set does not alter it, therefore should I choose to sort it for my own personal performance reasons, you should shut up. I will however sit up and listen dare you provide a significantly faster alternative which passes all tests with flying colours.

These routines [should] all work just fine under pwa/p2js, aka `with js`, although deep_copy() may be required fairly often when res!=s1 and/or new_set() is not being used (desktop/Phix tells you when and where). NB not yet fully tested. JavaScript does not support delete_routine(), so the (rather sneaky) plan is that (in a browser) sets.e does not actually return set_ids or have an internal table (which desktop/Phix absolutely needs, to sidestep its cow-semantics) and instead relies on the JavaScript pass-by-reference semantics. All a bit naughty I suppose, but not entirely unlike the way theGUI has happily been returning HtmlElements instead of integers, and no-one should really care. Likewise discard_set and bDestroy don't actually do anything under js, it relies on the JavaScript garbage collector instead.
2 One thing it does mean however is that implicit ignores simply will not work, hence set_ignore_atom() simply crashes. The compiler does not auto-map procedural calls to to that on desktop/Phix when with js is in force, and there is no good reason to ever invoke it explicitly, hence you should never actually see that crash. In short, set_ignore_atom() allows slightly neater desktop-only programs, but you have to do the seemingly pointless set_id := set_id for js compatibility, and should never explicitly ignore set_id results.
