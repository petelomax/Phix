gDropDown

Definition: include theGUI.e

gdx id = gDropDown([sequence options={},] [rtn value_changed=NULL,] string attributes="" dword_seq args={})
Description: Creates an interface element that is a simple text-only dropdown selector.

options: a sequence of text strings to populate the dropdown. It can be empty, sets the OPTIONS attribute.
value_changed: a procedure to invoke when the selection is changed, see VALUE_CHANGED below.
For more information on the attributes and args parameters see gSetAttributes().
This is a paranormalised function (see technicalia).

Returns the identifier of the created element.
pwa/p2js: Supported.
See Also: gButton, gCheckbox, gRadio, gLabel, gList
Example: 
-- demo\theGUI\gDropDown.exw
include theGUI.e

procedure changed(gdx id)
    ?{"changed",gGetAttribute(id,"VALINT"),
                gGetAttribute(id,"VALSTR")}
end procedute

constant dd123 = {"one","two","three"}
gdx drop1 = gDropDown(dd123,changed),
    drop2 = gDropDown(dd123,changed,"VALUE=3"),
    hbox = gHbox({drop1,drop2},"MARGIN=13x13"),
    dlg = gDialog(hbox,"gDropDown",`SIZE=240x80`)
gMainLoop(dlg)
Attributes:
EDITABLE NB broken on WinAPI - see demo\theGUI\hText.exw for progress on a canvas-based drop-in replacement.
(creation only) Specifies whether the user can type in a new value (true/"YES") or only select one of the given options (the default, false/"NO"). Note that when true there is no implicit validation or checking whether the entered value is or is not one of the specified options, and no facility for automatically "adding" the entered item, instead that would have to be done explicitly (by replacing the lot).
OPTIONS A list of text strings to populate the dropdown. Same as the first parameter on creation.
Can also be set later to purge the dropdown and re-populate it with an entirely different set of options.
VALUE Get/Set the selected item. gGetInt() returns an integer, 0 if no entry is selected, whereas gGetAttribute() returns a string or the length-0 "" if no entry is selected. Note that the dflt param is not actually used at all when fetching this particular attribute on a gDropDown, ever.

There is however a potential gotcha: should options be {"1","3","9"} then gSetAttributes(dropdown,"VALUE=3") could be interpreted as "3" (the second entry) or as 3 (the "9" or third entry). The near-aliases "VALINT" and "VALSTR" can be used to resolve any ambiguity, but when no such possible confusion exists you should be able to (and can) just use "VALUE" with either integers or strings and let theGUI figure it all out for you in a common sense fashion.
also ACTIVE, CANFOCUS, FONT, EXPAND, TIP, SIZE, VISIBLE.
Handlers: Set using eg gSetHandler(dropdown,"VALUE_CHANGED",value_changed), or specify value_changed at creation time
VALUE_CHANGED Event generated when the selection or entry is changed.

procedure value_changed([gdx id, ]string/integer v)
id: (optional) identifier of the element that activated the event.
v: equivalent to 
or 
string v = gGetAttribute(id,`VALUE`)
integer v = gGetInt(id,`VALUE`).
also KEY: All common handlers are supported.
Technicalia As this is a paranormalised function, and with no expectation that you should attempt to memorise this lot, except maybe "qraa", you can assume that all of the following are potentially perfectly valid, with the fairly obvious restrictions that any parameters must be in strict qraa-order, and no args without attributes: 

                        gDropDown()
                        gDropDown(changed)
                        gDropDown(changed, attributes)
                        gDropDown(changed, attributes, args)
                        gDropDown(attributes, args)
                        gDropDown(attributes)
                        gDropDown(options)
                        gDropDown(options, changed)
                        gDropDown(options, attributes)
                        gDropDown(options, attributes, args)
                        gDropDown(options, changed, attributes)
                        gDropDown(options, changed, attributes, args)  -- (the full set)

Note that in practice options and changed are actually defined as object, and attributes as a sequence rather than a string, but they are all manually and thoroughly verified to be of the documented types, after being repositioned via theGUI/paranormalise_qraa().
