gShow

Definition: include theGUI.e

gShow(gdx id, integer x=TG_CURRENT, y=TG_CURRENT)
Description: Displays a dialog in the given position on the screen, typically dead centre, or reverses a prior gHide(id) or gSetAttribute(id,`VISIBLE`,false) on any (sub-)element.

id: a result from gDialog() or similar [??use gPopupMenu for menus]
x, y: top left corner in pixels, or one of the predefined constants listed below. Ignored when id is not a dialog.
pwa/p2js: Supported.
Comments: Optionally the following definitions can also be used for x (instead of a specific pixel co-ordinate):
  • TG_LEFT: Positions the dialog on the left edge of the screen
  • TG_CENTER: Horizontally centralizes the dialog on the screen
  • TG_RIGHT: Positions the dialog on the right edge of the screen
  • TG_MOUSEPOS: Positions the left/right edge of the dialog on the mouse position
  • TG_CENTERPARENT: Horizontally centralizes the dialog relative to its parent
  • TG_CURRENT: use the current position of the dialog, TG_CENTER if not known.
  • TG_LEFTPARENT: Positions the element on the left border of its parent. Not valid for menus.
  • TG_RIGHTPARENT: Positions the element on the right border of its parent. Not valid for menus.
  • TG_ABUT: Positions the dialog at the bottom left of the element specified in y, flipping to top/right as needed.
  • TG_ABUTT: "" but prefers above over below.
  • TG_ABUTR: "" but prefers right edge alignment over left.
  • TG_ABUTTR: both flipped preferences.
and likewise the following definitions can also be used for y (instead of a specific pixel co-ordinate):
  • TG_TOP: Positions the dialog on the top edge of the screen
  • TG_CENTER: Vertically centralizes the dialog on the screen
  • TG_BOTTOM: Positions the dialog on the base of the screen
  • TG_MOUSEPOS: Positions the top/bottom edge of the dialog on the mouse position
  • TG_CENTERPARENT: Vertically centralizes the dialog relative to its parent
  • TG_CURRENT: use the current position of the dialog, TG_CENTER if not known.
  • TG_TOPPARENT: Positions the element on the top border of its parent. Not valid for menus.
  • TG_BOTTOMPARENT: Positions the element on the base of its parent. Not valid for menus.
  • id: When x is TG_ABUT[T][R], y must be a visible control-id.
Note that TG_CENTRE is for gDrawText() and the wrong constant, treated as 0 here (since it is) which in this case is taken to mean top/left, as in the absolute screen position {0,0}.

See also FULLSCREEN/MAXIMISED which can be set to take effect as soon as the window is first displayed.

When TG_CURRENT is used at the first time the dialog is shown then it is replaced by TG_CENTERPARENT.

When TG_CENTERPARENT is used but no parent is defined then it is treated the same as TG_CENTER.

When either x or y is TG_MOUSEPOS they must both be. The window is positioned such that the top left corner is at the mouse pointer co-ordinates, unless that would exceed the right hand side of the screen in which case it is positioned with the right edge touching the mouse pointer, and/or if it would drop below the bottom of the screen the dialog is positioned above the mouse pointer, in other words one of the four corners of the window will be on the mouse pointer. Should the window exceed half the height or width of the screen, then from certain positions the top left corner of the window may end up off-screen, in which case either use a smaller window or constrain where the mouse can be when this can happen towards the edges and away from the centre of the screen - so for instance you could still have a ridiculously huge popup when clicking on the properties of a monopoly board. For anything fancier you could always just calculate the co-ordinates yourself, it's not exactly hard.

TG_ABUT, which can only be passed in x, aligns a window to the edges of some other already shown control, which must be specified in y. It was implemented for the GTK version of gDatePick. The base version aligns the top of the window with the bottom edge, unless it does not fit on the screen, in which case it aligns the bottom of the window with the top edge. The two with an extra T prefer the latter over the former, so the pop-up appears above the y-control when it can. Likewise the base version aligns the left of the window with the left edge, switching to the right when it does not fit and those with an extra R the other way round. In truth, the [T][R] only really exist for testing purposes, which has only been done for the datepick control, and since they might yet prove useful there is no sense ripping them out. Behaviour is undefined should the window not fit entirely on-screen when either wholly above or below, or with left or right edges aligned (as above, anything better needs you to figure out the exact x,y needed). Abutment is always a vertical, never horizontal arrangement, similar in principal to the contents of a gDropDown, which you'd never expect to appear over to one side, or for that matter single-corner-only-touching. In other words the window and y-control will always share most/all of their top/bottom edges, but line-up their left/right edges with no overlap.

The main screen size does not include additional monitors.

This function can be executed more than once for the same dialog. This will make the dialog be placed above all other dialogs in the application, changing its Z-order, and update its position and/or size on screen.

IMPORTANT: Calling gShow() for a visible dialog previously shown with bModal:=true does nothing.
Example: 
gShow(dlg)
See Also: gHide
