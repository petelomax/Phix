ini

Simple ini/config file handling, implemented in builtins\ini.e (not an auto-include, but could easily be made one).

Such files keep important settings and other info in plain text, which can easily be read and manually edited in emergency cases.
Their contents should change as infrequently as possible, to wit this encourages the use of multiple ini files, instead of/in preference to allowing multiple groups/sections (with overloaded keynames) within a single ini file (with different lifespans and corruption consequences).
Multiple ini files may improve robustness simply by the fact that should a catastrophe occur it might only corrupt a subset of the settings and/or make restoring just the clobbered part significantly easier, plus unchanged files aren't overwritten, further reducing the risk of damage.
For instance it is not generally wise to mix, say, secure digital authentication keys with more transient things like recent files or usage stats: the latter are fairly frequently updated and should they be corrupted there is a reasonable probability of only some minor irritation, whereas key loss could lead to a far more serious and prolonged interruption.
Note that ini.e makes no attempt to make any backups itself, but does at least provide ini_filename() so you know what to save, and ini_dirty() so you know when and ideally not have to wade through thousands of identically-corrupted backups looking for an earlier one that still works, not that I can promise that won't ever happen. Aside: while there is inevitably a small risk that ini.e still contains a few bugs, the far bigger risk is that an application will tell it to store garbage, along with hardware failure, malware, acts of god, terrorism, and user error.

Each line in the ini file is either a comment line, blank or with line[1] in "-/;=" (caring not whether doubled as in "--" or "//"), or
a <keyname> = <setting> line, with an implied ' = ""' when no '=' is present (and with trim() applied to both halves).
Technically a comment (nb whole line only) is effectively held internally as `"" = "text"` (including the comment marker), just so you know.
Individual settings can be integer or string: other content (inc/esp floats) should be explicitly managed (at the point of call) via sprintf() and scanf().
[I might reconsider handling floats via "%g", but the "random precision" aspect rather deters me from doing that without a pretty good cause, which hasn't arisen yet.]
No settings can exceed a single line, but you should be able to manage your own '\n' embedding easily enough.
(Tip: ini.e would treat say "text[01]".."text[10]" as ten distinct keynames, so [I suppose] you could even split/stitch on that sort of basis manually.)
Note that ini.e also doesn't attempt any escape character or quote handling itself: that's all on you as well.
Likewise an ini file cannot hold say "123" as an unquoted string, you may have to quote-strip/add manually.
Duplicate keynames are overwritten, but it displays some hopefully helpful warnings on the console if any such are detected. It should be difficult and perhaps impossible to accidentally create duplicate keynames, except when they occur in the ini_default string, over which these routines have no direct control whatsoever, fairly obviously, about what an application decides to supply in that parameter.

These routines are supported by pwa/p2js, except that ini_load() relies almost entirely on the ini_default as given and ini_save() does nothing. I split the latter to create ini_as_string(), in the probably vain hope you may be able to do something with (say) local storage from that.

Example:

Copy 
include builtins\ini.e

constant TED = "theEditor",
     SESSION = TED&"_session",
ini_defaults = """
width = 240
height = 80
mainwin = {50,50}
""",
session_defaults = """
C:\Program Files (x86)\Phix\builtins\ini.e
C:\Users\Pete\theEditor.ini
"""
ini_load(TED,ini_defaults)
-- ...
    ini_restore(TED) -- (be sure "session" isn't active)
    integer w = ini_setting("width"),
            h = ini_setting("height"),
      {{x,y}} = scanf(ini_setting("mainwin"),"{%d,%d}")
-- ...
    -- reload files if any from the previous session:
    ini_load(SESSION,session_defaults) -- (optional here)
    for key in ini_keynames(SESSION) do
        object setting = ini_setting(key)
        if integer(setting) or length(setting) then
            ?{"dunno what this is",key,setting}
        elsif string(setting) then
            string text = get_text(key)
        else
            -- setting can only be integer or string
            crash("this should never actually happen")
        end if
    end for
-- ...
    ini_restore(SESSION) -- (it costs very little to play safe)
    -- add a new file to the session when opened:
    -- (override bMustExist when it likely won't)
    ini_set(filepath,"",false)
-- ...
    -- remove from session when file is closed:
    ini_restore(SESSION) -- (ditto)
    ini_delete_key(filepath)
-- ...
    -- update window placement if/when moved:
    ini_restore(TED) -- (ditto)
    ini_set("mainwin",sprintf("{%d,%d}",{x,y}))
    ini_set("width",w)
    ini_set("height",h)
-- ...
    ini_save(SESSION)
-- ...
ini_save(TED) -- see eg C:\Users\Pete\theEditor.ini
              -- (use ini_filename(TED) to be sure)
Using constants for ini_names is recommended to reduce the risk of typos getting past the compiler undetected.
The programmer is entirely responsible for ensuring that all ini_names are not just application but system-unique.


ini_load(string ini_name, ini_default="")
Fairly obviously, must generally speaking be invoked before anything else.
In the above sample, ini_load(SESSION,session_defaults) is only needed should you want to pre-populate it, should an empty file suffice the SESSION as passed to ini_keynames() would too.
In other words, you don't actually have to invoke ini_load(), however that would mean that bMustExist needs to be overridden for absolutely everything in that particular file.
If ini_name does not begin with '$' the file is stored in %HOME%/.ini_name on Linux, and %HOMEDRIVE%%HOMEPATH%\ini_name.ini on Windows (eg C:\Users\Pete\test.ini).
If ini_name does begin with '$' it is dropped and settings are stored in a system-wide location.
If there is a second '$' it is treated as $ini_name$configname otherwise configname is the literal constant "config". On Windows the file is stored in %ALLUSERSPROFILE%\ini_name\configname.ini, (eg C:\ProgramData\theEditor\config.ini) but currently Linux crashes since I have simply not decided where best to put it yet. Likewise this could be a good candidate for any initial p2js experiments, as next. [DEV]
Under p2js, relies entirely on ini_default with no attempt to read/write anything to disk.
[LocalStorage and/or some form of PHP API on a server remain untested options there.]
Invokes ini_filename() internally, ini_default is completely ignored once the file already exists.
The internal "structure" of ini_default, if you can call it that, is preserved indefinitely, including any comments, unless you really go to town with ini_delete_key(). The only way you could reorganise things (for your aesthetic reasons, the application should simply not care) or add comments, other than manually editing the file (at a time when it is not due to be overwritten, of course) would be to grab any details needed, ini_close(), physically delete the file, and re-create with a "nicer" ini_default.
Multiple invocations with the same ini_name, without an intervening ini_close(), will trigger a fatal error.

ini_restore(string ini_name)
Only ever needed when there (may) have been multiple ini_load() calls, but sensible and not at all expensive to call just in case additional ini files start getting used, for instance by some library component you later decide to start using. [The alternative would be to force ini_name to be provided on every fetch and set, which didn't feel right.] In other words, ini_restore() is "re-specify where we're storing things", rather than any form of recover or undo, and I suppose typically belongs at the start of a/each section/loop. For internal reasons, ini_restore("") does nothing, except prove at least one ini_load() has occurred, and is still "active" aka not ini_close()'d.

ini_save(string ini_name="")
Note there are no checks whatsoever that ini_save() is ever actually called, except via ini_close().
Can be invoked multiple times per session with no side effects other than the ini file itself being overwritten.
Does nothing unless ini_dirty() indicates it that needs/ought to (and then clears the dirty flag).
Invokes ini_as_string() automatically and writes the result to disk (but not under p2js).
Omitting ini_name should only be done when there is an explicit ini_restore() at the start of the code block, and there is no harm whatsoever or measurable performance overhead in explicitly (re-)specifying it.

ini_close(string ini_name="", bool bIgnoreDirty=false)
(optional) Completely removes the specified entry from the internal tables and sets the active index to 0.
An error occurs if there are any pending updates, aka ini_save() missing, unless bIgnoreDirty is overidden.
Omitting ini_name should only be done when there is an explicit ini_restore() at the start of the code block, and there is no harm whatsoever or measurable performance overhead in explicitly (re-)specifying it.
Provided for completeness only: there is no real benefit from ever bothering with this, other than verify-saved or you are for some strange reason attempting the awkward/longhand re-creation method mentioned in ini_load().

ini_dirty(string ini_name)
Made available in case it proves useful, used internally by ini_save() and ini_close().
Returns whether any pending ini_set() or ini_delete_key() have yet to be saved.
Note that ini.e makes no attempt to create any backups, so you might use this to decide when to make backups and/or periodically make them anyway. No files are left open between ini_load() and ini_save(), so copying is not a problem. Currently always returns false under p2js, since a browser might struggle to write backup files.

ini_as_string(string ini_name)
Made available in case it proves useful, used internally by ini_save().
Returns the keynames and settings as a single text string (with embedded '\n').
May possibly be useful in a JavaScript context, but probably not.
Alternatively you might want to use ini_filename() and this to hunt for and provide a suitable ini_default.

The bMustExist parameter is intended to help by shouting out missing ini_restore(). Should you override it to false, and/or fail to mention something in the ini_default as passed to ini_load(), it becomes the application's sole responsiblity to ensure the correct ini file is open/active. However some keynames, especially for instance filepaths, almost always need it to be overidden.

Should you want, for instance, to save the width and height of ten different windows, then rather than have ten "sections" with those fields duplicated in each, you'll want to have a unique details_width, etc.
