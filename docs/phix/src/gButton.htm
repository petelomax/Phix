gButton

Definition: include theGUI.e

gdx id = gButton([nullable_string title=NULL,] [rtn action=NULL,] string attributes="" dword_seq args={})
Description: Creates an interface element that is a button, which activates a specified routine in the application.
The visual presentation can contain text and/or an image.

title: Text to be shown to the user. It can be NULL, sets the TITLE attribute.
action: a routine which is invoked when activated, see ACTION below.
For more information on the attributes and args parameters see gSetAttributes().
This is a paranormalised function (see technicalia).

Returns the identifier of the created element.
pwa/p2js: Supported.
See Also: gImage, gCheckbox, gLabel
Example: 
include theGUI.e  -- demo\theGUI\gButton.exw (simplified)
gdx btn = gButton(`Close`,gQuit),
    hbx = gHbox({btn},`SPACE=AROUND`),
    dlg = gDialog(hbx,`gButton`,`SIZE=240x80`)
gMainLoop(dlg)
(Code to create a gImage and put it on the button as shown in the screenshot omitted for clarity)
Notes: Buttons can also be activated using Enter or Space keys when they have focus, or via Enter/Escape keys when they have been specified as DEFAULTENTER/DEFAULTESC on the dialog, as well as clicking on them, obviously.
Attributes:
IMAGE (non inheritable) a gImage. If TITLE is also set both are shown, however should you want the text to appear anywhere other than where it does, you should just use a bigger image with said text already on it.
The attributes/args would need to be in the "==" form to specify an image in the initial creation call.

There is also a IMAGE attribute on a gMenu and gTree, in case you were looking for them.
TITLE (non inheritable) Button's text. See here for mnemonic info.
If IMAGE is not defined before map, then the default behavior is to contain only text.
The natural size will be large enough to include all the text in the selected font, even using multiple lines, plus when specified an image, and the button borders.
also ACTIVE, CANFOCUS, EXPAND, FONT, MARGIN, SIZE, TIP, USER_DATA. VISIBLE.
Handlers: Set using eg gSetHandler(button,"ACTION",action), or specify action at creation time
ACTION
Event generated when the button is activated, similar to "click" events in other systems. When using a mouse, the event is only triggered when release occurs inside the button area, but you can also (tab-to and) activate a button using the keyboard, or use an accelerator key, or even ENTER/ESC when specified as that default on the dialog, via DEFAULTENTER/DEFAULTESC.

Note that in theGUI “CLICK” is a lower-level (mouse) event that applies to pretty much all controls except menus, including left/middle/right and single/double/release events as well as ctrl/shift/alt. Only a selected few of those w/should match what/when triggers "ACTION", and in any case said (mouse-click-only) event would not cover button activation via the keyboard.

procedure/function action([gdx id])
id: (optional) identifier of the element that activated the event.

A return of TG_CLOSE closes whatever dialog the button is a part of, all other values are ignored, with TG_IGNORE being the obvious recommended way to state that, or just use a procedure for any buttons that are not supposed to close the dialog.

There is also an ACTION handler on a gTimer, were you looking for that.
Shorthand:
(ACTION)
When nowt else needs doing use gQuit as the action on "Close"/"Cancel"/"Quit"(etc) buttons: 

        global function gQuit()  -- (as predefined in theGUI.e/js)
            return TG_CLOSE
        end function
Choose File:
(ACTION)
A button handler can also invoke, as can KEY and menu HANDLER, the JavaScript-compatible procedure
    gGetOpenFileName(gdx parent, rtn got, object arg2=NULL, filters=NULL, title=NULL, bool bMulti=false)
       - however invoking that from anywhere else is prohibited by the browser security model.
It causes a standard file open dialog to be displayed, allowing the user to select a file or (if bMulti=true) files.
  [My browser, at least, kicks things off with a little clipboard/current directory popup, but there's a button on the rhs to open the real deal.]
parent: specify the window that should act in that capacity, and regain focus when the pop-up closes. (can be NULL)
got: 
a procedure that accepts a file[s] result, and optionally arg2.
It recieves a single file when bMulti is false, or a sequence of them when not. See gGetFileText() below.
arg2: (optional) whatever else got[_txt]() needs, packed into a sequence if necessary.
filters: ("") In the form {{"Text Document (*.txt)","*.txt;*.text"}, {"All Files (*.*)","*.*"}} (or as below).
title: (WinAPI only) optional override of the default "Open" for the pop-up window.
bMulti: if true then got() is passed a list of files (even if only one selected), otherwise a single file.

In almost the lightest possible form of paranormalisation, when subsequent parameter[s] still have their standard default[s] and title/filters is true, it is assumed that was meant for bMulti.

The filters are pairs of string {friendly_name, pattern} in the WinAPI format, and assumes they perfectly match the GTK requirements, though of course they are case sensitive on Linux and may be using/need slightly different file extensions. They are automatically converted by theGUI.js/convert_filters() to an HTML-compatible <input type="file"> accept attribute: should any issues occur with that, or for any other reason, you can (nb platform()==JS-only) provide a single final string instead. In other words you may want if WINDOWS standard_form elsif LINUX case_sensitive elsif JS string, but that shouldn't be necessary for the simple/obvious/common cases.

Should you find a way to override the pop-up title on GTK or JS let me know, but I'm not about to let their lack of functionality impair my experience on Windows.

Ideally we would just have a JS-compatible get_text(), however JavaScript doesn't work that way and requires a callback once the results are in, so instead(/as well) we have:
    gGetFileText(object file, integer gtopt, rtn got_txt, object arg3=NULL)
       - which should only be invoked from the got routine as passed to gGetOpenFileName.
It bears much similarity to get_text(), except for the file and call-back parts.
file: 
on desktop/Phix this is [validated as] a string, however under js it is actually a somewhat opaque structure with, among other things, a fake directory in it. Note that if bMulti was true, so the first argument received by got() was a sequence of files, they need to be forwarded to gGetFileText() one-at-a-time. Also there isn't any formal(/js-compatible) support for getting some file[name][s] but not actually reading them, due to the aforementioned structure-opaqueness. [By opaque, I mean to transpiled-Phix-code: expect little luck trying to do anything with it in a got() routine under pwa/p2js other than forward it on. It may look perfectly reasonable in the Browser Development Tools, but it's just simply not Phix data.]
gtopt: see the options parameter of get_text() [manually replicated in theGUI.js/gtify()].
got_txt: 
the afore-mentioned callback, in this case a regular Phix procedure that accepts two or three parameters, being the filename, contents (potentially including -1 failures) and optionally an arg3.
arg3: (optional) can be identical to the arg2 as passed to gGetOpenFileName() and forwarded to got(), or part of it.

Confused yet? You have every right to be. The following little ditty should clear things up handsomely.

Copy 
-- demo\theGUI\file_open.exw - Simple use of gGetOpenFileName()
include theGUI.e

-- <some bog-standard gList stuff:>
sequence data = {{{"click the button above...",TG_GREY,TG_ITALIC}}}
integer maxlen = 1

function get_data(integer idx)
    if idx=0 then return length(data) end if
    if idx=-1 then return maxlen end if
    return data[idx]
end function
-- </some bog-standard gList stuff>

procedure got_txt(string filename, sequence contents, gdx lbl)
    gSetAttribute(lbl,"TITLE",filename)
    data = contents
    maxlen = largest(apply(data,length),return_index:=true)
    gRedraw(gGetDialog(lbl))
end procedure   
    
procedure got(object file, gdx lbl)
    -- (You'll need a loop here if bMulti was true)
    gGetFileText(file,GT_LF_STRIPPED,got_txt,lbl)
end procedure

procedure action(gdx btn)
    gdx dlg = gGetDialog(btn),
        lbl = gGetAttribute(btn,"USER_DATA")
    gGetOpenFileName(dlg,got,lbl)
end procedure

gdx lbl = gLabel(" ","EXPAND=H"),
    btn = gButton("Choose File",action,"==",{{"USER_DATA",lbl}}),
    hbox = gHbox({btn,gHbox({lbl},"SPACE=AROUND")},"GAP=5,EXPAND=H"),
    list = gList(get_data,`FONT="Consolas, 12", BGCLR=LIGHT_PARCHMENT`),
    vbox = gVbox({hbox,list}),
    dlg = gDialog(vbox,"File Open Demo",`SIZE=435x171`)
gMainLoop(dlg)

Technicalia
As this is a paranormalised function, and with no expectation that you should attempt to memorise this lot, except maybe "traa", you can assume that all of the following are potentially perfectly valid, with the fairly obvious restrictions that any parameters must be in strict traa-order, and no args without attributes: 

                        gButton()
                        gButton(action)
                        gButton(action, attributes)
                        gButton(action, attributes, args)
                        gButton(attributes, args)
                        gButton(attributes)                      -- (see 2nd note)
                        gButton(title)
                        gButton(title, action)
                        gButton(title, attributes)
                        gButton(title, attributes, args)
                        gButton(title, action, attributes)
                        gButton(title, action, attributes, args)  -- (the full set)

Note that in practice title and action are actually defined as object, and attributes as a sequence rather than a string, but they are all manually and thoroughly verified to be of the documented types, after being repositioned via theGUI/paranormalise_traa().

Note that gButton(attributes) is distinguished from gButton(title) by the presence of an '=' character, see here.

While in theGUI a return of TG_IGNORE from the ACTION handler somewhat more logically simply ignores the event, in pGUI a similar IUP_IGNORE completely removed the callback, and hence effectively disabled the button, and without any visual feedback of that I might add. To achieve that same sort of effect in theGUI one would have to explicitly invoke gSetHandler(id,`ACTION`,NULL), and/or (but with some visual feedback) gSetAttribute(id,`ACTIVE`,false).
