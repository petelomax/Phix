Logical Operators

The four logical operators and, or, xor, and not are used to determine the "truth" of an expression. e.g. 
        1 and 1     -- 1 (true)
        1 and 0     -- 0 (false)
        0 and 1     -- 0 (false)
        0 and 0     -- 0 (false)
        1 or 1      -- 1 (true)
        1 or 0      -- 1 (true)
        0 or 1      -- 1 (true)
        0 or 0      -- 0 (false)
        1 xor 1     -- 0 (false)
        1 xor 0     -- 1 (true)
        0 xor 1     -- 1 (true)
        0 xor 0     -- 0 (false)
        not 1       -- 0 (false)
        not 0       -- 1 (true)
Function-style equivalents of these operators can also be applied to entire sequences, as explained shortly in Sequence Operations, eg
sq_and({true,{false,true}},{true,false}) ==> {true,{false,false}}.

In all cases short-circuit evaluation is used for expressions containing and or or.

The constants TRUE, True, true, FALSE, False, and false are automatically builtin to the Phix compiler. There is no difference between the different cases, which makes for one less thing you have to remember.

You may also find these operators applied to numbers other than 1 or 0.
The rule is: zero means false and non-zero means true.
Replacing variable names with literal values to show exactly what is going on: 
        5 and -4    -- 1 (true)
        not 6       -- 0 (false)
One could argue the above is relatively poor practice. The equivalent expressions "5!=0 and -4!=0" and "6==0" yield the same results and can communicate the intent with greater clarity, at least when using sensible variable names rather than those meaningless literals, and likewise false rather than those three trailing 0s. Should there be any cases where the latter forms generate slightly less efficient code, they should be logged as a bug in the compiler. There are, however, countless instances of that sort of thing in legacy code, far too many to ever fully eradicate, especially given that attempts to do so are likely to introduce new bugs (even if they are schrödinbugs1), plus there is also the trick of using "not not" to prove something is or cast it to a bool (often seen as "!!" in other programming languages), though I will say the frontend is generally pretty keen on coercing things to 0/1, especially in the near vicinity of/results from these logic operations (aka test it first, as in you might not need "not not" anyway).

One case worth mentioning is the popular idiom [not] find(needle,haystack). Encouraging the programmer to tack ["=0" or] "!=0" onto said does not really improve anything, and smacks of pedantry and pettiness, but of course using a bare find() like that relies on exactly the same principle of 0 means false and anything else means true (which can also be a good argument in favour of 1-based indices over 0-based, and partially explains why some languages [without -ve subscripts] represent false as -1, which of course I deem deeply flawed).

Parenthesis is required to mix different logical operators in a single expression, as explained below the Precedence Chart.

See Also: and_bits, or_bits, xor_bits, and not_bits, which are the corresponding bitwise functions, or the ARM data processing operations.
Technicalia 1schrödinbug: a bug that occurs after the programmer notices that the code should never have worked in the first place, despite having passed all tests with flying colours for donkey's years. Typically the modified code works just fine, but you cannot actually test it without first eradicating the eighteen other near-identical bugs that have suddenly decided to start triggering, for no obvious reason other than the fact they too were all always obviously just waiting to go wrong and you simply cannot understand why they never have.
Not to be confused with a heisenbug: a bug that disappears when one attempts to study it (which incidentally refers to the observer effect, as first proposed by Heisenberg, rather than the Heisenberg uncertainty principle). There is also a mandelbug, which is a bug whose causes are so complex that the deeper a developer goes into the code the more bugs they find, forever.
