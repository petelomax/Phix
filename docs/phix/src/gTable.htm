gTable

Definition: include theGUI.e

gdx id = gTable(sequence columns, data, [integer rows=10, ] string attributes="", dword_seq args={})
Description: Creates an interactive, resizeable and sortable table with multiple columns.

columns: each element should be "title" or {"title",width[,"align"]}, see COLUMNS.
data: data[1] contains the raw sortable data, data[2] contains formatting/ed info/strings, see DATA.
rows: (optional/quickie) the default number of visible lines to display, see comments and technicalia.
For more information on the attributes and args parameters see gSetAttributes().
This is a (lightly) paranormalised function (see technicalia).

Returns: the identifier of the created element.

See Also: gCanvas, gList, tagset
Example: 
-- demo\theGUI\gTable.exw
include theGUI.e
constant columns = {{"Chq",30,"CR"},
                    {"Date",80,"C"},
                    {"Amount",80,"R"}},
         data = {{{1,{2025,1,13},1500.89},
                  {2,{2025,7,10},3646.86},
                  {3,{2024,3,20},3807.94}},
                 {0,"%[3]2d/%[2]02d/%[1]4d","%.2f"}}
gdx table = gTable(columns,data,3),
      dlg = gDialog(table,"gTable") 
gMainLoop(dlg)
-- 2nd image depicts the same table after top line selected, then sorted by date, and resized to (just) need scrollbars
-- Those are Windows 11 style scrollbars btw, that grow fatter & gain arrowheads when the mouse hovers on them
Comments: The rows parameter (more details in the technicalia) sets the default table height, with column widths initially acting as a minimum (and maximum if -ve) and accumulated to a default total minimum width.

When the table is resized it expands/shrinks to fit, distributing any slack/excess in a pro-rata fashion, for example column widths of {80,20,100} get 80/200ths, 20/120ths, and 100/100ths respectively, reducing by the amount actually alloted to each column after each step, and some or all of the columns can be prevented from automatically resizing by specifying negative widths, though they can still be manually adjusted (as below).

The SCROLLSIZE and VIEWPORT are fully automated, delivering scrollbars as and only when needed, and automatically shifted to ensure the current line/cell remains visible, except when multiple lines have been selected.

The table can (optionally, see COL_SORT) be sorted by clicking on the column titles.

The divider between any two column titles can (see COL_SIZE) be dragged to widen one and narrow the other, or a column title ctrl-clicked to make it a resize_sink (see next). Column titles can also (see COL_DRAG) be dragged left/right.

Supports (see SELMODE) single cell, single line, or multiple line selection, the latter via ctrl/shift clicks, shift [page]up/down and home/end, or <Ctrl A>. Single selections can also be moved by clicking, left/right (single cell only), and [unshifted] [page]up/down and home/end.

A gTable seamlessly handles all resizing, sorting, and column reordering: apart from restoring a user-modified layout, the application need not concern itself with any of that and pretend the table has not been altered/sorted at all. Should it be needed it can retrieve the internal tagsets and topline, so it knows where everything's at and still deliver finer control.
See demo\theGUI\gTable.exw for an example of saving/restoring user-modified layouts, noting that may contain some finer details about that process which have not quite made it into these these docs. [DEV/todo]

A gList() is designed to cope with rapidly changing data much more efficiently than a gTable, whereas in comparison the latter packs a pretty serious punch in terms of implicit builtin functionality (imnsho), while remaining easy to use, and not seemingly all that difficult to extend even further in the future.

I should perhaps remark that tables with more than 1,000 entries become increasingly difficult to navigate, then again for instance filedump.exw should have no difficulty showing the full dissassembly of a 4.5MB phix executable itself.
Resize sink: Ctrl-clicking on a column title (when >1 exists) toggles it as the resize sink, indicated by a "+", which then absorbs size changes from the parent container/dialog, and prevents other columns from shrinking even when they no longer fit.
Ctrl-shift-clicking any column title does the above and reveals/hides any zero-width columns so you can see where they are or should be, and/or make them properly (in)visible again, actually <5⇒5 and =5⇒0 each time, with the matching adjustment to the new/old resize sink, at least while there is still enough of that left. In other words cs-click on a narrow column could potentially make that zero-width and thereafter carry on "revealing" with +0s. Likewise should a horizontal scrollbar already exist, it immediately tries to get rid of (the excess causing) it by shrinking the newly-created resize sink.

Note: changing column widths, either by dragging the dividers between titles or activating a resize sink, automatically sets the minimum width, not just for that column but all columns in the table, and hence the minimum width of the entire thing, and thereafter the only way to reduce the latter would be via a resize sink (or ten). The moral of the story is that column widths should only be fine-tuned when the table is (roughly) the desired minimum size.

When there is no resize sink (ie the default) all container-driven-width-changes apply in a pro-rata fashion (as above).
Attributes:
COLUMNS The column titles, normally provided as the first parameter on creation.

If column[i] is a string, it is treated as width 80 and left aligned.
If instead it is {"title",width[,"align"]}, width specifies the minimum width, and maximum if negative, else it gets a pro-rata share of any available space. The align string should be "L", "C", or "R" for left/centre/right alignment, or two-characters to specify the {title,data} alignment differently, eg "CR" makes the column title centred but the data right-aligned.
The column widths are accumulated to a natural/minimum size (see also the rows parameter), but it is normally better to specify an explict SIZE on the table/dialog, with that including a little spare slack over the bare minimum.
Note that changing the FONT has no direct effect on the (minimum) widths specified here.
Technically speaking, you can effectively hide selected columns by setting/dragging their respective widths to zero.
It is also perfectly acceptable to initially display a long table with default widths and update them after some potentially lengthy background process has calculated the best/minimum sizes, eg using gGetTextExtent() on every entry. Apart from such "just width" updates, replacing columns will almost certainly require DATA be replaced as well.

Most applications are unlikely to need to read this back, likewise COL_DRAG and COL_SORT, except to restore the view/layout perfectly, should it actually want to do that: see demo/theGUI/gTable.exw for more details. Note the original widths may (or may not) have been overridden/populated by resize activity (as in the user dragging things about with their mouse). Replacing the columns automatically wipes resize_sink, and col drag/sort when not doing so would inevitably lead to a crash, but the recommendation is to [next] wipe(/restore) those explicitly yourself, that is when it is the sensible thing to do. Lastly note that COLUMNS must always be set before DATA.
COL_DRAG Whether the columns can be dragged/reordered. Default: true, logically equivalent to tagset(length(columns)), and returned/settable as some reordering of that when they have actually have been reordered.
Must be set after COLUMNS when restoring a user-modified table layout (see gTable.exw).
COL_SIZE Whether the dividers between column titles can be dragged to widen one and narrow the other. Default: true.
COL_SORT Whether the table can be sorted by clicking on the column titles. Default: true.
Can also be say {true,false,true} should sorting be illogical/unhelful/problematical on some, such as images or progress.
 (a table, for instance, sorted by progress every 2s would be an application's responsibility, not the gTable's, and require DATA be nulled/reset)
Clicking twice flips the order, thrice removes it (when allowed, that is). It maintains a stack (sans duplicates) of sort orders: should you click on amount then date, it shows items in amount order within each unique date, with any all-ever-clicked columns all-matching lines in their original order, and if you then cycle date to remove it, the table reverts to amount order, and obviously when that is also removed the table reverts back to the original order.

Returns (t/f or) {colset,tagset,invtag} where colset is a sequence of pre-COL_DRAG column indices, and tagset/invtag are the resulting display ⇔ data index mappings. When restoring (after COLUMNS and DATA) you need only specify {colset} and let it recompute the tagsets (as per gTable.exw), which of course you could then retrieve immediately. Note that eg {{3,2,1}} restores an actual sort order, whereas {t,f,t} sets which columns can be sorted, and theoretically {3,2,1} w/could be treated as all-true, however it does in practice spot when not length(COLUMNS) or max()>1, so the only gotcha is that {1} on a single column table is "which not order", and in that one case you really would need {{1}} to restore the sort. Lastly note that setting t/f/{t/f} deletes any sorted order, and an unsorted {t/f} is deliberately returned as just t.
DATA A length-2 sequence, normally provided on creation, essentially { <actual/sortable>, <displayable/control> }.

data[1] contains the master/sortable values, and fairly obviously determines the total number of rows in the table.
data[2] is reserved for (string) display versions of data[1] or format string/handlers:
if data[2] is too short or data[2][col] is 0, then data[1][row][col] is displayed as-is/sprint()'d
if data[2][col] is a string fmt then sprintf(fmt,data[1][row][col]) is displayed.
if data[2][col] is a non-zero integer it is assumed to be a formatting function [eg date_fmt() in gTable.exw],
or data[2][col] can be a sequence of pre-formatted strings, eg data[2][1] above could be {"(1)","[2]","{3}"}.

For dates, data[1][row][col] should be eg (int) 19050926 or (timedate-like) {1905,9,26,"special relativity published"},
          and data[2][col] a format string/routine, or data[2][col][row] a preformatted string.
For case-insensitive sorting, data[1][row][col] should be lower/upper(data[2][col][row]).
For images, data[1][row][col] should be {key, gdm} and data[2][col] should be "image".
Sorting images relies on a key/index. Alternatively data[1][r][c] can be just key, with data[2][c] a fn yeilding {"image",gdm}.
 (key can be an integer or string or anything you like, as long as it is sortable, so probably not a memory address).
Note that data[1] is one-per-row whereas data[2] is (<=) one-per-column.
As noted above, COLUMNS must always be set before DATA.

A gTable treats the data as strictly read-only. Can/shd be temporarily nulled to circumvent refcount issues.
Should these docs refer to "the unsorted data", it simply means "because it never is, it uses tagsets instead".

There is also a DATA attribute on a gTree, and a DATA handlers on a gList and gGraph, were you looking for them.
SELMODE,
CELLMODE 		Specifies cell or line based selection. Possible values: "CELL"/true, "LINE"/false, or "MULTIPLE"/-1 (lines). Default: "LINE".
Setting this automatically wipes SELECTED. The CELLMODE alias behaves identically.
Returns a string when fetched using gGetAttribute(), or true/false/-1 when consulted via gGetInt().

Note that when a single line or cell is selected, column sort keeps it visible (as per SHOWLINE), however if the user has selected multiple lines it gives up any and all attempts to do that (feel free to have a play in hTable.exw).

Incidentally, there is (also) a MULTIPLE attribute on a gList, were you looking for that.
SELECTED Returns 0 if nothing selected, the currently selected row (integer) when SELMODE is "LINE", or a {row,column} pair when "CELL", or integer row or full-length sequence of true/false when multiple lines have been selected (using ctrl or shift).
Line and column numbers are completely unaffected by any sorting/dragging, and can be applied directly to DATA[1].
Can be retrieved on demand or the identically-named SELECTED handler receives notifications whenever it changes.
There are also similar SELECTED attributes/handlers on a gList and gTree, were you looking for them.
SHOWLINE Ensure the specified row or {row,column} is visible, with both being direct indexes to the unsorted DATA[1].
Intended more for internal use (write-only), and automatically applies the tags as per COL_SORT and COL_DRAG.
Simply sets the VIEWPORT (after mapping to a display index) and triggers a redraw when necessary.

There is also a SHOWLINE attribute on a gList, were you looking for that.
TOPLINE,
LINEHT 		Determine/specify the display index (1..length(DATA[1])) at the top of the control, and/or (read only) the line height.
Should you want to know how many lines are visible, it is h/LINEHT, where h is from the VIEWPORT, and you'd have to make your own decision as to whether floor() or ceil() is more appropriate. TOPLINE may also be implicitly/indirectly set by SHOWLINE, also note that the VIEWPORT is really in charge, and should you micro-scroll such that TOPLINE is technically 1.9, you'll just get a 1 back, even though only a tenth of that is actually visible.
Note TOPLINE is a display index and when the table is sorted may need mapping from/to a data index.
LINEHT defaults to 24 for any size calculations prior to the first display (specifically using the rows parameter), thereafter FONT height + 9, so you might benefit from resetting it to 0 when changing the font.

There are also TOPLINE and LINEHT attributes on a gList, were you looking for them.
also ACTIVE, BGCLR, BGSEL, BGHOV, EXPAND (see note), FONT, MINSIZE, MAXSIZE, TIP, SIZE, VISIBLE

Note SCROLLSIZE and VIEWPORT are automatically handled on a gTable, and you should not tamper with them.
Handlers: Set using eg gSetHandler(table,"SELECTED",selected)
SELECTED Action generated when a row [and cell] is selected.

procedure selected([gdx id,] integer line[, col]) -- LINE[,CELL] mode only
-- or --
 procedure selected([gdx id,] object items) -- all modes

id: (optional) identifier of the gTable element that activated the event.
line[, col]: The row [and column] selected.
items: 0 if none, integer line if LINE mode, {line,col} if CELL mode, and integer line or a full-length sequence of true/false when multiple (ie ctrl or shift) lines are selected.
The items value is exactly the same as that returned (on demand) by the SELECTED attribute.

The line and column numbers can/shd be used to subscript the (unsorted) DATA[1], regardless of whether the user may or may not have sorted columns or dragged them into a different (display) order. Likewise should MULTIPLE be in force and the user selects the first two of five, you'll get {1,1,0,0,0}, and while they may jump about on the screen when the user sorts on different columns it does not change what gets passed to this handler or retrieved via SELECTED in any way.

There are also SELECTED handlers on a gList and gTree, were you looking for them.
also Note a gTable uses CLICK, KEY, MOUSEMOVE, and MOUSELEAVE internally, and you should not attempt to use/override them, shout should you need em.
Technicalia As this is a paranormalised function, with columns and data non-optional, and with no expectation that you should attempt to memorise this lot, except maybe "cdraa", you can assume that all of the following are potentially perfectly valid, with the fairly obvious restrictions that any parameters must be in strict cdraa-order, and no args without attributes: 

                        gTable(columns,data)
                        gTable(columns,data,rows)
                        gTable(columns,data,attributes)
                        gTable(columns,data,attributes,args)
                        gTable(columns,data,rows,attributes)
                        gTable(columns,data,rows,attributes,args)  -- (the full set)

Note that in practice rows is actually defined as object, and attributes as sequence rather than string, but they are all manually and thoroughly verified to be of the documented types, after being repositioned (if necessary) by theGUI/paranormalise_raa, in this particular case with bCheckRid set to false.

The rows parameter is just a simple shorthand which will hopefully help 9 out of 10 simple cases, but quite likely let you down the other 10% of the time. It depends on LINEHT, which might not match the FONT height until after display, too late to set a default size. May need to be 0 to avoid clobbering a SIZE, and makes no attempt to take the presence or absence of a horizontal scrollbar into account. Should rows not work as hoped (even with the correct LINEHT) stop trying and set a SIZE instead, on the table itself or some parent container, ideally the dialog.

Tables are (currently) essentially always read-only: instead of in-place editing you may need additional field[s] and an "Apply" button, probably disabled when anything other than a single item is selected, with it entirely your responsibility to update DATA. I'm not particularly keen myself anyway, but certainly steering well clear until all other emulated controls are completely finished. (feel free to have a hack in hTable.exw)
