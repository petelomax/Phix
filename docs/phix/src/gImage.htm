gImage

Definition: include theGUI.e

gdm image = gImage_from_XPM(sequence xpm, bool bPixbuf=false)
-- or --
 gdm image = gImage(integer w, h, atom bgclr=TG_WHITE, transparent=TG_WHITE, bool bClear=true)
-- or --
 gdm image = gImage_from_Pixbuf(gdp pixbuf, atom transparent=TG_WHITE) -- [NB not fast, aim for once-only]
-- and optionally --
 gdn dead = gImageDestroy(gdn dead)
Description: Creates a platform-specific GdkPixbuf/DIBitmap/DIBSection/HtmlCanvas that can later be drawn onto a gCanvas, or put on a gButton(), gMenu() [leaf nodes only], or gTabs(), or [the very same one] used on all four.

xpm: a [locally defined constant] text or sequence of text strings in human readable form (but not a filename).
bPixbuf: permit gPixbuf_from_Image(). [False affords savings on [unalterable] images, esp for buttons/tabs/menus.]
w, h, bgclr: width, height and initial fill colour of a blank gImage.
bClear: (intended for internal use only) - no point clearing when we are about to blat every pixel from a pixbuf/leave the garbage alone.
 transparent: probably only useful on images intended for WinAPI menus, same as "None" in an xpm.
pixbuf: (2d sequence or an even -ve integer gdp) see gPixbuf.
Note that gImage_from_Pixbuf() automatically destroys a theGUI-owned pixbuf (removes it from the internal table).
dead: (single or sequence with nulls skipped) free up slot(s) in the internal table for later reuse.

Note that a gImage is not an interface element and cannot be directly added to a layout hierarchy.
pwa/p2js: Supported, except gImage_get_transparent has not yet been written.
See Also: gCanvas, gDrawImage, gDrawText, gPixbuf
Example: 
-- demo\theGUI\gImage.exw
include theGUI.e
constant cut_xpm_txt = """
19 15 2 3
... c None
*** c #0000FF
.............................................***.........
.......................................******............
....................................******...............
.................................******..................
..............................******.....................
...************............******...............*********
***.........******......******.........************......
***.........*********************************............
...************.........************.....................
.....................******..............................
..................*********..............................
...............***......***..............................
............***.........***..............................
............***......***.................................
............*********....................................
""", txt = "(as redrawn on a gCanvas)",
 cut_xpm = gImage_from_XPM(cut_xpm_txt)

procedure redraw(gdx canvas, integer w, h)
    integer {tw,th} = gGetTextExtent(canvas, txt),
            -- split w as <hm,19,[5],tw,hm>:
            hm = floor((w-tw-24)/2)
    gDrawImage(cut_xpm,canvas,hm,h/2-7)
    gDrawText(canvas,hm+24,h/2,txt)
end procedure

gdx canvas = gCanvas(redraw,"BGCLR=LIGHT_PARCHMENT"),
    dialog = gDialog(canvas,`gImage`,`SIZE=240x80`)
gMainLoop(dialog)
Notes: A gButton(), gMenu(), or gTabs() can accept an xpm string directly and invoke gImage_from_XPM() automatically, hence invoking that directly is only recommended when at least one or more of the intended targets is a canvas. A gCanvas is usually resizeable (by the end user), whereas a gImage is totally fixed, although of course you can create one large enough for all cases and only use part of it right now, or just re-create whenever too small.
While a gCanvas is regularly blanked, that only happens to a gImage when deliberate and explicit.

Most applications probably need not bother with gImageDestroy() unless they are continuously creating lots of them, and slowly dying from that particular memory leak. It may be much better to create a single permanent image as large as might ever be needed (ie/eg SCREENSIZE) and just use part of that rather than create a brand new image every time (for instance) a REDRAW handler is invoked. You may prefer to code something like assert(image>=-200), where 200 is app-specific, the smallest finding new bugs quickly, as found by experiment, to check there is no runaway condition. Note that images placed on menu items can never be destroyed/reclaimed, though gImage_from_XPM() does check for "same text" and re-uses whatever it made earlier (but not [yet] under p2js).

atom transparent = gImage_get_transparent(gdm image)
Just in case you need it.

object xpm = gGreyscale_XPM(object xpm)
Does pretty much what it says on the tin, eg "#0000FF" ⇒ "#939393" in the example [text] above.
xpm: must be a [single] string, sequence(as in split(str,'\n'), or gdm (one xpm at a time only, please).
Return matches the input type, with non-XPM gdm returned unaltered.

Note this works by modifying the original xpm text: should it be a gdm on which you have performed gDrawLine() .. gDrawText() those do not appear. The equivalent can easily be performed on a pixbuf by invoking gGreyscale() on each pixel.
Aside: (re-)passing a gdm permits a mini-optimisation of not re-mangling text, whereas str/seq (re-)input must.
Example: A gTree uses this to automatically greyscale all the images when ACTIVE is false (see hTree.exw).

Application icons are usually 32x32. Toolbar bitmaps are 24x24 or smaller. Menu bitmaps and small icons are 16x16 or smaller. Transparent pixels are identified using the colour "None".

The first line of an xpm is <wide> <tall> <colours> <chars per pixel> [<hotspot x> <y>].
[No attempt has yet been made to utilise a hotspot in any way.]
The colour codes part consists of <colours> lines, each being <code> <key> <colour>.
Only the c key is currently supported (not m/s/g4/g), and given the tendency of copy/paste to mangle whitespace it is completely prohibited to use said anywhere in the pixels part of an XPM definition [in theGUI].
The colour code can be None for transparency or a hex RGB code (no other [symbolic] names or HSV codes).
The pixels part consists of <tall> lines, each <pre-defined-codes-only>, and exactly <wide>*<chars per pixel> bytes long.
No extensions are supported (such as XPMEXT or XPMENDEXT).
There is no use of nor any replication of any of the functions or structures of xpmlib [except within/private to GTK].

GTK does not like duplicate colour codes, hence although fine on WinAPI, theGUI.e validates that for you. However, that bit of code is not executed under GTK (since it handles xpm natively), so you'd just get a duff/mis-coloured image.

Aside: the sample xpm above uses three characters per pixel to get the proportions/aspect ratio to look a bit better and prove it can be done, but of course a single character each would work just as well and use less space.
Attributes: FGCLR default: TG_BLACK,
FONT default: NULL==`Helvetica, 9`,
LINESTYLE default: TG_CONTINUOUS,
LINEWIDTH default: 1,
USER_DATA default: NULL,
SIZE (read-only).

Note there is no SIZE/USER_SIZE-like situation of a dialog element, on a gImage SIZE is absolute and final, and read-only (should you create a replacement it is your responsibility to preserve any of these attributes it might need).
When needed, to wipe an image clean use gDrawRect(image,0,0,w-1,h-1,true,0,bgclr).
Technically you can also set BGCLR, but it is not actually used (or sensibly defaulted) anywhere.
While NULL effectively resets FGCLR, FONT, and USER_DATA, the others may need their defaults deliberately restoring.
Lastly note that gSetAttributes (the with-an-s one) is not applicable to a gImage; instead you have to set them one-at-a-time, or stick to overriding them on the individual gDrawXxxx statements.
Technicalia Of course I have no objections to there being a separate desktop-only component (marked without js) for loading images from disk, but theGUI itself (which cannot be so marked) should remain staunchly platform independent. It is important that simply including (or even auto-including) the bits you actually use is enough to get those tags right automatically. On WinAPI, images from gImage_from_XPM() can be 1, 4, 8, or 32 bpp (bits per pixel), whereas those from gImage_from_Pixbuf() are 32 bpp, not that any attempt has yet been made to use or test an alpha (/transparency) channel, and not that such differences should be a major concern. GTK handles xpm natively, and theGUI.js painstakingly constructs a 2d canvas (again, not that you should really care).

GTK, HTML/JS, and gDrawImage() all handle transparency properly, and buttons under WinAPI now I've got the imagelists working properly, however that is not the case for menu items under WinAPI (why oh why is there no way to use imagelists on menus?), hence for them theGUI creates multiple/as-needed bitmaps with the background colour replaced, maintaining a mini-cache of images - see the internal constants TG_MENUBG and TG_MENUHLT, and xpm_clrs, that is for WinAPI: under GTK each entry is a single image, and of course there is no need for any of that in theGUI.js at all. I know there is a way to create pre-multiplied alpha transparent bitmaps (or somesuch) for use on menus, I just haven't figured it out yet... [and now you see why I wanted/bothered to state 1/4/8/32 bpp above]

Multiple calls to gImageDestroy() with the same gdm would trigger a fatal error. Further, images created via gImage_from_XPM() are deemed static, relatively small and few in number, and reusable, as are any images ever added to a button, tab, or menu entry, and are never deleted - only those from gImage_from_Pixbuf() are ever actually removed. You should endeavour to avoid repeatedly recreating identical images via the latter, whereas theGUI detects when multiple calls are made to gImage_from_XPM() with the same string as previously provided, and returns a pre-existing/permanent slot (with a different transparent colour for menu items under WinAPI as noted above).

At some point in the probably distant future I plan to port some or all of the code/demos from demo\Arwen32dibdemo to theGUI, and fiddly little technical issues aside cannot think offhand of any reason why they would not work on both a gImage and a gPixbuf. Admittedly fancy textures and more importantly special effects like lighting are really an OpenGL thing, and I cannot see theGUI ever taking advantage of the GPU hardware.

To avoid pointless CORS violations (I take a very dim view of those stupid localhost restrictions), theGUI is much more concerned with loading images from locally defined data rather than file. At this point in time only xpm is supported. There are plenty of online converters for creating xpm from any image, personally I downloaded and installed and use Greenfish Icon Pro, for now at least that sort of thing should suffice. Longer term I would not mind handling svg and/or a standalone utility to convert other formats into a suitable/shortest source code snippet, especially png, with jpg, gif, ico, and bmp (etc) being a much lower priority, along with suitable guards and/or methods to achieve js compatibility.

You could of course use get_text() to open an xpm file, but that would be desktop/Phix only, and may as well just open that file in an editor and copy/paste the contents into a source file, as a string constant. Makes shipping and installing the final product a tad easier, too.
