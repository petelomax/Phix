Progress / hFake programs

Several programs such as demo\theGUI\hButton.exw are being/were written to replace the use of native controls, that is using a plain canvas and drawing everything and handling all mouse and keyboard events themselves. Surprisingly it is often debateable whether that approach is in fact any harder than using a native control on a single backend, but without any doubt whatsoever far easier than coercing two or more of them into some form of even loose agreement. In particular, if you are drawing everything yourself, obtaining the size is trivial, whereas playing multiple merry dances, all completely different, to get each backend to tell you, is quite simply a world of pain. Add to that subtle differences in the events generated, precisely when, and in which order, and fully emulated controls suddenly become much more appealing.
Concerns it will all be slower probably evaporated around the time processors broke the 0.1GHz barrier, and the widespread near-pointless use of things like a really really subtle gradient fill that at best improves the appearance by 0.01% kinda prove that, unless you happen to think any of the images above are just awful. The big downer would be compatibility with assistive technologies such as screen readers, but that's a bridge we'll just have to cross later.

These mini-programs (should theoretically) make porting to theGUI.js, and in the longer term experiments, enhancements, and bug fixing, all considerably easier. The following table summarises progress. There is no realistic way to isolate the core components gCanvas, gDialog, and gH/Vbox, and the emulated hFrame and hSplit are a bit of a fudge in that you can only represent their contents with a cross rather than nest any real/emulated controls within them. Incidentally gGraph predates all this so I have no plans to ever create hGraph, hList is slighly duff wrt key/mouse (but gList is fine), and hSpin may be done as part of hText. All hXxxx that have reached the top rung should have a much simpler and often trivial gXxxx version containing the code each of the above screenshots links to. NB all test programs are primarily concerned with appearance and direct interaction and there is simply no point whatsoever in wasting effort fixing minor glitches, esp VK_TAB/MOUSEWHEEL, in hXxxx that do not/no longer occur in the equivalent gXxxx - always test that first.
