min / max / etc

Definition: object res = min(object x, y)
-- or --
object res = min(sequence s)
-- or --
object res = smallest(sequence s, bool return_index=false, integer exclude=0)
-- or --
object res = sq_min(object a, b)
Description: Returns the smallest of x and y, or the smallest element/index in sequence s. Atoms are deemed less than sequences, with [nested] sequences compared case sensitively, ie 'Z'<'a', until the first difference.

There are similar functions max() and largest() for obtaining (you've guessed it) the largest.
Additional: atom res = median(sequence s)
s must be a non-empty sequence on entry, and pre-sorted for meaningful results (not verified).
Returns the middle element for odd-length sequences, or the average of the middle two elements for even-length sequences. Yields the same result as average() when the spread is perfectly balanced, eg {1,40,50,60,99}.
pwa/p2js: Supported.
Example: 
a = min(7,5)                            -- a is 5
b = min({9,7,5})                        -- b is 5
c = min("two","three")                  -- c is "three"
d = min({"one","two","three","four"})   -- d is "four"
s = sq_min(s,1)         -- no element of s is now > 1
?smallest({"this",{3,5,7},9})       -- prints 9
?largest({"this",{3,5,7},9})        -- prints "this"
Implementation: See builtins\pmaths.e (an autoinclude), and psqop.e, and the technicalia dropdown, for details of the actual implementation.
See Also: equal, compare, relational operators, sequence operations
Technicalia min(s) is a "single-parameter-only-alias" for smallest(s): the compiler front-end automatically substitutes that when only one parameter is provided. Note that providing the return_index parameter prevents such a substitution, ie min(s,true) is just plain wrong, as is smallest(x,y).
Both routines are declared in builtins\pmaths.e (an auto-include). Some extra care must be taken with routine_id("min"), as obviously no equivalent mapping occurs for the one or two parameter cases.

Replace the legacy names minsq() with smallest() and maxsq() with largest(), should you ever encounter them.
There are no sq_smallest or sq_largest routines.

The sq_min(a,b) function applies min() to the top-level elements of a and b when they are both sequences, in which case they must be the same length, otherwise it applies an atom to all the elements of the other, or if they are both atoms it behaves identically to min(a,b). The result is a sequence in all but the latter case. Note that it is not recursive, and passing a single argument to sq_min() is not allowed. sq_min and sq_max briefly found use in demo\rosetta\Canny_Edge_Detection.exw and demo\rosetta\Image_convolution.exw before I replaced them with a (local/private) single-pass sq_min_max_d() function, and generally speaking they have not proved particularly useful, and quite probably something like apply(min) is likely to be a better and more flexible approach/strategy.

The exclude parameter was added for theEditor, so it can keep track of the two longest lines: when not on the longest it only cares should that be exceeded, but when on it would otherwise have to rescan the whole file after every deletion to check it was still the longest - and all solely to keep the horizontal scrollbar correct. Note it already knows the longest, and it is a once-per-start-edit-longest-line thing, in theEditor. In other cases a totally different routine that yields N hi/lo items/indexes might be better.

The Euphoria version of min() only accepts a single argument and behaves (via self-recursion) as smallest(flatten(s)), whereas the Phix version is not recursive.
Unlike the Euphoria version, min(<atom>) generates a runtime error, as does min({}).
This function is only compatible with Euphoria when passed a single argument that is a non-empty and non-nested sequence of atoms.
Unlike several other similar functions the min(), sq_min() and smallest() functions are not recursive.
Should you (for some strange reason) believe that min({"one","two","three"}) should yield 'e', or min({}) should yield +inf, then may I politely suggest that you go away and write your own version of this routine.

The smallest/largest functions differ from the Euphoria versions in std\stats.e in (at least) five ways:
* They can return the index of the lowest/highest entry, rather than only the actual value of said.
* They can optionally ignore a specified entry, effectively yeilding the second lowest/highest entry.
* If return_index is true and s is {}, then 0 is returned, otherwise
* The set passed in the first parameter must be a non-empty sequence (compilation or run-time error if passed an atom or {})
* They can return non-atoms in the set (if no atoms occur, it will be the first/last in an alphabetical and case-sensitive ordering)

None of that should cause sensibly-written code to be incompatible or misbehave (imho/ymmv).
