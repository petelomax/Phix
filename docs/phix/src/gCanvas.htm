gCanvas

Definition: include theGUI.e

gdx id = gCanvas([rtn redraw=NULL,] string attributes="", dword_seq args={})
Description: Creates an interface element that is a canvas - a drawing area for your application.

redraw: a procedure invoked whenever the canvas needs to be redrawn, see REDRAW below.
For more information on the attributes and args parameters see gSetAttributes().
This is a paranormalised function. (see technicalia)

Returns the identifier of the created element.

A gCanvas is also used to implement gGraph, gList, gTable, and gTabs, as well as the resizer in a gSplit.

Note that gCanvas (and gImage/gPixbuf) are aimed squarely at being easy to use and covering all your basic needs, including "fast enough for most things", but not astonishing performance. Everything static should be perfectly fine, but esp rotating polygons may be a tad tardy. On the plus side, I cannot think of anything blocking theGUI for the basics plus something faster, just for the bits that need it, other than that other thing doing the blocking itself, such as GTK4 being arrogantly adamant that everything be wrapped in a meaningless GtkApplication thingy. See this performance summary.
pwa/p2js: Supported. Oooops, SCROLLSIZE and all that jazz isn't...
See Also: gImage, gPixbuf, gRedraw
Example: 
-- demo\theGUI\gCanvas.exw (significantly simplified)
include theGUI.e

procedure redraw(gdx canvas, integer w, h)
    gDrawLine(canvas,0,h/2,w,h/2,-1,1,TG_GREEN)
--  ... plus Arc/Circle/Image/Polygon/Text
end procedure

gdx canvas = gCanvas(redraw,"BGCLR=PARCHMENT"),
    dialog = gDialog(canvas,`gCanvas`,`SIZE=240x156`)
gMainLoop(dialog)
  (compare with gPixbuf)

Other examples include demo\theGUI\GraphR.exw, aaline.exw, and scroller.exw

  h,s,v: hue (0..<360), saturation (0..1), value (0..1). For more info see https://en.wikipedia.org/wiki/HSL_and_HSV
  a: An alpha of 0.0 is fully opaque, 1.0 fully transparent.

atom colour = gGreyscale(atom colour, bool bBlack2Grey=false)
Should be fairly self-explanatory, bBlack2Grey helps prevent originally black disabled text being drawn still too dark.

Note that for instance while yellow is a primary colour, in the RGB/additive (/lighting) colour model (as used by your screen and theGUI) it is made by combining red and green, whereas mixing those (physical/in a can, pot or tube) paints gets brown, likewise mixing blue and yellow paint gets green, since paints are/use the RYB/subtractive (/absorbtion) colour model, whereas mixing blue and yellow in the RGB model end ups with white! It is also the reason why your screen has red, green, and blue pixels but your printer wants cyan, yellow, and magenta ink (unless it predates the 1950s, in which case it might want red, yellow, and blue inks, and that's still the(/a) formal model/theory, just not used in practice anymore).

Also, to achieve consistency between backends, theGUI quietly performs quite a bit of platform-specific RGB⇔BGR mapping automatically, a fact you may need to know when porting some (legacy) code to theGUI. Colour codes are usually declared as atom, however integer is also fine except under 32-bit when the alpha channel is #40..#BF, which is probably fairly rare. Several routines below imply integer colour although it is actually an atom (esp when small) just to save space/keep it neat.

All the standard colours are fully opaque, ie have an alpha channel of #00 (which may get flipped to/from #FF on some backends). Note as yet no attempt has been made to actually test alpha blending at all. [DEV]
Functions: The following routines should only be used on a canvas inside its REDRAW handler, or anywhere/anytime on a gImage or gPixbuf (except gDrawText, and gDrawImage with gdx target). Use gRedraw() to force a repaint.
Non-1 colour/width/style transiently override FGCLR/LINEWIDTH/LINESTYLE, which would otherwise be used.

All co-ordinates are in pixels, with fractions permitted, but simply ignored/truncated, and 0-based, so on a 200x150 canvas {0,0} to {199,149} are valid, with any attempts to draw off-canvas pixels simply quietly ignored, though will probably incur some additional overhead.

gDrawLine(gcp canvas, atom x1, y1, x2=0, y2=0, colour=-1, integer width=-1, style=-1)
  Can also be invoked as gDrawLine(canvas,{x1,y1},{x2,y2}), however any colour/width/style must
    then use named parameters - throws "bad args" should you forget, aka original pre-from-y1 {x2,y2}!={0,0}.
  Draw a line on a gCanvas, gImage, or gPixbuf from {x1,y1} to {x2,y2} inclusive (zero-based).


gDrawArc(gcp canvas, atom xc, yc, w, h, angle1, angle2, integer flags=0, colour=-1, width=-1, style=-1)
gDrawCircle(gcp canvas, atom xc, yc, radius, bool filled=false, integer colour=-1, width=-1, style=-1)
  Co-ordinates are in pixels, angles in degrees, flags can be TG_FILLED with TG_CHORD or TG_SECTOR.
  The angles define the arc start and end, but they are not relative to the center, except for w==h aka a circle.
  The arc starts at the point (xc+(w/2)*cos(angle1), yc+(h/2)*sin(angle1)) and ditto ends/angle2 (always clockwise).
  A complete ellipse can be drawn using 0 and 360 as the angles (note implied *TG_DEG2RAD in the above).
  If angle2 is less than angle1 it will be increased by 360 until it is greater than angle1.
  gDrawCircle() just simply invokes gDrawArc( ... ,w=h=2*radius,0,360, ... ), nothing more than that.
  TG_FILLED has the same value as true, and hence can be passed to gDrawCircle.

gDrawCubicBezier(gcp canvas, atom x1, y1, xc1, yc1, xc2, yc2, x2, y2, colour=-1, integer width=-1, style=-1)
gDrawQuadBezier(gcp canvas, atom x1, y1, cx, cy, x2, y2, colour=-1, integer width=-1, style=-1)
  Draws a curve from x1,y1 to x2,y2 using control points xc1,yc1 and xc2,yc2.
  The (simpler) quadratic version just invokes the cubic one using the same control point twice.

gDrawPolygon(gcp canvas, sequence poly, bool bFilled=true, integer colour=-1, width=-1, style=-1)
  poly: a list of vertices aka starting points of each segment to be drawn.
  If a segment is just {x,y} a straight line is drawn to the end point (ie next/first segment).
  If a segment is length 4 aka {x,y,cx,cy} a quadratic bezier is drawn, likewise length 6 a cubic bezier.
  If a segment is length 8 aka {sx,sy,xc,yc,w,h,a1,a2} an arc is drawn.
    Warning: errors occur should start or end (aka start of next segment) not lie on the ellipse.
    NB: gDrawArc() does not [yet] support any form of rotation, so neither do polygon arc segments.
  The poly can also contain {} which makes it a multi-part polygon, allowing "holes" such as those in O,P,D,A,B, etc.
  Polygon filling occurs on the full collection of segments, rather than each individually.
  A filled polygon may not have a style other than TG_CONTINUOUS or width other than 1.
  bFilled/colour/width/style: assumed to be self-explanatory.
  [Be warned performance is far from stellar, esp on WinAPI, see demo\rosetta\DrawRotatingCube.exw,
   GTK2 is 4* better, no idea why/how GTK3 is 1000*(!) better, we can only hope things improve over time...
   I'm not gutted by 33FPS on WinAPI, slightly jealous of 135FPS on GTK2, blown away by 27,000FPS on GTK3...]
  see|run demo\theGUI\gDrawPoly.exw and/or PhixLogo.exw for some examples.

sequence poly = gRotatePolygon(sequence poly, object angle)
  poly: as gDrawPolygon above but less restrictive, so (eg) you can rotate a single point.
    Will crash if poly contains an arc (or any odd-length) segment: not only do arcs not support rotation,
     at least as yet, the w,h,a1,a2 such segments contain are not rotatable points (unlike the rest).
  angle: rotate all the points of poly. If angle is an atom, rotation occurs about poly[1][1..2],
    otherwise it must be a three-atom {angle,xr,yr} triplet with rotation about {xr,yr}.
    The angle should be specified in degrees, clockwise, usually but does not have to be 0..+/-360.
  [Be warned this seems significantly slower than anticipated, possibly only managing ~150 points per second,
    whereas I would have confidently predicted it was going to be 10 or even 100 times faster than that...
    A cache of previously calculated rotations may help, quite probably rounded to nearest degree/pixel.]

gDrawRect(gcp canvas, atom xmin, ymin, xmax, ymax, bool bFill=false, integer rc=0, colour=-1, width=-1, style=-1)
  Draws a rectangle, optionally filled and/or with rounded corners.
  xmin/ymin/xmax/ymax: inclusive, so {0,0,0,0} draws a single pixel top-left.
  rc: 0 for square corners, else the radius of the a quadrant/arc to use. Unpredictable results when > w/2 or h/2.
  bFill/colour/width/style: assumed to be self-explanatory.
  Replaces cdCanvas[Rounded](Box|Rect)() of pGUI.

gDrawImage(gdm image, gdc canvas, atom x=0, y=0, sx=0, sy=0, integer w=-1, h=-1)
  Draws a gImage, or part of it, onto a gCanvas or another gImage.
  image: (source) must be a gImage.
  canvas: (target) must be a gCanvas or gImage.
  x,y: locate top left corner on the target (fractions rounded to nearest integer).

Attributes:
EXPAND (non inheritable): The default value is "BOTH"/true. The natural size is the size of 1 character.

Note an explicit SIZE on the gCanvas(/gGraph) itself or whatever it expands into is strongly recommended. Said parent container may itself may be inheriting from or expanding into the size of its parent, and of course should any MAXSIZE or MINSIZE be in use, any (initial) size should not exceed those limits. Only the width is critical on a (non-expanding) gList, whereas a gTable has column widths in the column data itself, and the height of both can be deduced from the number of rows, unless that needs to be virtual (as next).
SCROLLFIX A 4-way bit field: 0b00 (default, neither), 0b10 (horiz), 0b01 (vert), 0b11 (both) scrollbars permanent.
 (There are no corresponding named constants, when needed or desired you will simply have to define your own.)

Permanent scrollbars are a matter of taste. Sometimes switching (between tabs, say) can be a little less janky, or comparison of screenshots a little easier, that is without the scrollbars popping in and out of existence.

Returns the integer 0..3 as previously specified.
SCROLLSIZE
Specify or retrieve some (larger) virtual canvas of which the real canvas should show some (smaller) VIEWPORT.
Were you to (eg) accidentally open a video file in theEditor, and that treated it as a text file of 500K lines, some of which were 500K characters long, it might try (and fail) to create a 236TB canvas, of which at most 16MB (0.000006%) would ever be shown, and obviously even if something significantly smaller did actually work it could quite literally be millions of times slower than needed. The file demo\theGUI\scroller.exw allows (after editing) the creation of a 138TB virtual canvas that only actually uses 1.05MB or 0.000001%. Such extreme cases probably aren't of themselves all that useful, by which I mean a scrollsize more than ten times as wide or two hundred times as tall as the viewport essentially becomes un-navigable, so "useful" probably caps out nearer a much more manageable 2GB, but not crashing (the whole machine), not hogging memory, and not running ridiculously slowly, probably are (useful, that is). Should you want an "infinite" canvas, scrollbars become meaningless, so you'd just implement a bespoke viewport without them (or these attributes).

If SCROLLSIZE is not set SIZE is used/returned instead.
It is whether SCROLLSIZE is larger than VIEWPORT (which also defaults from SIZE), or the bit in SCROLLFIX, that determines the presence or absence of scrollbars.

The SCROLLSIZE is automatically handled on gList, gTable (see the rows parameter), and gTree.
In contrast, SCROLLFIX is merely obeyed and never directly set by anything in theGUI.e/js itself.

This setting is entirely independent of MAXSIZE and MINSIZE, and should you (for instance) want to centre a smaller SCROLLSIZE within a larger SIZE that is entirely the responsibility of the REDRAW handler, ditto honoring VIEWPORT. Note however scrollbars are unlikely to look good at very small sizes, and in particular you should probably use a minsize to prevent the height dropping below 40, or better yet 50 - the same could be said for width, but that just happens to need it less often.

See demo\theGUI\scroller.exw for an example (and be amazed at how trivial and easy all this actually is).

Returns {w,h}, and as just mentioned with either being zero (their defaults) independently resorting to SIZE.
SCROLLPORT,
VIEWPORT 		Retrieve or specify which part of a larger virtual SCROLLSIZE should be shown on the actual canvas.
SCROLLPORT and VIEWPORT and just aliases of each other and refer to exactly the same thing.

Automatically handled on gList, gTable, and gTree.

Returns {x,y,w,h} where x,y are initially 0 and indicate the [virtual] pixel position of the top left corner of the canvas, and w,h are from SIZE, reduced to account for any visible scrollbars, with {0,0,w,h} being returned if SCROLLSIZE is not set, or it is smaller than the SIZE, but in the latter case not necessarily strictly enforced.

The scrollbars are drawn as/when needed after the REDRAW handler returns, and that should of course draw the (virtual) contents determined by x,y in the (actual) rectangle defined by 0..w-1 and 0..h-1, with any attempts to draw pixels outside of that area being, as usual, simply quietly ignored (or, to be excessively pedantic, duly soon obliterated by a scrollbar).

Note theGUI meticulously recreates Windows 10 or 11 style scrollbars, but has not been tested on any other version of Windows, nor does it make any attempt to match any Linux style scrollbars, as yet. Also, I will resist calls to replicate any of the stupid styles that make scrollbars "invisible to all but eagles with bionic eyes", at least that is without some explicit setting to turn such stupidity on (with it off by default). Feel free to post images of what you get vs. what you want, the latter from any other app on your system, ideally with and without hover effects, along with both horizontal and vertical examples - see the technicalia dropdowns of gButton(), gCheckbox(), gDropdown(), gMenu(), gTabs(), and gText() for examples of the sort of thing needed.

When setting this attribute, only the origin (first two elements) should be provided. Negative values are not permitted, however they can exceed the natural limits that using the scrollbars would normally attain, so that for instance a gList can "go to line" and show only a few lines, with "dead space" filling up the rest of the visible/actual canvas.

Setting this attribute automatically updates the otherwise private scrollbars. Any and all handling of the latter is achieved via the VIEWPORT attribute. Settings that result in the entire visible area being "dead space" and/or off-canvas scrollbar thumbs are not specifically guarded against, consider yourself warned.
LINESTYLE Default: TG_CONTINUOUS(==1), valid styles (DEV needs work and img redoing) are:
TG_DASHED TG_DOTTED TG_DASH_DOT TG_DASH_DOT_DOT
Note that values other than TG_CONTINUOUS with a line width other than 1 may appear as rhomboids, and perhaps for that same reason native WinAPI/GTK do not support such, but while that effect may be less than ideal it is reasonably acceptable on slightly thicker lines, and fine on horizontal/vertical lines, and just seems churlish to remove it completely.
The style may also be a sequence (of 1/0, see theGUI.e) which sets TG_CUSTOM_DASH(==6).
(TG_CUSTOM_DASH is currently set at the system level rather than per canvas, for no good reason, shout should that sting.)

REDRAW A procedure that is invoked when the canvas needs to be redrawn.

procedure redraw(gdx canvas[,integer w, h[, ox, oy]])
canvas: identifier of the canvas element that needs drawing.
w,h,ox,oy: (optional) same as integer {ox,oy,w,h} = gGetAttribute(canvas,`VIEWPORT`) as 1st line of 1-arg proc.
[When a SCROLLSIZE has been set, it must honour the VIEWPORT.
 Note w,h may be smaller than gGetIntInt(canvas,`SIZE`) to account for the presence of any scrollbars,
 and likewise ox,oy slightly past SCROLLSIZE-VIEWPORT or even explicitly set further beyond that.]

The routine is normally expected to be almost entirely composed of gDrawLine()..gDrawText() statements,
though (eg) gTree() [also] uses it to populate width/widest and (quickly) perform deferred loading updates.
Originally theGUI would crash if you tried to display a canvas without a redraw routine, these days a redraw parameter on creation of NULL or "edge" is replaced by a (private) default drawing routine with the outcome as shown:

local procedure tg_canvas_edge(gdx canvas, integer w,h)
    gDrawRect(canvas,1,1,w-2,h-2)
    gDrawLine(canvas,1,1,w-2,h-2)
    gDrawLine(canvas,w-2,1,1,h-2)
    integer l = length(sprintf("%d",max(w,h)))
    string fmt = sprintf("%%%dd x %%%dd",{l,-l})
    gDrawText(canvas,w/2,h/2,fmt,{w,h},TG_C,style:="Monospace,8")
end procedure

For obvious performance reasons, namely being it can be invoked 50 times per second when manually resizing,
this routine should as far as possible avoid any "heavy lifting", except perhaps in a once-only fashion.

The background colour used for initially clearing the canvas should be set before the first paint event and is applied before redraw() is invoked, and automatically restored after (so you don't have to).
also CLICK, MOUSEMOVE, KEY: All common handlers are supported.

Note the canvas-derived controls gList, gTable, and gTree, specifically those that handle SCROLLSIZE/VIEWPORT automatically, have internal KEY, CLICK, and MOUSEMOVE/WHEEL handlers (for arrow, pgup, pgdown, home, and end keys, dragging, hover effects, and so on - some of which may still yet be sub-par, and/or interfere with or take precedence over user-defined ones in a way they should not), whereas a plain canvas does not have any such default/internal handlers: as usual I will consider adding such if and when they start to become a tediously common need, and not before.

Tehcnically a plain canvas does not have a VALUE_CHANGED handler, and it would never be invoked anyway, but almost all the canvas-derived controls do, and it is simply more trouble that it is worth to try and explicitly ban it for the plain canvas case. In other words, there has been very limited testing in that regard, don't moan should it let you pointlessly set one, or for that matter if it does complain. Several of the demo\theGUI\hXxxxxxx.exw programs (no doubt) actually rely on such pointlessness being allowed and privately handle the actual callback invocation themselves.
Technicalia As this is a paranormalised function, and with no expectation that you should memorise this lot, except maybe "raa", you can assume that all of the following are potentially perfectly valid, with the fairly obvious restrictions that any parameters must be in strict raa-order, and no args without attributes: 

                        gCanvas()
                        gCanvas(redraw)
                        gCanvas(attributes)
                        gCanvas(attributes, args)
                        gCanvas(redraw, attributes)
                        gCanvas(redraw, attributes, args)   -- (the full set)

Note in practice redraw is actually defined as object, and attributes as sequence rather than string, but they are all manually and thoroughly verified to be of the documented types, after repositioning via theGUI/paranormalise_raa().
As mentioned in REDRAW, redraw can also be (the string) "edge", to clarify you want that default doobrie.

Also, in practice, the canvas parameter of gDrawText is not actually gdc but instead declared as a gcps in theGUI.e with an assert failure ("gDrawText cannot be used on a gPixBuf!") should it be a sequence/app-owned gdp, and is reported on the calling line, in preference to a type check failure on the routine definition inside theGUI.e itself.

Compatibility note: On a gCanvas (as shown) {0,0} is TG_NW, logically matching (say) the {x,y} of gShow(), whereas on an IupCanvas() it was/is CD_SOUTH_WEST. In other words the vertical axis is inverted: "height-y" usually works, that is when porting existing code from pGUI/IUP to theGUI, and in some cases, even better, deleting a "height-".

Platform/backend considerations

GTK, bless it, decrees top left to be {0.5,0.5} and (eg) cairo_rectangle(0.5,0.5,0,0) ie width,height of 0,0, draws 1 pixel.
WinAPI, bless it, decrees top left to be {0,0} and excludes right/btm edge, so Rectange(0,0,1,1), ie ltrb, "".
Of course theGUI abstracts such differences away, such that gDrawRect(0,0,0,0) draws the one top left pixel, and maps everything for GTK and other backends appropriately, and for a canvas size of {w,h} valid pixel coordinates are 0..w-1 and 0..h-1. [I could perhaps have made them 1..w and 1..h, but I didn't, and predict I'll thank myself when I get round to OpenGL.] In the same fashion as slices, 1..1 is 1 wide/high, and (eg) 1..0 or 7..6 logically draw nothing, not yet fully tested, though adding a few scattered if>0 as/when/should they turn out to be helpful is not likely to be all that stressful.

Let's take a moment to clear up the ingrained stupidity of the cairo mental model of fill/stroke/align and pixel centres/boundaries. In theGUI, pixels are identified by their centres: an unfilled rectangle from {0,0} to {5,2} is 18-4 crisp pixels, and the filled version is 18 crisp pixels, 6 wide and 3 deep. A one-pixel wide line from {7,0} to {7,2} is a crisp one pixel wide line of three pixels. Let me quickly paint that for you:


The cairo developers have somehow locked it in their heads that you cannot align edges and fills, which I simply don't get: there is a paragraph in the cairo FAQ which neither Stack Overflow nor ChatGPT can adequately explain to me, probably simply because theGUI draws lines then [optionally] fills the resulting shape, whereas GTK treats stroke and fill as entirely separate and independant operations.
The reason that cairo does it this way is so that fills align nicely, at the cost that some strokes do not. It is not possible to set up sample locations in a way so that both fills and strokes with integer coordinates work nicely, so one had to be preferred over the other. One argument in favor of preferring fills is that all fills with integer coordinates align nicely this way. The best that can be done with strokes is to make all even-integer-width strokes align nicely (as they do in cairo) or to make odd-integer-width strokes align (which would then break the fill alignment).
Otherwise cairo is fine, apart from that very serious blunder at the get-go, which of course theGUI shields you from, and of course I get there is no way to fix a thirty year old brainfart. Technically it may be fair to say that odd-width lines on pixel centres (whole numbers) are crisp, likewise even-width lines centred on pixel boundaries (ending in .5 that is), whereas vice-versa are blurry/faded, which is the direct opposite of cairo, however it is probably better to drop "the cario mindset" as much as possible and try and do everything based on pixel centres, perhaps even going so far as to replace even-width lines with filled rectanges, and/or deliberately round/floor/ceil pixel offsets to make the edges crisp. [DEV that all deserves thorough re-testing]

Scrollbars


Scrollbars are automatically displayed when a smaller VIEWPORT than some larger virtual SCROLLSIZE is specified (by you or theGUI itself). Note scrollbars are drawn and managed in pure hll code and do not rely on any native backend functionality, and in particular a VIEWPORT is entirely unconnected with any similarly-named native feature. It was perhaps surprisingly all much simpler that way. One serendipitous outcome during development was that clicking to one side of a scrollbar thumb and holding down did not stop once the thumb hit the mouse position, which it does on WinAPI/Arwen, but carried on like it does on GTK, which I prefer anyway, so I kept that. Some relatively minor platform-specific tweaks may yet be needed.
I did not particularly want and saw no need to replicate the 42 attributes that IUP uses to manage scrollbars, not that I would stand in the way of any genuinely useful and specifically requested ones. In other words it replaces forty-two attributes with three, which I have thus far found not only perfectly adequate but in fact more powerful and flexible.

Aside: Should a menu overlap a scrollbar, hover effects still occur even though the menu is "in the way". I note that Windows Notepad "cheats" a little by hiding the horizontal scrollbar when the menu is displayed, which strongly suggests there is no proper way to prevent that sort of thing.

Scrollbars should act as a pretty good model for faking any other controls: just be aware there is quite a bit going on, from mouse hover effects, timers, and keyboard handling, to clipping the various offsets and hiding things should resizing make them no longer valid or necessary. You can find all the implementation details in tg_redraw_scrollbars() in theGUI.e, ~200 loc. See also Progress / hFake programs.

Anti-aliasing, bezier curves, and polygon filling

I stumbled across a few useful images in my travels and thought I'd include them here, to show you and remind myself how all these things are supposed to work. While GTK and JavaScript have suitable methods already, the raw WinAPI (as far as I can tell) does not perform proper anti-aliasing (and suchlike), so the following should explain what it is supposed to do on all platforms, but really only applies (in actual code terms) to WinAPI, and gPixbufs on all backends.

A bresenham line
An anti-aliased equivalent
Shown magnified 35 times, the above shows individual (square) pixels being used to imitate the theoretical cyan line.
On the left we set seven pixels to the requested colour, in this case black, which results in a jagged/blocky line.
On the right we set seven pairs of pixels to a mixture of foreground/background, making it look much smoother.
Note: should you repeatedly draw the same thing in the same place, for instance in the same invocation of a redraw handler, then the first mixes black and white (or whatever) to get the greys shown, the second black and grey to get darker greys, then black and darker grey to get even darker greys, and so on, ending up as a thick bresenham line. Likewise black text on a parchment background can end up with a bitty red shadow. Such effects, as I have often got in testing, especially of the timing kind, are from a bug in your code, and not a bug in theGUI.
On a steep line segment we'd show each intersection with a y unit axis as two horizontal pixels, whereas as above on a flat line segment we show each intersection with an x unit axis as two vertical pixels. By "unit axis" I really mean (mathematical, horizontal/vertical) lines through the logical centres of the pixels, rather than their edges. Of course it is not perfect, but can almost make a 1080p (1K) monitor look like a (non-anti-aliased) 4K UHD one. Every modern font (for example) is anti-aliased in much the same way, and you would soon grumble were they suddenly not.
The above were nicked from Wikipedia, and should give you a rough idea of how bezier curves work, with the next diagram below hopefully clarifying things further. Note that a quadratic bezier is exactly the same as a cubic with (the same end points and) the two control points being the same. Just in case your brain works like mine, remember that polynomial equations have daft names: unlike the more sensible Monomial, Binomial, Trinomial, Quadronomial, and Pentanomial (matching bicycle, tricycle, and quad bike) which stipulate number of terms but not the highest power, polynomials of degree 0..4 are called Constant (eg 6), Linear (eg ax+b), Quadratic (eg ax^2+bx+c), Cubic (eg ax^3+bx^2+cx+d), and Biquadratic or Quartic (eg ax^4+bx^3+cx^2+dx+e). And yes, I know the Latin word for square is quadratum, but it was still a daft naming choice, at least for English speakers. Quadratic beziers (first two above) are simpler than cubic beziers (last two).

theGUI uses the de Casteljau method to repeatedly bisect a bezier curve until S/Q/E are within one pixel of each other, at which point it (deems it safe to treat as a straight line and) estimates the intersection with a unit x or y axis along the line from S to E, skipping any fragments that don't actually cross a boundary.
For any curve S,C1,C2,E we first find the midpoints L,M,R (via (a+b)/2). We then find the secondary midpoints N and P.
The tertiary midpoint, Q, is on the curve, at the point where the two halves formed by S,L,N,Q and Q,P,R,E meet.

A zoomed in comparison of bresenham and anti-aliased bezier curves. Such a line could of course be drawn in either direction. Notice how the lower/steep part is (naturally) constructed using more horizontal pairs of pixels than vertical, whereas the top/flat part is the other way round. (Originally I planned to only draw horizontals on steep fragments and only vertical on flat, as per Xiaolin Wu's original (straight) line anti-aliasing algorithm, but drawing both turned out much nicer.)

Filled shapes

Filled polygons are achieved using a simple scanline method. First all y axis unit intersections are collected, using a micro fudge factor where needed to ensure no segment starts or ends on an exact y axis boundary, hence avoiding any need to handle special cases. (Consider horizontal scanlines on a diamond shape: obviously mid-line hits count as 1, but (without said fudge factor) direct hits on the two side corners should be counted as 1 hit (despite touching two lines), whereas top and bottom as 0 or 2 hits.)
We sort on {y,x}, verify each y has an even number of hits, then drawline each pair with ceil(first) < floor(second).
Technically, a few scattered +1 and/or +0.5 under GTK make a bit of a mockery of that last statement, but the sentiment remains valid. Plus we need to take a tiny bit of extra care with polygons that run off the screen edges, and sometimes calculate wildly off-screen edges just to keep the number of intersections even.

This is done after the anti-aliased outline, to leave outer smoothing effects but cover unwanted lighter inner pixels.
Fairly obviously and as intended the fill completely obliterates previous content uniformly with the specified colour.
(That may change if/when I get round to supporting alpha channels properly, various API permitting.)

Polygons can be composed of a mixture of straight and curved lines. While not recommended, complex (self-crossing) polygons are supported, and filled using an odd-even rule similar to cdCanvasSetFillMode(canvas,CD_EVENODD): the winding rule would only ever be applicable to polygons constructed from all straight edges, so that is not even attempted in theGUI. (Fairly sure adding winding would be relatively easy if/when needed, on the understanding it'd object to any curves.)
Of somewhat more interest, multi-part polygons (with embedded {}, see demo\theGUI\gDrawPoly.exw for some examples) do not shade any inner holes they might have. Just in case you were wondering, there is no scheme to interpolate hard corners (and if you don't know what that means you can quite safely ignore it).

Dots and dashes look a bif naff especially for curved lines, but that is definitively something for someone else to solve.
