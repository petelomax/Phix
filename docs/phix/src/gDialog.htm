gDialog

Definition: include theGUI.e

gdx id = gDialog(gdx child, [parent=NULL,] [string [title="",] attributes="", [dword_seq args={},]] bool bEsc=true)
Description: Creates a dialog element, also known as a window, which contains and manages user interaction with the interface elements. For any (other type of) interface element to be shown, it must ultimately be encapsulated in a [headerless] dialog.

child: the identifier of an interface element. The dialog has only one child.
parent: must be specified at creation time, or NULL (/omitted) for top-level windows.
title: optional, can also be set via `TITLE="xxx"` in the attributes parameter, or even later.
For more information on the attributes and args parameters see gSetAttributes().
bEsc: sets CLOSE_ON_ESCAPE (additional notes in technicalia).
This is a paranormalised function (see technicalia)

Returns the identifier of the created element.
pwa/p2js: Supported.
See Also: gShow, gHide, gMainLoop, gRedraw
Example: 
-- demo\theGUI\gDialog.exw
include theGUI.e
gdx dlg = gDialog(NULL,"gDialog","SIZE=240x80")
gMainLoop(dlg)
Notes: All elements (including menus) must be inside a dialog to interact with the user.
An interface element will only be visible if the containing dialog is also visible.

Technically the child can be NULL, as shown above, but that is only useful for test/incomplete applications, since (unlike pGUI) there is no way to insert or reparent controls, and besides it is pretty easy to show/hide elements.
(Aside: NULL without an explicit SIZE is formally unsupported, yielding platform-specific or even random results.)

A parent must be supplied on all child windows, to make it a popup and restore focus back to that parent window when the child closes. Closing any window with a NULL parent (at least on Windows) terminates gMainLoop(), though at a push you could perhaps have a top-level loop of your own around that, not that I've tried.
Attributes:
CLOSE_ON_ESCAPE When true (the default) the dialog is automatically closed when escape is keyed.
DEFAULTENTER,
DEFAULTESC 		NULL, TG_CLOSE, TG_IGNORE, or the gdx of a button whose ACTION handler should be to be invoked when the user presses (an otherwise unhandled) Enter/Escape when the keyboard focus is on any control within the dialog. Alternatively or when no suitable button exists just use a KEY handler on the dialog.
Note that DEFAULTESC takes priority over bEsc/CLOSE_ON_ESCAPE.
HDR Show/hide the dialog titlebar (creation only, see also technicalia). Default: true.
MNZ,
RST,
CLX 		MNZ: (not GTK) show/hide/disable the minimse button. Default: true on desktop/Phix, false in JS.
RST: show/hide/disable the maximise/restore button. Default: true.
CLX: show/hide/disable the close button. Default: true.
Not all backends support all combinations, see technicalia.

Aside/minor point: In the rather pointless test case of gDialog(NULL), setting RST to false under GTK completely messes up the size, that itself fairly pointless and even weirder-looking on GTK on Windows, but it should be fine when the dialog has a non-null child.
FULLSCREEN,
MAXIMISED,
MINIMISED,
MAXIMIZED,
MINIMIZED 		Sets or indicates whether the dialog occupies the whole screen over any system bars, or is maximized or minimized. MAXIMISED differs from FULLSCREEN in that decorations (ie titlebar and border) are visible when maximised but not when fullscreen, likewise with the system tray and similar, depending on your setup. S/Z are simple aliases.
See additional notes in the technicalia.

Some links to the older pGUI equivalents FULLSCREEN and MAXIMIZED, should you be looking for them.
MARGIN,
PADDING,
EXPAND 		There are no such things on a dialog itself, however like all containers a dialog is responsible for handling child margin[s], hence instead specify a MARGIN on the (sole) child for some space around the edges of a dialog, assuming of course that is a gH/Vbox, which also has the additional options GAP and SPACE. EXPAND is meaningless on a gDialog: there is nothing for it to expand into (you may be thinking of resize). A dialog is given a natural size sufficient for its contents, unless explicitly overidden by the SIZE, MINSIZE, and/or MAXSIZE attributes.
MODAL (creation only) Only meaningful for windows that have a non-null parent. Default: true
A modal window must be completed or cancelled before you can do anything else, eg a file open popup.
TITLE Dialog title. Default: "".

Note that Windows 11 has introduced a mini-drag bar for non-maximised and parent-less title-less windows, so don’t be blaming me for that (to be fair I think that’s sensible).
VISIBLE Returns true or false. Calling gShow or gHide for the dialog fairly obviously implicitly updates this attribute.
also ACTIVE, FONT, MAXSIZE, MINSIZE, SIZE, TIP, Note ACTIVE and FONT also affect all controls inside the dialog.
Handlers: Set using eg gSetHandler(dialog,"CLOSE",close_handler)
CLOSE Called just before a dialog is closed when the user clicks the close button of the title bar or an equivalent action.
NB: other handlers returning TG_CLOSE do not invoke any CLOSE handler, instead any required final processing, such as saving files, must be performed manually/explicitly first.

function/procedure close_handler([gdx id])
id: (optional) identifies the dialog being closed.

Fairly obvously it may not be named close(), but you're miles better of using say close_delivery_address_lookup() or close_main_window() anyway.

Returns: TG_IGNORE prevents the dialog from being closed. For all other return values, and when a procedure, the dialog is closed.

There is also the builtin file close() procedure, and the CLX attribute above, were you looking for them.

Technicalia As this is a paranormalised function, with a non-optional child, and with no expectation that you should attempt to memorise this lot, except maybe "cptaab", you can assume that all of the following are potentially perfectly valid, bar the commented out one in the middle, with the fairly obvious restrictions that any parameters must be in strict cptaab-order, and no args without attributes: 

                        gDialog(child)
                        gDialog(child, attributes)        -- (NB attributes must contain >=1 '=')
                        gDialog(child, attributes, args)
                        gDialog(child, attributes, args, bEsc)
                        gDialog(child, attributes, bEsc)
                        gDialog(child, title)                       -- (NB no '=' in [any] title)
                        gDialog(child, title, attributes)
                        gDialog(child, title, attributes, args)
                        gDialog(child, title, attributes, args, bEsc)
                        gDialog(child, title, attributes, bEsc)
                        gDialog(child, title, bEsc)
--                      gDialog(child, bEsc)                -- (NB invalid/mistreated, see below)
                        gDialog(child, parent)
                        gDialog(child, parent, bEsc)
                        gDialog(child, parent, attributes)
                        gDialog(child, parent, attributes, bEsc)
                        gDialog(child, parent, attributes, args)
                        gDialog(child, parent, attributes, args, bEsc)
                        gDialog(child, parent, title)
                        gDialog(child, parent, title, bEsc)
                        gDialog(child, parent, title, attributes)
                        gDialog(child, parent, title, attributes, bEsc)
                        gDialog(child, parent, title, attributes, args)
                        gDialog(child, parent, title, attributes, args, bEsc)   -- (the full set)

In practice parent, title, attributes, and args are actually defined as object, but manually and thoroughly verified to be of the documented types, after being repositioned via theGUI/paranormalise_ptaab(), which isn't actually passed the non-optional child parameter.

bEsc, additional notes

Attempting gDialog(child,bEsc) would be mis-treated as gDialog(child,parent), since false is the same as NULL aka no parent, and true is the same as 1 which (almost) could be the valid id of a parent dialog, whereas any and all of the following should be fine:
   gDialog(child,NULL,bEsc),
   gDialog(child,"",bEsc),
   gDialog(child,bEsc:=bEsc),
   gDialog(child,"CLOSE_ON_ESCAPE=%t",{bEsc}), or even
   gDialog(child) with a later gSetAttribute(dlg,"CLOSE_ON_ESCAPE",bEsc)

Typically only the main dialog on a non-trivial program needs bEsc setting, and it is pretty rare you would not want to specify a title or other attributes, so that one problematic case (child,bEsc) almost never occurs anyway. It should perhaps go without saying, but even in an application such as an editor which would be pretty much unusable if escape closed the main window, there are almost always many more sub-windows where close is the desired behaviour, and it is clearly trivial to override to false when needed, which is why the default for bEsc is true.

Unlike pGUI there is no PARENTDIALOG attribute in theGUI, either global or dialog-specific, so should one be needed it must be specified up-front. It may be necessary to restructure some (legacy) pGUI programs such that the main dialog is created before any child windows, possibly by putting the creation of child dialogs into a subroutine (or two) that are only invoked at or after the point where the old code was trying to set PARENTDIALOG. Likewise there is [as yet] no suppport for reparenting, or any way to insert or append controls, besides it is far easier to show/hide elements, or maybe do that with two otherwise identical dialogs, or perhaps even destroy (not yet implemented and possibly never will be) and re-create a dialog (ditto).

While you can set the dialog title via `TITLE="xxxxxx"` in the attributes parameter, it can also be specified separately up-front, for four simple reasons: it is almost always wanted, you can avoid the "TITLE=" bit, thirdly you can avoid the need for nested quotation marks, especially when there are any spaces involved, and fourthly in quite a few of my demos, at least, the title is a separate constant anyway. Note however that a title specified separately cannot contain an '=' character, since that is used to distinguish it from an attributes string, and the internal paranormalise_ptaab() routine strictly validates that, and further there is no sprintf() shorthand on the title parameter, whereas there is one on the attributes parameter (ie the args parameter). See also this note.

You could also set the title much later, in a separate gSetAttribute[s]() call. It is not actually an error to never set it, just fairly unlikely to be what you genuinely want, unless you are hiding the whole title bar (as per FULLSCREEN or some as yet unattempted "same size" method) and even then a hidden name could still prove quite useful for debugging purposes.

HDR/MNZ/RST/CLX, additional notes

A headerless window can be and is used for the dropdown on a gDatePick and gDropDown, and menus.
Displaying desktop-like window headers in a browser probably isn't really all that useful, beyond delivering a sense of comforting familiarity the first time you run a desktop-developed program in a browser, since a browser shows single pages/apps per tab rather than a shared desktop with a taskbar, but child dialogs generally still need one. Without a taskbar, there is no practical/builtin mechanism to de-iconify a window, hence theGUI.js doesn't even have a minimise function, and closing a window may or may not close the browser tab, depending on how the page was opened and the "security interpretation" of the particular browser being used. Many top-level dialogs in a browser might simply be better off shown headerless.
Of course fullscreen games generally don't want window decorations cluttering up the screen either.
Note headerless windows cannot be moved or resized, that is via mouse/keyboard (unless re-implemented).
Technically toggling titlebar display is do-able, but fairly likely to trigger platform-specific niggles.

Disabling the close button may be beneficial in a browser, where closing the browser tab is probably more appropriate anyway, but not in general a good idea on desktop/Phix, unless perhaps relying on ESC.

FULLSCREEN/MAXIMISED/MINIMISED, additional notes

While you can toggle between MAXIMISED, normal, and MINIMIZED any way you like, FULLSCREEN is a somewhat separate thing such that although you can toggle between maximised/normal and full screen, direct switching between minimised and fullscreen is not formally supported in either direction: should you ever for some strange reason feel the need to do that, you are expected to use a two-step process via maximised/normal explicitly yourself. (Not supported includes not caring one bit should it seem to work in a few rare cases.) Also note that consulting before display can yield nonsensical results, such as all three true, unless you have been scrupulous in explicitly turning the others off - it uses a simplistic last-first scan of the internal "to-do" list.

Note that under WinAPI, theGUI uses plain windows, not "dialogs", as traditionally defined in a resource section, something Phix does not have/use. While dialogs (and making windows behave more like dialogs) offers some goodies, such as tabbing between controls, they take away just as much, such as making VK_ESC look for an IDCANCEL button (and should I define a button, I'd rather give it my own id, thank you very much). In the end, the code that explicitly manages tabbing between controls was needed for GTK as well anyway, since that (implicitly assumes a tab order and) sometimes gets it quite crazily wrong, plus the migration to fully emulated controls made it mandatory to manually re-implement all that sort of stuff anyway.
