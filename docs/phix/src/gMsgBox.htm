Description: Display a modal popup containing an optional image, message, and one or more buttons.

title: (optional) text to be shown on the popup window's header. Default: "Error".
msg: (mandatory) text to be shown on the label, msg:=sprintf(msg,args) occurs when args!={}.
buttons: (optional) sequence of button text(s). Default: {"OK"}.
handler: (mandatory when >1 button defined) a procedure to be invoked when a button is clicked, see HANDLER.
image: (optional) a gImage to be placed before the msg.
This is a paranormalised routine (see technicalia)

Notes: This is a relatively straightforward (hll) convenience wrapper for gShow() on a (single/private/reusable) gDialog containing a [not always visible] gCanvas, gLabel, and one or more gButton(currently max 5). Should this not meet your precise needs, simply roll your own.

For JavaScript compatibility reasons this does not "stop the world", instead control returns immediately and normal processing continues, which may require a certain amount of refactoring when porting legacy code to theGUI. Specifically, i = message_box(), i = messageBox(), i = IupAlarm(), and i = IupMesageAlarm() must be reworked to perform any required actions based on the value of i inside the (new) handler, rather than immediately next in the code, along with any subsequent actions, such as invoking abort(). ? Attempts to display more than one gMsgBox at the same cause them to be "queued" such that closing the currrently open one immediately triggers display of the next in line.

btn: a 1-based index corresponding to buttons.

? The popup is automatically closed before the handler is invoked.
An error occurs when there are multiple buttons but no handler has been specified.
A common handler can be used, just so long as it can determine unequivocably what btn[1..5]/title actually means, bearing in mind theGUI might be sitting on a stack of things you've asked it to display, so they will not necessarily relate to the last thing asked for.

There is also a HANDLER on a gMenu, were you looking for that.

Don't read too much into {button[s]} - it's just a pseudo-type-notation that helped me clarify a few things.
Note that in practice all parameters bar title are actually defined as object, but everything is verified to be of the documented types (or NULL in the case of image), after being repositioned via theGUI/paranormalise_tmabhi().

Obviously when two dword-sequences are passed they are assumed to be args, buttons. When only one is passed it determines whether it is args or buttons by the presence or absense of one or more % characters in msg. A handler is >0 whereas an image is <0, which helps distinguish those when only one or the other is specified, and lastly "only one string" is always taken to be msg rather than title.
