match

Definition: integer i = match(object needle, sequence haystack, integer start=1, bool case_sensitive=true)
-- or --
integer i = rmatch(object needle, sequence haystack, integer endx=length(haystack), bool case_sensitive=true)
Description: Try to match needle against some slice of haystack.

match() returns the positive subscript where the (first at or after start) matching slice begins, or 0 if none.
rmatch() returns the last/rightmost (ending on or before endx) slice beginning rather than the first occurance.
In both cases, when needle is "the", a non-zero result would locate the appropriate 't'.
pwa/p2js: Supported.
Notes: By default [r]match is case sensitive, so "the" does not match "THE", unless case_senbsitive is false.

You can override the effective start of haystack in match() and its effective length in rmatch(), with the usual support for negative subscripts (in practice endx actually defaults to -1), however neither routine lets you override both. Overriding the start/end may improve performance by avoiding expensive slices on a very long haystack, and make subsequent use of the result simpler, eg match("the",txt[35..$]) needs +34 to subscript txt, whereas match("the",txt,35) needs no adjustment.

If needle is an atom (unlike Euphoria, which crashes) then match() and rmatch() behave as find() and rfind() respectively.

The match_from() routine is deprecated and in fact now completely and permanently removed, simply use match() with the optional third parameter instead.
Auxiliary Functions bool res = begins(object sub_text, sequence full_text, integer start=1)
bool res = ends(object sub_text, sequence full_text, integer endx=length(full_text))

While sub_text can be a string or character, as per examples below, and it may be easier to think of it in that way, there is nothing wrong with say begins({{2,2}},{{2,2},{3,2},{7,1}}) (which yields true), however note that while begins('a',"ab") and begins("a","ab") both yield true, begins({2,2},{{2,2},{3,2},{7,1}}) results in false, for much the same reasons that "ab"=={'a','b'} but "ab"!={"a","b"}, namely there being no slice of {{2,2},{3,2},{7,1}} which is {2,2}, and/or these routines handling atom first args specially.

Code such as match(word,txt)=1 is going to slog through a long txt before returning (say 21573, or 0) and may as an obvious consequence be literally thousands of times slower than begins(word,txt).

Note that a sub_text of {} or "" always yields true (aka ask a silly question...).
Example 1: 
?match("hi","Phix") -- 2
?match("the","the cat sat on the mat", 4) -- 16 (not 1)
?rmatch("the","the dog ate the steak from the table.") -- 28
?rmatch("the","the dog ate the steak from the table.", -11) -- 13
?begins("abc","abcdef") -- 1 (true)
?begins("bcd","abcdef") -- 0 (false)
?begins("bcd","abcdef",2) -- 1 (true)
?begins('c',"catamaran")  -- 1 (true)
Example 2: 
line = line[1..match("--",line)-1]  -- remove comment
-- A nifty clever clogs trick: if match returns 0 then line is unaltered, 
-- (ie line:=line[1..-1]), otherwise only the text before "--" is kept.
Example 3: The following illustrates the correct use of the start/endx parameter for match() and rmatch(): 

string mm = "tatatatatat"
integer tt = 1
sequence mr = {"fwd"}
while true do
    tt = match("tat",mm,tt)
    mr = append(mr,tt)
    if tt=0 then exit end if
--  tt += 1 -- overlapping
    tt += 3 -- non-overlapping
end while
mr = append(mr,"rev")
tt = length(mm)
while true do
    tt = rmatch("tat",mm,tt)
    mr = append(mr,tt)
    if tt=0 then exit end if
--  tt -= 1 -- overlapping
    tt = max(tt-3,0) -- not
end while
?mr -- output:
    -- +/-1: {"fwd",1,3,5,7,9,0,"rev",9,7,5,3,1,0}
    -- +/-3: {"fwd",1,5,9,0,"rev",9,5,1,0}

Adjusting tt by +/-1 will find overlapping matches, +/-3 [ie length("tat")] non-overlapping ones.

Note that while it is fine to let start run past the end of haystack when using match(), an additional guard may be needed for rmatch() to prevent endx going negative. [There is a very strong probability it would start retreading what you've already done in an infinite loop, and trying to use negative subscripts all along (relying on flying past -length being ok, which technically it would be) does not really help since rmatch() always returns >=0. You could of course replace the (second) while true do with while tt>0 do instead of the max() call.]
