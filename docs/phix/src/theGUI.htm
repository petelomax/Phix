theGUI

theGUI is The Hyper Easy Graphical User Interface.

WARNING: Incomplete, very much still in progress (15/3/2026), p2js support badly broken - theGUI.js needs a complete rewrite.
  • Easy to learn and use - the API was inspired by that of IUP but is even simpler (about four times imo).
    At the time of writing theGUI (but not including OpenGL) has just 140 global constants and 105 entry points, plus
    to be fair (but still not including OpenGL) 114 named attributes and 20 named handlers, a combined total of 379,
    or just 356 if not counting aliases, which must make it one of the smallest and easiest ever. In comparison:
    pGUI (which it is intended to replace) has ~450 entry points, 640 global constants, over 70 named handlers
    and well over 500 named attributes, which makes it about 1,660 in all (w/o OpenGL).
    The GTK3 dlls contain (I can back this up) at least 21,774 public entry points - and that's not including any
    of an equally frankly frightening number of constants, structures, and callbacks (or OpenGL).
    The mini.chm I built ages ago from MSDN has 22,451 pages and indexes 12,135 things from WinAPI, no
    OpenGL, and not all of which is going to be GUI related, though fwiw my gut instincts say maybe around ⅔ is.
    The JavaScript DOM might actually boil down to something more reasonable, all I can say is that my time in
    the browser development tools is generally utterly dominated by paging through seemingly endless
    screenfuls of functions and attributes and nested prototypes on each and every element node.
    Of course (other search engines/sites exist) googling things and searching StackOverflow is pretty easy, but
    generally speaking for the most part only after you already know the names of things to look for.
    The whole point of theGUI is not so much to be complete, but instead easy to fix and extend. While anything static
    should be fine, some moving things (esp rotating polygons) may be a tad sluggish, but should improve over time.
    It may not have all the bells and whistles, however I reject the apparently common premise that a "proper" API must
    be tortuously complicated and difficult and confusing to make it flexible and powerful. Other libs will no doubt be
    better at blitzing billions of bits but for boring business basics I am pretty confident this lib and these docs will
    knock spots off anything similar you have ever seen.
  • No reliance on native controls, except for menus, or resorting to the lowest common denominator (WIP).
    It seems a slam-dunk decision to use platform-native controls, and in fact the first verion of theGUI did precisely that, however it turns out to be completely wrong. Of course I fully agree that the user should hardly notice the use of emulated rather than real controls, but near-perfect emulation is actually much easier than first feared. With native controls you end up fighting to make them behave consistently and find the lowest common denominator, which all goes away with emulated controls. Resizing and layout management becomes trivial, in comparison with sending messages to individual controls and watching it propagate knock-on changes in a completely unwanted fashion. One of the most difficult problems with using native controls was figuring out their exact size and what secret margins they might have, which suddenly ceased to be any kind of issue at all. Dealing with mouse and keyboard input on emulated controls is surprisingly trivial, at least that is after the one-off cost of making the basic CLICK, KEY, MOUSEMOVE, MOUSEWHEEL, and (perhaps) MOUSELEAVE handlers as cross-platform consistent as possible, as opposed to dealing with a plethora of control-specific messages, especially those with poorly documented but critical "high-bits" flags. Still very much a work-in-progress, and admittedly I haven't yet finished anything with a pop-up, or written my own versions of hit-tests and focus handling (and perhaps MOUSELEAVE).

    I don't normally like advertising vaporware, but it's clearly the way things are almost inevitably going.

    If you look in the technicalia of gButton, gCalendar, gCheckBox, gDialog (for js only), gDropDown, gMenu, gSpin, gTabs, or gText, you should see some screenshots of native controls taken to implement their emulation, and you should take those as a guide for what I need for any other look and feel you would like me to try and recreate. In particular I need to be able to zoom in and count pixels and sample their colours, including any hover-effects and the like. Try and avoid jpeg and other lossy image formats that will only serve to distort things, and of course crop to the minimum size, rather than fullscreens that I don't need or even know where I should be looking. Sometimes you will indeed need to take a fullscreen image and crop it, for instance drop-downs that extend past the window boundary, or that vanish as soon as the Alt key is pressed. Some words along with the pictures will of course help, but I'm sure you know the age-old adage. I should perhaps note that I'm personally not too keen on wasting time on p(r)etty little animations, that is of the stock-control-element variety, and have no advice on capturing those micro-second existant images or timings, but I'd try and help someone else out as best I could.

    There are several hButton..hTree.exw demos that contain all the things involved, with the latter clocking in at one and a half thousand lines or so, which is probably the most complicated it gets. Of course once a control has been emulated on one platform, the next is a fairly straightforward variation on that, in contrast to reading documentation which is quite often more convoluted for a single control than these entire [theGUI] docs!
  • With the deepest regret, unfortunately there is at best only very limited support for assistive technologies.
    My sincere apologies, theGUI applications are unlikely to score well on accessibility, in particular screen readers.
    Were it a simple matter of registering a callback that could answer queries, I'd be all over it, but it seems it must,
    on Windows be some full-blown COM object (apparently an IUIAutomationProxyFactory that creates IRawElementProviderSimple
    interfaces)     that would likely take me over a decade to master, what little I've read about GTK suggests it may need
    GTK4 and even then a GtkAccessible interface that only works on GtkWidget derived controls, and alas semantic
    HTML for emulated controls on a single canvas simply isn't possible. Any actually useful suggestions welcome.

    OTOH, it should be fairly easy to write (say) theGUI.AT.e which (amongst other things?) logs a focus callback to announce their tips,
    at least that is by someone who knows what is actually needed/helpful and what system settings should be honoured, and perhaps
    when settled make that a permanent part of theGUI.e/js. However it'd be part of the app itself rather than any third party add-on,
    which means of course even should it end up being better than anything else currently available, it'd likely still "score" a big fat 0/10
  • Minimal dependencies - requires a working GTK install on Linux, or just a Windows OS, or just a browser.
    I should probably admit that all development and testing has occurred on Windows 10/11 (plus VirtualBox).
    I should also say I'm only testing GTK2/3, with GTK4 (and prettiness) left as an exercise for someone else.
    Oh, you do actually need some kind of computing device, and it currently has to have an x86/64 chip inside,
    though one of the main motivations for starting theGUI was making ARM a worthwhile target for Phix version 2.

    Currently targets Windows (my main focus is to get that perfect), Linux (via GTK), and a web browser (for the
    most part, I'm only concerned about single-canvas under p2js). Android/iOS support will (one day) be via PWA
    (Progressive Web Application) since bundling the slightly-tweaked p2js output with five or so other files is a slam
    dunk compared to the utter madness that is Java/Kotlin/Flutter/Gradle and the like, plus you can convert a PWA
    into an APK and (hopefully) upload it to Google Play, the Microsoft Store, or iOS App Store, not that I personally
    plan on spending any money to prove that can be done.
  • Zero build overhead - no additional compiler/toolchain needed, just edit & go.
    Of course some bugs may be difficult to find/remedy, but if you can edit theGUI source code, you're done,
    unless you need the same fix in both theGUI.e and theGUI.js, which is (hopefully) also a pretty trivial step.

    Note however targetting Java/Android will likely be quite the faff, should that ever become an actual thing.
  • Using a GUI should ideally be perhaps 5% of the effort required to write a (real) program, not over 50%.
    Nuff said. Of course most trivial demos here such as next, or a gGraph with static data, are inevitably the full 100%.

Example

include theGUI.e   -- demo\theGUI\gLabel.exw
gdx lbl = gLabel(`This is a label`),
    hbx = gHbox({lbl},`SPACE=AROUND`),
    dlg = gDialog(hbx,`gLabel`,`SIZE=240x80`)
gMainLoop(dlg)
For the whirlwind aerial tour, visit Elements and click on each of the twenty-one images to see the code that made it.
You should also be able to run Phix\pdemo.exw (currently still pGUI), filter on theGUI.e, and run each of the examples that shows.
Also try demo\Edix\theEditor.exw (very much an empty shell in which I plan to start sharing as much gui-agnostic Edita code as I can).

Roadmap / todo

  • Current focus is on replacing all the "native" controls with canvas-based ones (as per second point above).
  • gTabs: insert/delete/swap tabs (might now be trivial, needed for theEditor)
  • get SCROLLSIZE (etc) working in a browser (hTable/h_scrollbars.e seems the way forward).
  • port every existing pGUI example to theGUI, and of course get all that can working in a browser.
  • Work on integrating with OpenGL not started at all, but hopefully that should prove reasonably superficial.
  • Speaking of integrating, pdemo should probably also allow open in p2js, or via that in browser (as well as desktop/Phix).
  • theGUI.js is always likely to lag behind theGUI.e: in my experience desktop->browser is far easier than browser-first.
  • Ultimately, this paves the way for Phix on ARM, since with no viable GUI that would be pretty pointless.
