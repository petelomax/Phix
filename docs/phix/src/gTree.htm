gTree

Definition: include theGUI.e

gdx tree = gTree(sequence nodes, string attributes="", dword_seq args={})
Description: Creates a nested tree control with expandable/collapsable nodes from a simple [recursive] data structure.

nodes: a nested or flat version of the tree, sets DATA.
For more information on the attributes and args parameters see gSetAttributes().
Note standard optional parameters perfectly suffice, no paranormalisation is performed or required.

Returns: the identifier of the created element.

Example: 
include theGUI.e    -- demo\theGUI\gTree.exw
sequence nodes = {"Figures",true,   -- branch with 6 children
                  {"Other",         -- leaf
                   {"triangle",     -- branch with 3 children
                    {"equilateral",   -- (no true = closed)
                     {"isoceles",{"acute","obtuse"}},
                     "scalenus"}},
                   {"parallelogram",true,{"square","diamond"}},
                   "2D",{"3D",{}},  -- 1 leaf, 1 empty branch
                   {"4D",{"some","other","branch"}}}}
gMainLoop(gDialog(gTree(nodes),`gTree`,"SIZE=240x180"))
Notes: In the nested/recursive form shown above, the whole tree and every subnode must be in the form:

desc or {desc[,expanded,[checked,[image,[userdata,]]]],children}

where desc is the descriptive text string for the node, with just-string being interpreted as a leaf,
expanded is false (the default), true (ignored on leaf nodes), or -1 for deferred loading (see DATA),
checked is false (the default) or true, ignored when CHECKED is false,
image is NULL (the default) or a gImage or an xpm (ie string/sequence), ignored when IMAGE is false,
userdata (default NULL) can contain anything the application needs, per-node (cf the per-tree USER_DATA),
children is always the last item and (when expanded!=-1) can be 0 to signify a leaf, or a sequence of sub-nodes.

In other words there are up to four optional items mid-node, some of which may need prior dummy values, at worst 0,0,0, and should node not be a just-string leaf node, length(node) must always be 2..6 inclusive.
Lastly, should userdata only be needed for deferred loading you can put it in the children slot instead, when expanded==-1, and in fact {desc,-1} is a special case of deferred loading with (presumably) no userdata needed, and {desc,-1,stuff} would mean you're getting your userdata from the children slot in the DATA handler.

The nested form is automatically converted into a flat internal one as detailed in the technicalia.
Attributes:
CHECKED If true show checkboxes against every node. Default: false.
The identically-named CHECKED handler, if present, receives individual node notifications,
or the DATA can simply be fetched and scanned for checked items at any time.
There is also a CHECKED attribute on a gMenu, in case you were looking for that.
DATA[+] As per the nodes parameter you can set this in either nested or internal flat form.
Returns the internal form plus a set of the internal control routines for more advanced use, see technicalia.
The (internal form) data is deemed theGUI-owned: should you feel the need to modify it yourself, other than via the methods given in the above advanced use link, fetch "DATA+" (which nullifies the internal copy), update, and put it back pronto, and when necessary deep_copy() any copies kept for debugging purposes.
The identically-named DATA handler is, in contrast, used exclusively for deferred loading.
There is also a DATA attribute on a gTable, and a DATA handlers on a gList and gGraph, were you looking for them.
IMAGE If true (the default) show an image against every node, using stock icons if none provided (as above).
It may also be a 16x16 gImage, which replaces the stock "open branch" image on all expanded branches,
that is, unless the branch already has an {open,closed} pair of images itself. No images shown when false.
It may also be {open[, closed[, leaf]]} to specify up to three stock images for nodes without those specifically set.
There is also a IMAGE attribute on a gButton and gMenu, in case you were looking for them.
SELECTED Retrieve/set a node index to the internal form, which almost certainly requires DATA be retrieved to be of any use.
Can be retrieved on demand or the identically-named SELECTED handler gets notified when changed.
There are also similar SELECTED attributes/handlers on a gList and gTable, were you looking for them.
SELFULL Specify whether the full line should be highlit when selected or just the text. Default: true.
also ACTIVE, BGCLR, BGSEL, BGHOV, CANFOCUS, FONT, EXPAND, MINSIZE, MAXSIZE, SIZE, TIP, VISIBLE.
Note that USER_DATA is per-tree and unrelated to the userdata on each node.
Handlers: Set using eg gSetHandler(tree,"CHECKED",checked) [in that one case having also enabled it with gSetAttribute(tree,"CHECKED",true)]
CHECKED Action generated when a node is (manually) checked or unchecked.

procedure checked([gdx id,] sequence nodes, integer ndx)

id: (optional) identifier of the gTree element that activated the event.
nodes: A sequence in the internal format, of the entire tree.
ndx: The index of the item which has just been checked or unchecked.

Only ever invoked when the identically-named CHECKED attribute has been set to true, but that does not mean you also have to have a handler as well: the DATA (ie the same thing as nodes) can be fetched at any time, and:
To determine all checked nodes, assuming >1 may have been checked, you must iterate all down the [flat] tree.
DATA A deferred loading handler, invoked for any branches with expanded set to -1, when actually needed and not before, which is usually when the branch is first displayed (vs first expanded, ie perhaps one step sooner than you might first imagine).
Deferred loading is not worthwhile for a tree the size of the one shown above, but it certainly could be when loading a hard drive, eg right click on C:\Windows, select Properties, watch it start counting and imagine staring at a blank screen until it finishes - still quite noticeable even on a decent SSD, but back in the day on spinning rust it often took over ten minutes! I should perhaps note that Windows 11 (and maybe 10) incoporates a drastic improvement over previous versions for a similar operation on C:\, technically a bit of a downgrade in that it is much faster but there is no file count, and they may yet extend that everywhere.

sequence subbranch = defer_load_data([gdx tree,] sequence subbranch)

tree: (optional) the gTree to which this applies.
subbranch: a sequence of length 6 (a "filled-in" version of the length 2..6 nested node), as per the following: 

    global enum TGTDL_TEXT,         -- 1
                TGTDL_EXPANDED,     -- 2    
                TGTDL_CHECKED,      -- 3
                TGTDL_IMAGE,        -- 4
                TGTDL_USERDATA,     -- 5
                TGTDL_CHILDREN      -- 6

The text can be used for sanity checks or updated from say "loading...".
The expanded flag will be -1 and you *must* set it to true or more likely false.
[Should you set all expanded flags to true it would immediately defer-load the lot.]
Checked, image, and userdata can likewise be overidden now.
The real payload needing replacement however is children, of course, not that it actually always has to be.

It is/was your choice whether the information needed now was stored in the children or userdata slot.
The latter is I suppose a more logical place to put such things but may need a dummy checked/img to be specified before it, whereas children is perhaps more simply always last [in the nested form], and a perfectly reasonable way to effectively auto-discard info that will never be needed again.

While the nodes parameter as specified when the control is first created and/or DATA can be either nested or internal form, the returned subbranch[TGTDL_CHILDREN] must however always be in the nested form (besides, getting all those numeric indices spot on when you're half way down a table you don't even own does not sound like fun, and you should be grateful theGUI is doing all the vnc updates, not you). It is in fact permitted to return 0 in [TGTDL_CHILDREN], on the understanding the children themselves have already been specified (briefest of checks on that only) and you were actually deferring some other work needed on text..userdata, for instance you might be using fancy images and not want to generate thousands of them up-front, and/or prioritise the ones the user is actually looking at.

Note that deferred loading might be invoked one step earlier than you might first imagine, for instance the above example could be written with nodes of {"Figures",-1}: even when closed the gTree would not know whether it was a leaf or a branch, and in the latter case whether it had zero or more children to determine whether a '+' should be displayed. Similarly should you display all the drives on your system, it would likely want to perform a top-level dir() on all of them first, to get their display correct. It would be perfectly permissible to initially create the tree with dummy empty closed nodes that displays instantly and use an IDLE_ACTION to replace them one-by-one, see [DEV not yet written] see demo/theGUI/gTree2.exw for an example of just that, which loads all hard drives. What it should not do, of course, is hammer down all the branches whose parents were collapsed, that is unless you actually wanted it to, and of course you would normally defer load as many nodes in whatever you have just loaded as you can, by putting -1s in their expanded (and false in this node's expanded).

Here is the above nested form modified to defer load all branches, with the trick that all children are actually already in place (in practice they are left in the nested/non-displayable form until the deferred load takes place, and in fact this way the root children starts off and ends up as {2,3,4,5,6,7} rather than the {2,3,9,12,13,14} as shown below for the non-deferred load). I would ideally like to be able to defer-load leaf nodes, but the ideal place to do the deferred loading is in tg_tree_get_widest(), and for good reason that does not "dive into" closed branches or leaf nodes, and in fact needed extra code to deal with empty branches, so at least for now deferred loading is branch-only. 

sequence nodes = {"Figures",-1,         -- branch with 6 children
                  {"Other",             -- leaf
                  {"triangle",-1,       -- branch with 3 children
                   {"equilateral",
                    {"isoceles",-1,{"acute","obtuse"}},
                    "scalenus"}},
                  {"parallelogram",-1,{"square","diamond"}},
                  "2D",{"3D",-1,{}},    -- 1 leaf, 1 empty branch
                  {"4D",-1,{"some","other","branch"}}}}

And here is perhaps the simplest possible deferred loader, as mentioned relying on children already being in place, expanding all branches, as noted above triggering an immediate full-depth load which kinda defeats the whole purpose / is not in general a very good idea, all false or t/f in user data would be much more sensible, but at least making the effort to not mark empty branches as expanded, since they cannot be (and stating that being why I decided against just using false here): 

function defer_load_data(sequence subbranch) 
    subbranch = deep_copy(subbranch) -- see note
    subbranch[TGTDL_EXPANDED] = subbranch[TGTDL_CHILDREN]!={}
    return subbranch
end function

Technicalia The nested form above is automatically converted into the equivalent internal flat form: 

   {{`Figures`,{2,3,9,12,13,14},0,-1,0,0,0,1,-1,8}, -- [1] (branch with 6 children, open)
    {`Other`,0,1,-1,0,0,0},                         -- [2] (leaf)
    {`triangle`,{4,5,8},1,-1,0,0,0,0,-1,3},         -- [3] (branch with 3 children)
    {`equilateral`,0,3,-1,0,0,0},                   -- [4] (leaf)
    {`isoceles`,{6,7},3,-1,0,0,0,0,-1,2},           -- [5] (branch with 2 children)
    {`acute`,0,5,-1,0,0,0},                         -- [6] (leaf)
    {`obtuse`,0,5,-1,0,0,0},                        -- [7] (leaf)
    {`scalenus`,0,3,-1,0,0,0},                      -- [8] (leaf)
    {`parallelogram`,{10,11},1,-1,0,0,0,1,-1,2},    -- [9] (branch with 2 children, open)
    {`square`,0,9,-1,0,0,0},                        -- [10] (leaf)
    {`diamond`,0,9,-1,0,0,0},                       -- [11] (leaf)
    {`2D`,0,1,-1,0,0,0},                            -- [12] (leaf)
    {`3D`,{},1,-1,0,0,0,0,-1,0},                    -- [13] (branch, empty)
    {`4D`,{15,16,17},1,-1,0,0,0,0,-1,3},            -- [14] (branch with 3 children)
    {`some`,0,14,-1,0,0,0},                         -- [15] (leaf)
    {`other`,0,14,-1,0,0,0},                        -- [16] (leaf)
    {`branch`,0,14,-1,0,0,0}}                       -- [17] (leaf)

While not actually very difficult to follow, it is of course far too tedious to be typing that sort of stuff in by hand.
There may very well be cases where building the flat form is in fact much easier than constructing a nested form.
[uses private enum TG_TREE_(TEXT | CHILDREN | PARENT | WIDTH | CHECKED | IMAGE | USERDATA | EXPANDED | WIDEST | VNC)]
In this flat form, each leaf and branch node must respectively be: 

    {desc,     0    ,parent,width/-1,checked,image,userdata} 
    {desc,{children},parent,width/-1,checked,image,userdata,expanded,widest/-1,vnc}

where children and parent are expressed as (node rather than display) indices (see next paragraph),
width/widest of -1 are replaced within the redraw process, with width being that of this node and widest being a node index (oft reset to -1) within it found by scanning all widths taking expanded/collapsed into account,
and on branches the last element, vnc, is the uber-critical visible node count. It is at least the number of direct children, and expanding/collapsing auto-propagates changes right up to the root, for both vnc and widest.
The total number of visible lines at any moment in time (after any necessary deferred loading has completed) is obviously just 1 when the root is collapsed or a leaf node, and 1+nodes[1][vnc] when expanded. That value is also the maximum possible display index, and along with widest/width sets the VIEWPORT aka scrollbars.

Internally a gTree uses two forms of index: display being 1..1[+vnc] and node being 1..length(nodes).
The internal flat structure is specifically designed to make it relatively painless to convert between them.
Most programs, however, should never have to deal with display indices themselves.

The vnc of 8 in [1] includes 6 direct children plus 10,11 since 9 is expanded, and were [9] collapsed, the 2 would be deducted from [1] making it just 6.
The total vnc is always one more than vnc[1], or 1 if [1] is collapsed.
Expanding/collapsing a branch does not change its vnc, but propagates a change to the vnc of its parent and continues that all the way up to the root node.
The vnc of 3 in [3] is the three direct children and were [5] expanded, it would become 5 and, assuming [3] were expanded, the 2 also be added to the vnc of [1].
Technically speaking vnc propagation upwards should stop when it finds a collapsed parent branch, but of course the user cannot see a collapsed branch within said in order to be able to click/expand it, so that would only apply if you are for some strange reason performing an auto-expand but stopping/only on hidden bits.

Fairly obviously, the expanded/checked/image/userdata fields are copied verbatim from the nested form and the same notes for those four fields equally apply.

Advanced Use

gGetAttribute(tree,"DATA") (nb not "DATA+", since that would just encourage you to attempt these operations on a nullified tree) returns sequence {nodes, tctrl} where nodes is the internal flat form and tctrl is a set of normally internal routines for advanced use: 

--      {
--       integer vnc = tg_tree_vnc(gdx tree)    -- visible node count [1..length(nodes)]
--       integer d = tg_tree_n2d(gdx tree, integer n) -- convert a node index to a display index 
--       integer n = tg_tree_d2n(gdx tree, integer d) -- convert a display index to a node index 
--       tg_tree_collapse_node(gdx tree, integer n)
--       tg_tree_expand_node(gdx tree, integer n)
--      }
-- aka
integer {tg_tree_vnc,
         tg_tree_n2d,
         tg_tree_d2n,
         tg_tree_collapse_node,
         tg_tree_expand_node} = tctrl

You would typically examine nodes to gather what needs to be done, nullify your copy before expand/collapse, and possibly re-fetch things to see what else might need doing, which may need to be some time later, after any deferred loading has had a chance to occur.

All scrolling and positioning is performed via the SCROLLSIZE and VIEWPORT.
The top line of the viewport is (generally) some multiple of the line_height, aka use 

    integer {sw,sh} = gGetAttribute(tree,"SCROLLSIZE"),
      {ox,oy,vw,vh} = gGetAttribute(tree,"VIEWPORT"),
//      line_height = sh/tg_tree_vnc(tree), 
        line_height = sh/(tg_tree_vnc(tree)+1), -- temp??
           top_line = floor(oy/line_height),
          view_rows = floor(vh/line_height)

as the starting point for any positional work, and if you want to move the VIEWPORT I think you know what to do. Fairly obviously expand/collapse will change the vnc, and possibly trigger deferred loading, but the latter does not actually happen until the next display, and there may very well be cases where you should start a timer to finish your work after that has had a chance to occur. It should go without saying that n +/-1 has no meaning at all, instead node index navigation must occur via parent/child links as documented above, whereas d +/-1 (obviously bound to 1..vnc) corresponds to the visible screen. All I can really say is good luck, you might also find some useful clues by inspecting the source code of tg_tree_key_handler().
