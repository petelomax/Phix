gdx (/m/c/p)

Definition: include theGUI.e

gdx identifier [= gDialog/gTree/...(...)]
Description: Type. On desktop/Phix, a non-negative integer, being 0/NULL or a positive integer index into the private control tables of theGUIe, or HtmlElement in theGUI.js, or a flat sequence of them (which may sometimes contain embedded nulls).

Use this to define interface element variables. It is probably unwise to use this to determine whether an object is in fact such, as many things which are clearly not (eg 0, 1, '2', {3,4,5}, {'6'}) may pass, the latter only when 54(==#36=='6') or more interface elements have been defined, it does however check [each element] is a non-negative integer between zero (inclusive) and the table length, at least on desktop/Phix. Some uses of a gdx may prohibit [embedded] nulls, but it is never an inherent error for a variable declared as type gdx to hold such.

Additionally, (feel free to fallback to plain integers rather than fret about memorising this lot)
A gdm is a gImage (not a gdx/odd -ve1). A gdn is the nullable/sequence version of that.
A gdc is a single gCanvas (which is a gdx) or gdm (which isn't).
A gdp is a gPixbuf (not a gdx/even -ve1 or a sequence, designed to support automatic pbr in gPixbuf_from_Pixels).
A gcp is a gdc(/gdm) or integer gdp.
A gcps is gdc or (any) gdp, used with assert(not sequence(),"gPixbuf must be theGUI-owned") to better place the error.
These six additional types are primarily intended for internal/documentation/clarity purposes, but can freely be used anywhere else they might help. Also, but private, a gcx is a non-null single gdx or gdm or gdp, with gcxs the sequence-able version of that, those two intended only for use within theGUI.e itself (and incidentally no equivalents in theGUI.js).
See the feature summary of gPixbuf, and the gCanvas drawing functions where these are most used.
pwa/p2js: Supported, or at least understood by pwa/p2js and mapped to "let /*gdx*/" aka (almost) completely ignored, since JavaScript is a typeless language. As per unwise, theGUI.js does not even have a fake version, which you should never want anyway. Applications should refrain from test/print of gdx variables, except during debugging on desktop/Phix.
[Equality tests and find() are in fact fine, but gdxvar1 < gdxvar2 and similar are potentially problematical.]
Example: 
include theGUI.e
gdx quit_btn = gButton("Quit",gQuit)
See Also: gButton, gQuit
Technicalia The pGUI distinctions between IupHandl(e|n)[s] have been dropped in favour of crashing when some element cannot be created or attempting to use a null or multiple handle[s] inappropriately. One thing in particular is that IupG/SetAttribute(NULL,...) being equivalent to IupG/SetGlobal(...) has been completely dropped, in favour of having gGet/SetAttribute(NULL,...) immediately crash, since that should help catch a few more silly little slip-ups.

It is now a more routine-specific choice as to whether an [embedded] null is ignored or invalid, for instance the parent of a new dialog can be null, whereas (eg) [as it should] gSetAttribute() crashes.

Likewise the distinction between Ihandle and Ihandles has also been ditched, but you will for instance get an error should you try to specify more than one parent (and in some cases child) when creating a new interface element.

Note that gDrawImage() and gDrawText() are actually defined with gcp canvas rather than the documented gdc, along with assert(not gdp), to provide a more helpfully placed runtime error message. Likewise gDrawLine() through to gDrawRect() actually use gcps+assert instead of gcp.

Setting variables of any gdm/gdc/gdp/gcp variables to null, non-integer, or (except gdp) sequence triggers a typecheck error (on desktop/Phix), assuming the compiler doesn't throw a wobbly first. Of course gcd() is something else entirely.

1 Incidentally, gdm/gdp variables are[/can be] negative indices to different private internal tables in theGUI, and in order to help differentiate between them, at least on desktop/Phix, theGUI makes
gdm odd, as -id*2+1 aka -1, -3, -5, .. and using [(-gdm+1)/2] internally, and
gdp even, as -id*2 aka -2, -4, -6, .. and using [-gdp/2] internally, or a 2d sequence.
Under pwa/p2js a gdm is a HtmlElement and although a plain -ve would do, integer gdp follow the same convention.
A gdp being integer or sequence determines ownership by theGUI or the application proper, but in all cases it means precisely one gPixbuf, not several.
