gSetGlobal

Definition: include theGUI.e

gSetGlobal(string name, object v)
Description: Sets an attribute in the global environment.

name: name of the attribute.
v: value of the attribute.
Notes: Like gSetAttribute(), v can be atom/string/sequence, but when retrieved it will be as decoded and internally stored.
Writeable See gGetGlobal for the read-only ones.
DIAGNOSTICS (write only, desktop/Phix only and/or quite different for JS) Enable selected diagnostic messages from theGUI, see the source code of gSetGlobal() for more details. The main idea is you don't have to run around switching things off to run or debug something else, and it is far easier to uncomment some gSetGlobal("DIAGNOSTICS",?) statements when you thought you were done but were not.
Obviously if you're not actively debugging theGUI.s/js itself, you quite simply never have to bother with this at all.
IDLE_ACTION (write only) Invoked when there are no events or messages to be processed, to perform background operations.

function idle_action([params])
params: (optional) as set via gSetGlobal("IDLE_ACTION",{rtn,args[,low_priority]}), with gSetGlobal("IDLE_ACTION",rtn) being equivalent to gSetGlobal("IDLE_ACTION",{rtn,{}[,false]}), and where rtn is a normal function identifier (aka routine_id), not a call_back, args must be a sequence suitable for passing to call_func(), and low_priority defaults to false.

Returns: If TG_IGNORE is returned the handler is removed and normal processing/idling continues.
If TG_CONTINUE is returned, as it should be every 50ms (1/20ths) or so, the main processing loop checks for any messages to process (keeping the system responsive) and automatically re-invokes this routine as soon as they have been dealt with.
No other values should be returned, specifically TG_CLOSE is not handled (instead of, I suppose, closing some random window), and TG_DEFAULT is now treated as meaningless and hence triggers a fatal error (treating it the same as TG_IGNORE, as it used to in the past, just muddifies things).

Normally, the main processing loop goes properly idle when there are no events or messages to be processed, consuming little or no CPU cycles. Setting an idle action changes that loop to a more CPU consuming one. It is important it be set NULL when not needed, that is for TG_IGNORE to be returned, otherwise the application consumes CPU even when the handler is doing nothing.

Each idle action must (eventually) terminate itself, and gSetGlobal(`IDLE_ACTION`,NULL) is specifically prohibited. Internally theGUI maintains a stack of idle routines, and promotes normal above low priority entries. [Which also happens to be the reason why it is non-readable, and besides not only does TG_IGNORE permanently discard it, you'll know anyway precisely which routine is active when it calls the active routine!] For instance, a gTree() uses a low-priority idle/redraw task to determine the widest visible node, so it can set the VIEWPORT correctly, and there may well be things which are a tad more urgent than that, not least perhaps being deferred loading of the very tree nodes it should be measuring.

Ideally the idle process should return twenty or at the very least five times per second, to keep the system responsive.
Should you create a loop or an operation that takes a long time to complete inside the idle handler of your application then the user interface message loop processing is interrupted until the handler returns, so the user cannot click on any control, and/or typed characters do not show up immediately, but there are ways to handle that:
  • split the operation into several smaller parts and return TG_CONTINUE between each.
  • split the operation into several smaller parts but use a timer to process each part.
  • move the operation to a background thread.
For something as simple as a background redraw of a gCanvas, a better idea may be to handle the "idle" state yourself. For example, register a timer for a something like 500ms, and reset it in the mouse and keyboard handlers of the gCanvas. If the timer is trigged then you are (probably) in an idle state. If the gCanvas loses focus stop the timer.

If the work that needs to be done cannot be split into suitably small time slices, you could consider migrating it to a background thread instead of using either an idle action or a timer, albeit imposing additional effort on the part of the programmer and quite probably deserving rather more rigorous testing.

pwa/p2js: Uses window.requestIdleCallback() which is marked on MDN as experimental, but that has not changed since 2015. Note that according to caniuse.com for this to work in Safari you (and/or you users) must enable requestIdleCallback in Experimental (Webkit) Features, and the same may be true for Chrome on macOS (not that desktop/Phix works on macOS, yet).
If at all possible you should strive to return within 50ms/break up idle processing into chucks of work shorter than that, to avoid (otherwise fairly harmless) JavaScript violation messages being logged to the (hidden) console. A future version may incorporate a means to get (deadline.timeRemaining() > 0 || deadline.didTimeout) in pwa/p2js, with a matching mechanism of some as yet undecided genre for desktop/Phix, should that ever be desired/requested.

For an example see demo\rosetta\DrawRotatingCube.exw [aside: the currently poor performance of gDrawPolygon() does however mean that is not really the best place to compare an idle handler with a timer, esp under WinAPI.]
