gSetAttributes

Definition: include theGUI.e

gSetAttributes(gdx id, string attributes, dword_seq args={}, integer nFrames=1)
Description: Set several attributes on one or more interface elements.

id: Identifier(s) of the interface element(s).
attributes: Two cases exist:
  • "==": args is a pair|triplet list, as in {{“name1”, attr1[, args1]}, {“name2”, attr2[, args2]},...}
    Triplets trigger a sprintf(attrn,argsn), which obviously crashes when attrn is not a string.
  • [legacy] string with the attributes in the format “name1=attr1, name2=attr2,...”
    when args!={} this case invokes a sprintf(attributes,args).
nFrames: Fine grained control for error handling, intended primarily for internal use.
pwa/p2js: Supported (except for nFrames which is simply ignored).
Comments: Most interface elements can be created with an optional implicit inline invocation of gSetAttributes(), which can make code much cleaner than invoking this function directly/subsequently, as the example below shows.

Fairly obviously the "==" case is a trivial loop invoking gSetAttribute(id,namei,attri) with individual values of any type.

The legacy string-only case requires a more involved parsing operation and that attribute values must be expressed(/ible) in textual form: a bool, for instance, can be set using "1"/"0", "true"/"false", "YES"/"NO", "ON"/"OFF" and similar, or multiple values using say "240x80" or "{240,80}", as documented for the specific attribute.

This routine itself does not care or check whether id is a single or sequence of ids, and instead leaves all such handling to the "without-s" routine.
Example: 
gButton(dlg,"ACTIVE=TRUE") -- (no image this way)
gButton(dlg,"==",{{"ACTIVE",true},{"IMAGE",xpm}})
Admittedly that is more of an example of when not to use gSetAttributes(), or more accurately implicitly inline it, but of course there would be no difficulty whatsoever extracting attr/arg into a separate following statement or two that invokes gSetAttribute[s]().
See Also: gSetAttribute, gGetAttribute, gButton, gDropDown, gList
Technicalia While I might sometimes feel like consigning the legacy string-only stuff to the dustbin, a single string can be neater and easier to write, besides there is a quite fair bit of legacy code that uses it, so it stays, and always will.

However, any potential problems, especially errors parsing embedded quotes, equals, and commas, or cannot be expressed in string form or doing so fails to transfer a delete_routine() properly, should be addressed by switching to the non-legacy "==" form or using gSetAttribute() directly, one at a time.

C code that uses nested IupSetAtt[ributes[f]]() calls should normally be converted into a single paranormalised function call when possible, otherwise be broken down into consecutive statements that all get passed the same id, and/or explicit variables created (suitably/sensibly/nicely named) for any (previously anonymous) inner elements. Sometimes that can be a bit of a drag, but more often than not the code ends up both easier to read and easier to modify at some later date.

Translate C code such as 

list = IupList();
IupSetAtt(NULL,list,"VALUE","3","1","Brazil","2","USA","3","Japan","4","France",NULL);
-- to
gdx list = gDropDown(changed,"==",{{"OPTIONS",{"Brazil","USA","Japan","France"}},{"VALUE",3}})
-- or
gdx list = gDropDown({"Brazil","USA","Japan","France"},changed,"VALUE=3")

Which creates a list dropdown with country names and defines Japan as the selected/default option.
[Those may look a little odd when placed side-by-side like that, gDropDown explains the paranormalisation that makes them both work just fine, aka there is an "optional" "OPTIONS" as the first parameter on the last one.]
