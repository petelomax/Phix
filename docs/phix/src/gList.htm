gList

Definition: include theGUI.e

gdx id = gList(object data, [rtn selected=NULL,] string attributes="", dword_seq args={})
Description: Creates a simple interface element that displays a non-sortable, header-less list of text-only items, with automatic scrollbars.

data: (non-optional) a sequence of string[ fragment]s or (recommended) a fetch function, see notes and DATA below.
selected: (optional) a procedure which is invoked when the selection changes, see the SELECTED handler below.
For more information on the attributes and args parameters see gSetAttributes().
This is a paranormalised function. (see technicalia)

Returns: the identifier of the created element, which is based on and inherits from a gCanvas.
pwa/p2js: Almost, no scrollbars as yet, for one thing. [DEV - see also hList.exw]
See Also: gCanvas, gTable, largest, apply, length
Example: 
-- demo\theGUI\gList.exw
include theGUI.e
constant data = {"one",{{"too",TG_RED,TG_ITALIC}},"three"}
gdx list = gList(data,`FONT="Consolas, 12",BGCLR=LIGHT_PARCHMENT`),
     dlg = gDialog(list,"gList",`SIZE=240x80`)
gMainLoop(dlg)
Notes: A gList is technically a read-only control, however there is nothing to prevent an application from handling keyboard/mouse input and modifying the data itself, and then presumably invoking gRedraw(). While it may sometimes be easier to store relatively small and static data directly in the gList itself, as shown above, it is usually much more flexible to use a DATA fetch function, especially should you ever want to modify the data. The following works identically to the above:

-- demo\theGUI\gList.exw
include theGUI.e
sequence data = {"one","too","three"}
integer maxlen = largest(apply(data,length),return_index:=true)

function get_data(integer idx)
    if idx=0 then return length(data) end if
    if idx=-1 then return maxlen end if
    -- Perform dynamic spell checking/syntax colouring:
    sequence res = data[idx]
    if res="too" then -- spelling mistake?
        res = {{"too",TG_RED,TG_ITALIC}}
    end if
    return res
end function

gdx list = gList(data,`FONT="Consolas, 12",BGCLR=LIGHT_PARCHMENT`),
     dlg = gDialog(list,"gList",`SIZE=240x80`)
gMainLoop(dlg)

First-class performance is a significant concern in the design of this (deliberately simple) interface element.

Text is always left aligned, with everything in the same font face/pointsize, but optionally scrollable and each line can be specified as a list of coloured and/or normal/bold/italic text fragments. Line numbers (for instance) would be the sole responsibility of the get_data() function, including any spacing, colours, and possibly even (a non-proportional) font.

Each element of data/returns from the DATA fetch function can be a plain string or a sequence of string or {string,colour[,style]} fragments. Of course you could pre-analyse (/syntax-colour) the whole of data in one go up-front, or perform that per-line/on demand, and the maxlen index might also change over time.

When gList() asks data(-1), aka "what is the longest line?", it is entirely the application's responsibility to figure out a clever way to answer that efficiently, and not the fault of gList() should an inefficient method be used, for instance apply(data,length) is fine when done once but not ideal should it repeat >99.99% (quite literally) of the exact same work on every single keystroke. For that reason, 0 is allowed to say "don't care", with the obvious catch of no horizontal scrollbar to view the rest of any over-long lines, and/or 0 initially with a gRedraw() once known.

The double braces on "too" are required since {"too",TG_RED} would display "too" (in the current foreground colour), but then complain that TG_RED is neither a string nor {string,colour[,style]}, where colour and style are as per gDrawText() (inc -1 for "as is" and {fg,bg} colour pairs). {"t",{"o",TG_RED},"o"} is also fine, as is [an autocorrect] "two".
Attributes:
CARET Display a flashing caret (only when BGSEL!=BGCLR), for use when the gList is being used as an editor. Default: false.
While properly updated when the control is clicked, the initial default {CURSORX,CURSORY} of {0,0} is off-screen.
The caret stops blinking after 5s so programs can go properly idle, but any mouse, keyboard, or focus event reactivates it.
CURSORX,
CURSORY 		Position of the caret, of limited practical value when CARET is false. Indirectly updated by SELECTED, CLICK and KEY, though CURSORX only by click, except for line-length-clamping.
MULTIPLE Allow selection of several items simultaneously. Default: 0/false/"NO".
To select multiple items, the Ctrl or Shift key must be held down.
Note this may change the required signature of any SELECTED handler.
Also note this is whole lines, inappropriate for and distinct from (say) block selection in an editor, which is much more of a syntax-colouring thing (see DATA). The latter can however co-exist quite happily with single line selection.

There is also a "MULTIPLE" setting on a gTable's SELMODE, were you looking for that.
SELECTED When MULTIPLE is false, returns the index of the currently selected line, or 0 if none.
When MULTIPLE is true, returns an integer or a sequence of length(get_data(0)) of true/false.
Can be retrieved on demand or the identically-named SELECTED handler receives immediate change notifications.
No validatation occurs when set, besides changing MULTIPLE later could effectively invalidate it anyway.
Note theEditor does its own character-based [column/block] selection, on top/instead of/as well as [single] this.

There are also similar SELECTED attributes/handlers on a gTable and gTree, were you looking for them.
SHOWLINE Ensure the specified line (1..length(data)) is visible, in other words set TOPLINE and the vertical scrollbar as needed. Does nothing if already visible, reset to 0 by the redraw operation, which is what actually applies it.

There is also a SHOWLINE attribute on a gTable, were you looking for that.
SIZE As standard (see link). Should you want to specify a specific number of lines, or ensure the control is wide enough to display a specific string, your best bet is to use (some guess at margins plus) gGetTextExtent() after it has been displayed/mapped, and use that as the (fixed/constant) line width/height in the next iteration of the program.
TOPLINE,
LINEHT 		Determine/specify the line (1..length(DATA)) at the top of the control, and/or (read only) the line height.
Should you want to know how many lines are visible, it is h/LINEHT, where h is from the VIEWPORT, and you'll have to make your own decision as to whether floor() or ceil() is more appropriate. TOPLINE may also be implicitly/indirectly set by SHOWLINE, also note that the VIEWPORT is really in charge, and should you micro-scroll such that TOPLINE is technically 1.95, you'll just get a 1 back. Lastly, LINEHT is automatically updated by FONT changes.

There are also TOPLINE and LINEHT attributes on a gTable, were you looking for them.
also ACTIVE, BGCLR, BGSEL, BGHOV, CANFOCUS, EXPAND (see note), FONT, MINSIZE, MAXSIZE, SIZE, TIP, VISIBLE.

Note that SCROLLSIZE and VIEWPORT are automatically handled on a gList, and you should not attempt to tamper with them, except perhaps for save/restore.
Handlers: Set using eg gSetHandler(list,"DATA",fetch_data), or specify fetch_data and selected at creation time
DATA A fetch function for retrieving the gList entries/lines/rows.

function fetch_data([gdx id,] integer item)
id: (optional) identifier of the element that activated the event.
item: index of the desired line, 0 to get the total length, -1 to get the index of the longest.

When item is 0, as shown above, should simply return the outer/vertical length().
When item is -1, returning 0, or the index of a line that fully fits, simply disables the horizontal scrollbar.

Obviously if the data in the list is fixed, so too is maxlen (as it is called in the example code above), otherwise:
It is absolutely critical that fetch_data(-1) is implemented efficiently and intelligently, to achieve decent performance.
Technically speaking, for perfect horizontal scrolling, fetch_data(-1) should effectively invoke gGetTextExtent() on every element of data, though when a fixed-width font is in use length() perfectly identifies the correct line, and it may in fact be good enough even when a proportional-width font is being used. Either way, repeating all that exact same work on every call of fetch_data(-1) is a sure-fire way to introduce performance issues. It is far better to maintain a cache of such [getsize] calls, or a table of line lengths, and only update the maxlen when [relevant] items actually change. It is also perfectly acceptable to initially return 0, and invoke gRedraw() once some background processing has completed.

The fetch_data function can simulate a cuebanner by returning 1/1/"Enter xxx" when the internal application data is {}, of course with appropriate colour/style on the fake fragment if desired.

The function should return the integer length|idx, a text string, or a sequence of {text|{text,colour[,style]}} fragments.
Only those lines which are actually being displayed are ever requested, plus, optionally, the longest.
See gDrawText() for full details of permitted colour and style.

There is also a DATA handler on a gGraph, and a DATA attribute on a gTable and gTree, were you looking for them.
SELECTED Action generated when an item in the list is selected.

procedure selected([gdx id,] integer item) -- when MULTIPLE=NO
-- or --
 procedure selected([gdx id,] object items) -- when MULTIPLE=YES

id: (optional) identifier of the element that activated the event.
item: Number of the selected item starting at 1, or 0 if nothing selected.
items: integer as above or full-length sequence of true/false when multiple ie ctrl or shift selections have [recently] been made.
The item[s] value is exactly the same as that returned (on demand) by the SELECTED attribute.

Naturally, specify a gdx id should you have some use for it, and just omit it otherwise.
Also note it is backend/browser-specific as to whether the display is updated before or after this handler is invoked.

The MULTIPLE attribute determines whether the item[s] parameter may be a sequence.
Note that theGUI only cares about 1 vs 2 parameters and performs no further typechecking itself
on any parameters, it does however always deliver 1 or 0 selected items in integer form (as does the attribute).
Should you for any reason want to modify or keep a copy of a sequence you obtain, you should deep_copy() it.

There are also SELECTED handlers on a gTable and gTree, were you looking for them.
also Note a gList uses gCanvas()/REDRAW and MOUSEMOVE internally, and you should not attempt to use/override them [shout should you need em]. It also provides default handlers for CLICK and KEY: that code is still invoked after any user-supplied routine, unless it returns TG_IGNORE, hence doing so may inhibit the natural behaviour.
Technicalia As this is a paranormalised function, and with no expectation that you should attempt to memorise this lot, except maybe "dsaa", you can assume that all of the following are potentially perfectly valid, with the fairly obvious restrictions that any parameters must be in strict dsaa-order, and no args without attributes: 

                        gList(data)
                        gList(data, selected)
                        gList(data, attributes)
                        gList(data, attributes, args)
                        gList(data, selected, attributes, args)  -- (the full set)

Note that in practice selected is actually defined as object, and attributes as sequence rather than string, but are thoroughly verified to be of the documented types, after being repositioned via theGUI.e/paranormalise_raa(), which isn&apos passed the non-optional data parameter and hence that is never moved.

A gList is used by theEditor to implement the syntax-coloured editable text area, though it is the application code which interprets all keystrokes (bar basic navigation) and modifies the text, not the gList() itself.
In that particular case the data will contain the plain text lines, with get_data() splitting requested lines into the appropriate syntax-colouring fragments on demand, in a similar fashion to what Edita has always done, with bracket/comment carryFwds[]s, and a linelengths[], since recalculating 50,000 lengths gets a bit costly when you do it on every single keystroke, etc.
Since it may be opening hundreds of files and some of those could be rather long, it initially returns 0 for the longest line, which (temporarily) disables the horizontal scrollbar, runs a background process, then redraws with a proper scrollbar.
Likewise it might display before computing the bracket/comment carry forwards, and later correct, which is only going to matter when it is re-opening a file from the previous session, with the viewport not at the very top.
During the implementation of all that, I will of course move as much reusable code into theGUI.e as I can, for potential use on a gTable(), which I already know will be needed for demo\rosetta\7guis\Cells.exw and possibly several other things.

Another example: it might be wise to keep the last 1,000 log entries in a circular list, rather than keep all and gradually get slower and slower before inevitably running out of memory and crashing and finally creating a ridiculously humongous ex.err.
It is the responsibility of the gList() to deal with all that quickly and efficiently, whatever you decide to do, and of course when using a circular list there should be no difficulty mapping the 1..1000 as asked for to (say) 675..1000,1..674.

Should this control not meet your needs then perhaps you might consider a gTable() instead, which can handle multiple sortable columns, with headers, numeric entries, icons[??], etc. Also, a gList is just one half of the original IupList functionality which has been completely redesigned and split, so you may actually be looking for a gDropDown() instead. Should you be looking for a "dropped down with edit area" then what you probably want is actually a gVbox() with a pair of {gText,gList}) children instead, and performing the whole "add entry" thing explicitly yourself, along with setting the gText whenever selection changes in the gList. (Not really a fan of that kind of silly nonsense, be I, but at least whenever theGUI starts to show signs of getting a bit highly opinionated, it should be relatively straightforward to change.)

The file demo\theGUI\hList.exw contains a slightly flawed hll implementation that you might find useful for study/bugfixing.
