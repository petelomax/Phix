gSpin

Definition: include theGUI.e

gdx id = gSpin([rtn value_changed=NULL,] string attributes="", dword_seq args={})
Description: DEV badly broken, needs to be rewritten as a canvas-based control, see demo/theGUI/hSpin/hText.exw for progress...

Creates a spin control.

value_changed: (optional) a procedure invoked when the contents change, see VALUE_CHANGED.
For more information on the attributes and args parameters see gSetAttributes().
This is a paranormalised function. (see technicalia)

Returns: the identifier of the created element.
pwa/p2js: Supported.
Example: 
-- demo\theGUI\gSpin.exw
include theGUI.e

procedure value_changed(integer v)
    ?{"value_changed",v}
end procedure

gdx spin = gSpin(value_changed,"MARGIN=13x13,VALUE=5"),
    dlg = gDialog(gHbox({spin}),`gSpin`,`SIZE=240x80`)
gMainLoop(dlg)
Attributes:
SPIN (non inheritable) Accepts/returns {lo,hi[,step[,wrap]]} where lo and hi must be integers with lo<=hi, step must be an integer>=1, and wrap is a boolean. Default: {0,100,1,false}. Not that dynamic toggling between the two is actively supported, nor is there any attempt to prevent such foolish behaviour, in practice a gSpin() is in fact just a gText() with SPIN!=false, aka a few relatively minor tweaks, obviously affecting the expected drawing, input field length/limit/right hand margin, and mouse and key handling.
also ACTIVE, CANFOCUS, FONT, EXPAND, MARGIN, MINSIZE, MAXSIZE, SIZE, TIP, VALUE, VISIBLE.
Handlers: Set using eg gSetHandler(spin,"VALUE_CHANGED",value_changed), or simply specify value_changed at creation time
VALUE_CHANGED Called after the value was interactively changed by the user.

procedure value_changed([gdx id, ]integer v)
id: (optional) identifier of the element that activated the event.
v: same as gGetAttribute(id,`VALUE`), which can o/c be used on-demand instead of a handler. Note that v will simply be 0 should say "1j3" or worse have been pasted into the control, and 13 when the 'j' is deleted. The one thing it does do is automatically trim() whitespace.
also CLICK, KEY: All common handlers are supported.
Technicalia As this is a paranormalised function, and with no expectation that you should memorise this lot, except maybe "raa", you can assume that all of the following are potentially perfectly valid, with the fairly obvious restrictions that any parameters must be in strict raa-order, and no args without attributes: 

                        gSpin()
                        gSpin(rid)
                        gSpin(attributes)
                        gSpin(attributes, args)
                        gSpin(rid, attributes)
                        gSpin(rid, attributes, args)    -- (the full set)

Note that in practice rid is actually defined as object, and attributes as a sequence rather than a string, but verified to be of the documented types, after being repositioned via theGUI/paranormalise_raa().

See demo\theGUI\hText.exw for progress on a canvas-based drop-in replacement.
The following are images of the native controls, Windows(IUP), GTK2, and GTK3 (for reference only):


No attempt has been made to replicate the GTK3 style. Clicking the up/down arrows applies +/- step to VALUE, as do the up/down arrow keys.
Reduces the editable area to prevent it from ever overlapping those buttons.
Prohibits (visible/non-editing) keys other than '0'..'9', and '-' if lo<0 and at start, or any that make VALUE > hi.
Note that blank is a legal out-of-range 0, as are "-"("") and values <lo within blank ⇔ valid [ie num|del, iyswim].
It is also fair to say that, to permit reasonable common-sense input/interaction, lo is a softer limit than hi.
There is however no prohibition on paste, on the understanding the user is expected to edit away any illegals.
Inherits any platform-specific gText styling but does none of it's own, except (perhaps) to marry in with that.
It should be perfectly feasible or even trivial to cope with fractional steps, left as an exercise for the reader.
Likewise, despite any implications above, I simply have not bothered to implement or test negative lo (or hi).
