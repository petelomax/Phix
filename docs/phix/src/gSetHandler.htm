gSetHandler

Definition: include theGUI.e

gSetHandler(gdx id, sequence name, rtn handler=-1)
Description: Associates one or more handlers for an event or several on one or more controls.

id: identifier of the interface element(s), can be a single integer or a (flat, null-free) sequence of them.
name: a string identifying the handler, eg "KEY", or an even-length sequence of string,rtn pairs.
handler: a (bare) function or procedure, or NULL. Must be -1 (/omitted) if name is pairs and not a string.

A NULL rtn when name is a string, or any NULL even element of name, simply removes any association.
 [-1 is not a valid rtn, so the implementation actually uses integer but verifies handler is a rtn/NULL and not -1 if/when actually used.]

Refer to each specific interface element for valid/meaningful names and their required routine signatures.
pwa/p2js: Supported.
Notes: All elements with a parameter of type rtn permit/encourage an implit invocation of gSetHandler().

You can also invoke this as the simple alias gSetHandlers(), with the implication but not enforcement that id or name or both are non-unitary, or just stick with the non-s name for everything. Note however that is in stark contrast to (say) gSetAttribute() and gSetAttributes(), which are in fact quite different, although they too can both set something on one or more elements, likewise unconnected to a trailing s, though only the latter can set somethings in a single call.

Handlers are used by the application to receive and process notifications from the system that the user or the system itself has interacted with the user interface of the application. In particular that means everything (directly and indirectly) from the mouse and keyboard, as well as things like timers and other alerts, such as "going idle" (ie/aka "msg queue is empty").

All handlers (optionally) receive at least the element for which the event was triggered.

Most handlers (except those defined as procedures) must return one of the following values:
  • TG_CONTINUE (-4): Pass the treatment of the event to the parent element.
  • TG_DEFAULT (-3): Proceed normally with user interaction (but see KEY, esp wrt menus).
  • TG_IGNORE (-2): Perform no further processing of this event.
  • TG_CLOSE (-1): Close the parent window and in some cases exit the application.
Only some handlers support the last 3 return values, check each handler documentation.
TG_CONTINUE and TG_DEFAULT only really differ for a KEY handler: TG_CONTINUE is "if anyone asks, I was never here" whereas TG_DEFAULT is more "propagate only unhandled Alt-keys", (it isn't always immediately obvious which of those is needed in any given circumstance, and in fact the precise behaviour of TG_DEFAULT is subject to change, since it is really just a shorthand to help avoid needing the exact same code in every single KEY handler).

NB: returning TG_CLOSE does not automatically invoke CLOSE handlers. [DEV maybe it should...]

Note: Avoid using TG_IGNORE (and TG_CLOSE) unless it is specifically documented for that handler.
For example, it is and always was the documented behaviour for an IDLE_ACTION handler to be removed when it returns TG_IGNORE. [However, the act of IUP_IGNORE removing the callback on an IupButton in pGUI (which was in fact undocumented until I myself piped up) has not been replicated in theGUI, you may be relieved to hear.]

An important detail when using handlers is that they are only triggered when the user physically performs an action on an element. In particular, a handler is not invoked when the programmer sets a value via gSetAttribute().
For instance: should the code change the selected item on a list, no handler is called and it is up to the programmer to ensure any additional appropriate processing occurs, by some more direct means.

rtn handler = gGetHandler(gdx id, string name)
can be used to retrieve a previously set handler, primarily intended for use inside theGUI itself, but you might, for instance, want to emulate mouse or keyboard input (non-native controls only) by invoking the CLICK or KEY handlers respectively. An invocation would then simply be handler(x,y,z), as a procedure or function appropriately, and which incidentally is a far more elegant syntax than any fetch-and-call routine could ever be, especially perhaps for those (such as CLICK) which can be either a function or a procedure. [In other words replacing a pair of statements such as rtn handler = gGetHandler(id,name); [res =] handler(...args); with some single mythical [res =] gInvoke(id,name,args); statement would simply be too tricky, syntax-wise.] Technically all the handlers below except VALUE_CHANGED are pseudo-inherited, in that theGUI scrambles up the layout hierarchy tree looking for some other control to send a message to, using a currently private gGetInheritedHandler() function, but that is not part of gGetHandler(). Returns NULL when not previously assigned.
Examples: 
-- set one handler on one interface element:
gSetHandler(canvas,"REDRAW",redraw)
-- set the same handler on two elements:
gSetHandler({canvas1,canvas2},"REDRAW",redraw)
-- set two handlers on both elements:
gSetHandler({canvas1,canvas2},{"REDRAW",redraw,"CLICK",click})
Common: The following handlers are common to several controls/interface elements:
CLICK Event generated when any mouse button is pressed or released.

function/procedure click(gdx id, sequence status[, integer x, y])

id: identifies the interface element that activated the event.
status: a sequence of two characters and three bools that qualify the precise event that just occurred:
 The first character is 'L', 'M', 'R' for Left, Middle, or Right button, or 'X', 'Y' for the mouse-under-thumbs.
 The second character is 'S', 'D', or 'R' for Single-click, Double-click (not XY), or Release events, see note.
 The third element is true if the ctrl key is depressed, false otherwise.
 The fourth element is true if the shift key is depressed, false otherwise.
 The fifth element is true if the alt key is depressed, false otherwise.
x,y: (optional) pixel co-ordinates of the mouse cursor when the click/release occurred.

The ACTION event is usually much more suitable for a gButton.
The VALUE_CHANGED event is usually much more suitable for a gCheckbox.
The HANDLER event is always much more suitable for a gMenu.
The TABCLICK handler must be used instead on a gTabs control.
Scrollbars were initially implemented using CLICK, riding roughshod over attempts to use it for anything else (anywhere else on the entire canvas), however that has since been replaced with an explicit local call (with clicks on the scrollbar itself not forwarded to any other handler). Similar tweaks may yet be required for similar gaffes.

In many/most other cases the system would naturally ignore a click and perform no further action anyway, hence
if the handler is declared as a procedure it is treated the same as a function which always returns TG_IGNORE.
Returning TG_IGNORE prevents the event from being propagated, all other values do not, though it is probably best to use TG_CONTINUE to implicitly imply that propagation.

Note that for a double click (and for clarity showing only the second character of 5/4 consecutive status codes),
GTK sends "S", "R", "S", "D", "R", whereas slightly more sensibly
WinAPI just sends: "S", "R", "D", "R" (ie no second "S").
[DEV I have yet to test what JavaScript does... Sending a fake second "S" under WinAPI seems silly, suppressing the second "S" under GTK would necessitate fetching the system double-click speed setting, and I simply don't fancy doing that either.]

Affects: All, except gTabs and menus.
KEY Event generated when a keyboard event occurs.

function/procedure key_handler(gdx id, integer c, [[bool] ctrl[, shift[, alt [, up]]]])

id: the interface element that had focus when the key event occurred, or maybe its parent.
c: the typed key. Some possible (special) values are listed below.
ctrl, shift, alt: (optional) any key modifiers that were also pressed.
up: (optional) required should you wish to receive key release events.

A key handler can also invoke gGetOpenFileName().

Note that, unlike pGUI, a key such as <Ctrl C> is passed to key_handler as 'c' or 'C' with ctrl=true, depending on the state of caps lock. While the latter is honoured, there is no forwarding of its state with every keypress, and absolutely no guarantee of any correlation at all between shift and 'C' or 'c'. In general you get the key the system is going to act on, as opposed to something you might find a little easier to digest, not that an upper()/lower() will kill you.

The following key constants are defined, in particular for those that cannot intuitively be written as a normal character constant, like say 'A' can, plus, as you can see, a few/five that could be:

VK_BS (or '\b') #08 = 8 VK_CAPSLOCK #1C = 28 VK_F1 #B1 = 177
VK_TAB (or '\t') #09 = 9 VK_NUMLOCK #1D = 29 VK_F2 #B2 = 178
VK_LF (or '\n') #0A = 10 VK_PAUSE #1E = 30 VK_F3 #B3 = 179
VK_CR (or '\r') #0D = 13 VK_SCROLL #1F = 31 VK_F4 #B4 = 180
VK_SP (or ' ') #20 = 32 VK_DEL #7F = 127 VK_F5 #B5 = 181
VK_LCTRL #13 = 19 VK_POUND #A3 (£) VK_F6 #B6 = 182
VK_RCTRL #14 = 20 VK_APPS #A6 = 166 VK_F7 #B7 = 183
VK_LSHIFT #15 = 21 VK_INS #A7 = 167 VK_F8 #B8 = 184
VK_RSHIFT #16 = 22 VK_HOME #A8 = 178 VK_F9 #B9 = 185
VK_LALT #17 = 23 VK_PGUP #A9 = 179 VK_F10 #BA = 186
VK_RALT #18 = 24 VK_PGDN #AA = 170 VK_F11 #BB = 187
VK_LWIN #19 = 25 VK_END #AB = 171 VK_F12 #BC = 188
VK_RWIN #1A = 26 VK_UP #AC = 172 VK_LEFT #AE = 174
VK_ESC #1B = 27 VK_DOWN #AD = 173 VK_RIGHT #AF = 175

NB: actual values are made up to get WinAPI/GTK/JS talking the same language, may change between releases, are only detailed for debugging a key handler, and are even easier to change in theGUI.e/js than these docs, but of course testing everything still works is a tad tricker. Avoid using assumptions and clever coding tricks like the plague, specifically any range or bit-testing.

string res = gGetKeyName(integer c) converts key codes into something more human-readable, but you are expected to cope with/prefix ctrl/shift/alt/up yourself.

In a similar fashion to paranormalised functions, only without any reshuffling of the parameters, you can assume that all of the following are potentially perfectly valid, with the fairly obvious point that the third parameter, if present, always contains the ctrl flag, and likewise the fouth contains shift, the fifth contains alt, and the sixth up, in spite of whatever the parameters may actually be named, and even if you only wanted to check for key releases, without caring about some/all of ctrl/shift/alt, you would still be required to declare the full six parameters, and perhaps having previously relied on the absence of an alt parameter so you never got any alt-keys, say, sudddenly needing to guard against them should you bolt alt, up on: 

                        key_handler(gdx id, integer c)
                        key_handler(gdx id, integer c, ctrl)
                        key_handler(gdx id, integer c, ctrl, shift)
                        key_handler(gdx id, integer c, ctrl, shift, alt)
                        key_handler(gdx id, integer c, ctrl, shift, alt, up)
                        key_handler(gdx id, integer c, bool ctrl)
                        key_handler(gdx id, integer c, bool ctrl, shift)
                        key_handler(gdx id, integer c, bool ctrl, shift, alt)
                        key_handler(gdx id, integer c, bool ctrl, shift, alt, up)

When ctrl/shift/alt are absent, the backend avoids relaying characters and other keystrokes that it knows the handler cannot cope with or might mis-interpret. In the case (sic) of shift, you'll still get '!' (#21) all the way to '~' (#7E) but not space or below, or VK_DEL and above (including all the function keys), that is when shift is down but there's no shift parameter on your key_handler routine (or whatever it is actually called), and you would of course still get all un-shifted keys. Should you want to treat say <F1>, <Ctrl F1>, <Shift F1>, <Alt F1>, and the other four possible combinations of multiple modifiers the same, you would have to declare all five parameters but then ignore the last three. Note that get_routine_info() returns exactly the same thing for traditional named params as un-named, hence theGUI.e treats such identically. Caps lock and Num lock do not of course alter which physical keys are relayed, just some of their values.

Returns:
If TG_CONTINUE is returned the key is propagated to anything that might handle it.
If TG_DEFAULT is returned or it is a procedure VK_ESC (close window as per bEsc) and VK_F5 (refresh browser when under pwa/p2js) are propagated, but other keys are not. NB if your handler accepts the alt key, aka has five or more params, then TG_DEFAULT disables the alt key on menus and everything else - this "feature" allows you to stop unhandled alt keys from belling (on WinAPI at least, and in fact handled ones too), but you have to be selective about which ones you let through. As stated above, when the alt param is absent your handler is not invoked when that key is down anyway, since it cannot possibly cope correctly and would be guaranteed to mis-interpret things.
[Warning: subject to change, something tells me TG_IGNORE should/can already be used to achieve that menu thing...]
If TG_IGNORE is returned the key is ignored and not processed by the control and not propagated.
Warning: your defence may suffer if you (accidentally) disable any keys that you later rely on in court.
There are cases where TG_IGNORE may start disabling generic OS/backend-specific behaviour, and cases where it simply doesn't get that chance, such as <Alt F4>, ymmv. There is of course a whole slew of keyboard-handling code scattered throughout the various backends (windows/linux/gtk/browsers) that neither you nor I have any control over whatsoever. That said, 95% or more of the things you might want should all work just fine.
TG_CLOSE will be processed.

Notes
Keyboard handlers depend on the keyboard usage of the control with the focus.
So if you return TG_IGNORE the control will usually not process the key.
But be aware that sometimes the control processes the key in another event so even when returning TG_IGNORE the key can still get processed, although it will(/should) not be propagated, again ymmv.
It may also be that a keystroke went to the wrong control (eg the containing dialog) which graciously invoked some child handler in case it could deal with it, but no amount of returning TG_CONTINUE will make the key have been originally entered in the right control, that is should you be hoping/expecting the standard system behaviour to kick in. Should things only work as desired after manually clicking on a control, then maybe you should think about somehow forcing an appropriate gSetFocus().

If there is no handler attached to a control the event is automatically propagated to the parent container.

Affects: All elements with keyboard interaction.

Legacy, porting, and other concerns

While pGUI defined several hundred (416 to be exact) key constants, theGUI limits itself to a much more reasonable subset (42 to be exact): typically as and when the compiler complains, replace eg VK_a with 'a', and any specially prefixed keys such as VK_cC with (say) "ctrl and upper(c)='C'". Note that IUP allows a custom callback to be attached specifically to (eg) VK_cC, whereas theGUI has just the one generic KEY handler.

Instead of (re-)using say K_DEL from pGUI, the equivalent constant is VK_DEL (as per arwen), partly because I did not want to break pGUI help lookup too soon (which is neither a concern nor a reality with what little arwen docs there are in this file), but more importantly I certainly did not want to end up with K_C/K_D/K_E linking to pGUI but K_DEL linking to (this) theGUI help section, and probably getting left in that mess pretty much forever.

Note that (eg) VK_F1 is (the freshly made up, unsigned) #B1 (177) rather than the potentially confusing WinAPI value of 'p' (ie #70 or 112, or for that matter the GTK value of #FFBE which is 65470 or -66), and likewise most other keys. The function key range #A0..#BF was chosen because that range is unaltered by upper/lower(), unlike say #F1 ⇔ #D1. There is a relatively straightforward mapping table in theGUI.e which should be fairly easy to tweak if ever needed, and some (hopefully) matching definitions/manual mapping in theGUI.js. (obvs. just search for VK_F1)

An error occurs if c is defined as an atom: the original pGUI went to great lengths to enforce that and prevent a "time-bomb" routine signature error as I then dubbed it. Should such occur you would be advised to review and update the (legacy) code as necessary, at the same time as changing the c parameter to be an integer.

Small differences in handling will almost certainly exist and quite likely remain forever, for instance GTK on Windows does not generate VK_CAPSLOCK, and GTK2 does not generate the VK_PAUSE key.
On GTK, pressing and releasing the right alt key (aka Alt Gr), when up is present, gets you VK_LCTRL+ctrl, VK_RALT, VK_LCTRL+alt+up, and VK_RALT+up, whereas on WinAPI you get VK_LCTRL+ctrl, VK_RALT+ctrl+alt, and VK_RALT+up, one less message for reasons beyond me, and no doubt partly because GTK provides an event.state whereas WinAPI calls GetKeyState() - it is probably best to ignore ctrl/shift/alt when dealing with those actual keys. GTK also appears unable or unwilling to permit the capture of Alt-Space or Alt-F4. On the other hand, Windows does some things with the Alt key (alone) and F10 that sort of put it into a "special mode" until hit a second time, all of which I don't completely grok. And of course a browser is going to do some browser-like things with some keystrokes and that may differ between browsers. Oh, and of course there is a "Windows" key one away from the spacebar, but that's for the OS, not your app, OK? (ie/aka ymmv) However otherwise, more normal things are generally speaking pretty consistent, and avoiding the odd dicey one shouldn't really be a problem when there are perhaps 252 other key combinations that you could certainly always use, as in 4 combinations of ctrl/shift times ~63 suitable keys, at least, and nearly but not quite as many again that involve the alt key.

Note that VK_POUND is the pseudo-UTF-32 code point #00A3, aka £, and may well need special treatment, perhaps by being substituted in UTF-8 with x"C2A3", ie two bytes.

Also there are no VK_CTRL, VK_SHIFT, or VK_ALT keys without an L/R, and some legacy code may need "L or R" changes.

Finally, I own/use a UK keyboard, and haven't done any testing of non-latin-alphabet input, but I strongly doubt changes that might yet be needed in that regard would be spectacularly difficult, at least assuming there is no need for tens of thousands of new vk_constants, and I will of course be happy to assist as best I can (once provided with suitable details/links/photos/whatever). It may be that several VK_XXX constants, especially #90..#FC, need/deserve (say) +#100 or +#FF00 to permit proper unicode input.
MOUSEMOVE Event generated when the mouse moves.

procedure mousemove([gdx id, [integer x, y[, [bool] left,middle,right[, ctrl,shift,alt]]]])

id: (optional) identifier of the interface element that activated the event.
x, y: (optional) position in the canvas where the event has occurred, in pixels.
left,middle,right: (optional) status of the mouse buttons, true for down and false for up.
ctrl,shift,alt: (optional) status of the modifier keys, true for down and false for up.

In a similar fashion to paranormalised functions, only without any reshuffling of the parameters, you can assume that all of the following are potentially perfectly valid (ie "0, 1, or in sets of 3"): 

                mousemove()
                mousemove(gdx id)
                mousemove(gdx id, integer x, y)
                mousemove(gdx id, integer x, y, bool left, middle, right)
                mousemove(gdx id, integer x, y, bool left, middle, right, ctrl, shift, alt)

An error occurs if the handler does not have 0, 1, 3, 6, or 9 parameters, or the last 2..8 are not integer/bool.
Since bool is a subset of integer, and in fact get_routine_info() returns 'I' for parameters of both, it doesn't actually make any difference should the "bool" type specifier be omitted.

On Windows, as similarly noted in KEY above, if you dab on the Alt key, then try moving the mouse, you may find it is effectively disabled until you tap the Alt key again. Holding the Alt key while moving however works fine.

Affects: gCanvas, and by implication all the canvas-based controls: gGraph, gList, gTable
MOUSELEAVE Event generated when the mouse leaves - automatically taken care of on several/most controls, and in truth this is more for documenting how that is done, not that I would try to stop you from using it.

procedure mouseleave([gdx canvas])

canvas: (optional) identifier of the canvas (or canvas-based control) being left.

Controls often need to be redrawn when the mouse leaves, for instance to remove hover effects on scrollbars or the hover line on a gList. This handler is invoked before the redraw handler, and in all honesty was implemented for the benefit of the demo\theGUI\h<Fake>.exw programs, at least more than anything else. [Note this handler post-dates demo\theGUI\hList.exw, and the internal scrollbar handling, so they do not actually use it, but certainly would had things been written/implemented the other way round.]

There is no corresponding MOUSENTER since that is expected to be perfectly/better catered for by MOUSEMOVE, albeit triggered much more frequently than once/alternately.

Affects: gCanvas, etc.
MOUSEWHEEL Event generated when the mouse wheel is used - automatically taken care of on several/most controls, and in truth this is more for documenting how that is done, not that I would try to stop you from using it.

procedure mouse_wheel(gdx id, integer direction[, [bool] ctrl[, shift[, alt]]])
ih: identifier of the element that activated the event.
direction: -1 when rolled toward the user/down, +1 when rolled away/up.
ctrl, shift, alt: (optional) true if that key is being held down, false otherwise.
If any of the last three are omitted, the handler simply does not receive any messages when those keys are down.
[Technically, they're bools, but since bool is just an alias of integer it doesn' actually make any difference to the routine signature, and on that basis "bool" is therefore optional, in that they would inherit the integer type from the direction parameter.]

The default builtin handler for a canvas, which automatically comes into effect when a SCROLLSIZE larger than SIZE is specified, uses ctrl to scroll horizontally and/or shift to scroll faster.
Be warned that as per KEY, WinAPI does it's usual strange things with the Alt key (alone).
The direction is kinda backwards, but it's the same sign as WinAPI, and simply replicated on the other backends.

Affects: gCanvas, and therefore (or will be) all controls except menus.
VALUE_CHANGED Event generated when the value of a control is changed.

function value_changed([gdx id, ]<object> v)
ih: (optional) identifier of the element that activated the event.
v: (non-optional) 




and 
on a gCheckbox() v is (optional/) a bool bChecked,
on a gDatePick() v is a timedate dt,
on a gDropDown() v is a string/integer,
on a gSlider() v is an integer,
on a gSpin() v is an integer,
on a gText() v is a string.
Note that should id be omitted, v must be the correct type, otherwise you'll either get a signature error trying to set the handler, or the id in a sole object parameter. Obviously when the handler provided only ever deals with one control you are unlikely to need the id, but when it deals with more than one, you will need it or perhaps better yet gGetAttribute(id,`USER_DATA`) to understand what is actually happending.
Technicalia Since theGUI uses normal first class routines as handlers instead of callbacks, IupSetCallback() has been renamed/needs to be replaced with gSetHandler(), and importantly any quotes around the routine name must be removed, though theGUI_from_pGUI() does all that automatically for you if you let it.
There are no equivalents in theGUI for Icallback(), cbfunc(), iup_name_from_cb(), or iup_cb_from_name(), and likewise IupGetCallback() should be/is auto-replaced with gGetHandler(), but of course if the old code was actually invoking the callback that would need altering as well.
