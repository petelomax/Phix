time travel debugging

There is a pretty inspirational video here.

The likes of rr, undo, WinTTD sound like alot of blood, sweat, and tears, and I'm looking for/thinking of something much simpler.
The notion of "hot reload" is good, but being part of the "timelime" as per the tomorrow corp video is just plain daft.
What you want is a seamless "get back to where you were", and I think we can accept a 95% solution rather than demand perfection.
An obvious prerequsite is to restore a "session view", which the existing debugger lacks entirely.

Firstly, there are no plans to defeat non-determinism. I will probably give get_rand()/set_rand() a go, but make no attempt to control any other random seed generators, or capture/restore time(), or replicate system events at very specific times, etc. I might offer to run a session-specific batch file to restore files as part of a restart, but have no plans to sandbox file i/o, and especially not for anything like SQLite. Record/playback of user input (keyboard and mouse) strikes me as a generally useful thing to have, and as such should be implemented entirely independently of (and be usable by) and certainly have no dependency on debug whatsoever.
Secondly there are no plans to synchronise debug and GUI appearance.

Staring us in the face is a very simplistic addition of a trivial (circular) buffer (or ten) in the debugger of breakpoints and those variables it has already been asked to show, with "step back" leaving the running program paused where it is but showing details from said mini-history. Should you want to look at anything else, or step back further, then a "hot reload"/"full restart" is required. When I said 95%, what I was thinking of is: it will get you to the same point in the code where the variables you are currently looking at match, rather than precisely where you were. A "step back" probably needs two forms: the previous line of code, and the last time any of the currently viewed variables changed. Precise handling of "no match happened" has yet to be nailed down, in my head, that is.

The last thing I want to do is embed a full GUI in the heart of the interpreter. Instead it needs to be a standalone program (or part of theEditor) that communicates with the runtime via ipc or somesuch. A simple "fake driver" could be written to get that going.
