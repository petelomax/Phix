gSetAttribute

Definition: include theGUI.e

gSetAttribute(gdx id, sequence name, object v, sequence args={}, integer nFrames=1)
Description: Set a single attribute on one or more interface elements, or several attributes to the same value.

id: Identifier(s) of the interface element, may not be NULL, or contain any embedded nulls.
name: name of the attribute[s], usually a literal string such as "SIZE", several more examples below.
v: value of the attribute. If NULL the default setting will be used (at least in most cases).
args: (optional) if not {} then v = sprintf(v,args) is performed.
nFrames: Fine grained control for error handling, intended for internal use.
pwa/p2js: Supported (except for nFrames which is simply ignored).
Comments: Note that, unlike say gSetHandler[s](), the "with-s" version of this routine, ie gSetAttributes(), has a different purpose/api, being multiple attributes with a different value for each in one call, the common implementation of inlined attributes as part of most interface element creation routines, and in fact invokes this routine multiple times, whereas this "without-s" version sets one or more attributes to the same value, with both being able to apply said to one or more elements. It is also fair to say this routine's second parameter should not contain any '=', whereas that of gSetAttributes() must.

When args is not {}, the implicit sprintf obviously crashes when v is not string, contains bad format specifiers, etc.
Otherwise v can be anything, including integer, atom, string, [nested] sequence, or an equivalent string, eg: 

        integer w = 100, h = 180
        gSetAttribute(id,"SIZE",{w,h})
        gSetAttribute(id,"SIZE","%dx%d",{w,h})
        gSetAttributes(id,"SIZE=%dx%d",{w,h})
        gSetAttribute(id,"SIZE",{100,180})
        gSetAttribute(id,"SIZE","100x180")
        gSetAttribute(id,"SIZE","{100,180}")
        gSetAttribute(id,{"SIZE"},{100,180})
        gSetAttribute({id},"SIZE",{100,180})
        gSetAttribute({id},{"SIZE"},{100,180})
        gSetAttributes(id,"==",{"SIZE",{100,180}})
        gSetAttributes(id,"SIZE={100,180}")
        gSetAttributes(id,"SIZE=100x180")

(obviously that's twelve slightly different ways to achieve exactly the same thing,
 the first being the fastest, inlined like the last generally the neatest overall code,
 and the aim is to nit-pick over the petty details as little as possible, not confuse,
 or in other words let you code[/store] things in the way that feels most natural)
Aliases: The following aliases can be used for a clearer statement of intent and to make the code a little more self-documenting:

gSetInt(gdx id, string name, integer i)
gSetDouble(gdx id, string name, atom a)

Apart from the tighter validation of the third parameter (on desktop/Phix only), and perhaps more crucially a clearer declaration of intent which can help make the code a bit more self-documenting, they behave identically to gSetAttribute(). Since JavaScript is a typeless language, under pwa/p2js, there isn't actually any additional validation (not that I'd struggle to add some), and for now they really are just pure aliases. Similar aliases exist for gGetAttribute. [While there is a gGetIntInt(), there is no gSetIntInt(), as yet.]
Example 1: 
gSetAttribute(dlg, "VISIBLE", true) -- (and/or "YES"/"NO")
gSetAttribute(text, "VALUE", "Hello!")
Example 2: 
-- Enable or disable several buttons and/or menu entries simultaneously
gSetAttribute({b_cut,b_copy,m_cut,m_copy}, "ACTIVE", selected_text!={})
Example 3: 
-- Set several attributes (on one or more controls) to the same value
gSetAttribute(list, {"ACTIVE","CANFOCUS","CARET","MULTIPLE"}, false)
See Also: gSetHandler, gGetAttribute, gSetAttributes, gSetGlobal, gGetGlobal, GAP, SPACE, or the full list here.
Common: The following attributes are common to several controls/interface element types.
See also gGetAttribute for some other common read-only attributes.
ACTIVE (inherited) Activates or inhibits user interaction. Value true/"YES" (active/default), false/"NO" (inactive).
An interface element is only active if all parents up to and including the dialog are also active.
Can also be set for non-interactive controls and may affect their visual feedback.
Affects: All controls that have visual representation, except gGraph.
BGCLR (container-inherited) The interface element's background colour, eg TG_WHITE.
Not inherited by gButton, gDatePick, gDropDown, gList, gMenu, gProgressBar, gSpin, or gText, but the containers gHbox, gVBox, gFrame, and technically speaking gSplit, along with gCanvas, gCheckbox, gGraph, gLabel, gRadio, gSlider, gTabs, and gTree scan up the hierarchy all the way to the containing gDialog if necessary, which gets a TG_WHITE default. Since a gSplit has no margin or space itself, the only potential use is inheritance purposes.
On a gCanvas this is used to clear before each REDRAW, and hence should be set outside that handler. Technically can also be set on a gImage and a gPixbuf, but since you must specify something appropriate at creation time and they are not regularly blanked, it is not actually used anywhere, unless you fetch/use it yourself.

See also: BGSEL, BGHOV, FGCLR
BGSEL,
BGHOV 		Similar to BGCLR (set same as that to disable) but for selected and hover effects.
Defaults #CCE8FF, #E5F3FF, as nicked from Windows Explorer (W11).

Affects: gList, gTable, gTree.

Background colour can also be specified on each text fragment returned by the gList DATA fetch function, and of course that will not extend all the way across like these do, plus on a gTree, SELFULL has some say in the matter (for selected but not hover).
BGSEL (but not BGHOV) can also be set on a gTree (only) as {fg,bg} as per gDrawText (there is no separate FGSEL attribute).
CANFOCUS Enables the focus traversal of the control.
Defaults: true for gButton, gCheckbox, gDatePick, gDropDown, gList, gSlider, gSpin, gText, gTree,
               false for gCanvas, gGraph, gLabel, gProgressBar, gTable, and gDialog, gH/Vbox, gFrame, gSplit, gTabs.
Obviously there should be no problem disabling anything in the first set, but no official support for enabling it on anything in the second set.
This setting has no influence whatsoever on an explicit gSetFocus() call.
CURSOR (non inheritable, write-only) Defines the element's cursor.

Value: Name of a cursor. Supports the following stock cursors: "RESIZE_NS", "RESIZE_WE", along with the aliases "NS_RESIZE", "WE_RESIZE", "EW_RESIZE", "RESIZE_EW", and lower-case versions, and that currently rather short list will be extended as and when needed, and only after suitable compatible cursors for WinAPI/GTK/JavaScript are identified. NULL restores the default cursor, usually an arrow. Cursors are handled very differently by each backend; it makes no sense either to try and decipher/re-constitute, or save the input just in case you want to read it back. One day in the probably quite distant future it should be possible to create a cursor from an xpm/gImage (and that would probably give both of us even more headaches were you allowed to read it back).

There is also a legacy console graphics cursor() routine, were you looking for that.
EXPAND Allows the element to expand, taking all or a fair share of any spare space inside the parent container.

It is a non inheritable attribute, but fairly obviously the overall behaviour depends on the expansion settings of both a container and its children. If expansion is disabled on a container then it gets a (fixed) minumum width/height that accomodates the largest of its children, none of the other children (including any sub-containers) being able to expand into anything beyond that. Should none of the children of a gH/Vbox expand in a given direction it can still expand but uses the SPACE attribute to distribute any excess and reposition the children appropriately. See also sole child.

Value: 
"NONE"[/"N"/"NO"/"FALSE"/'N'/'F'/false/0],
"BOTH"[/"B"/"YES"/"TRUE"/'B'/'Y'/'T'/true/1],
"HORIZONTAL"[/"H"/'H'/2], or
"VERTICAL"[/"V"/'V'/4] 

gGetAttribute(id,"EXPAND") returns one of the above (first) four strings, whereas
gGetInt(id,"EXPAND") returns 0/1/2/4: 
and 
bits 0b011!=0 indicates horizontal expansion
bits 0b101!=0 indicates vertical expansion.
(ie 0b000 is N-one, 0b001 is B-oth, 0b010 is H-only, 0b100 is V-only, it is not a standard bit-field, 3/5/6/7 are illegal|meaningless)

Defaults: "NONE" for gButton, gCheckbox, gDatePick, gDropDown, gLabel, gProgressBar, gSlider, gSpin, gText,
               "HORIZONTAL" for gTabs,
               "BOTH" for gCanvas, gGraph, gList, gTable, gTree, and [gDialog], gH/Vbox, gFrame, gSplit.

Note that the natural size of a gCanvas, and hence all the (next three) canvas-based controls, is just the size of one character, and a gTree, with a natural size of {0,0}, is even smaller. Hence should you disable expansion on any of them you will almost certainly need to specify an explicit SIZE, that is unless the sole child rules prohibit/override the attempt to disable expansion.

Affects: All elements, except menus and the gDialog itself.

See also: MARGIN, GAP, SPACE, MINSIZE, MAXSIZE, Layout Management.
FGCLR The interface element's forground colour, default TG_BLACK.
On a gCanvas and gImage this can be (transiently) overidden on each drawing operation instead.

Affects: Currently only gCanvas and gLabel

FONT (inherited, except for gImage) A string defining the font, in the format "<face>, <styles> <size>". Default: "Helvetica, 9".

Since face names are not a standard between Windows and GTK, a few are specially handled to improve application portability:

Recommended Windows GTK Description
Helvetica Arial Sans without serif, variable spacing
Courier Courier New Monospace with serif, fixed spacing
Times Times New Roman Serif with serif, variable spacing

All nine fonts listed above work on any backend. Face names are case-sensitive and must exactly match those shown above. Fonts are automatically cached/pooled to minimise resource usage - note that gGetAttribute() returns the post-mapped platform-specific version.

The font style can be omitted (aka normal) or a combination of bold and italic (case insensitive).
(underline and strikeout are yet to be properly supported).
[UPDATE: I may have got that the wrong way round: it may be fundamentally impossible on a font and easy-peasy (to emulate) on a canvas.]

Font size is in points (1/72 inch) or in pixels (using negative values).
[Technically optional, but no formal guarantee of what happens when omitted.]

Examples: 

    "Times, Bold 18"
    "Arial, 24" -- (no style)
    "Courier New, bold italic -30" -- (size in pixels)

Affects: All, except menus and pixbufs. See also JavaScript's elem.style.fontXxx and ctx.font settings.
Aside: for inheritance reasons, theGUI needs to take special care to use/set the font on the dialog elements rather than (/ as well as) on any (private) images it may or may not be using.
MARGIN (not inherited) Specifies the initial spacing around any content. See Layout Management.

Specified as between one and four integer values:
    m or {m} or "m" or "{m}" ⇒ {m,m,m,m},
    {tb,rl} or "tbxrl" or "{tb,rl}" ⇒ {tb,rl,tb,rl},
    {t,rl,b} or "txrlxb" or "{t,rl,b}" ⇒ {t,rl,b,rl}, or
    {t,r,b,l} or "txrxbxl" or "{t,r,b,l}" ⇒ {t,r,b,l} (unchanged).
The final four values correspond to top, right, bottom, and left margins respectively / clockwise.
Memory aid: margins spell trouble !  (Also note how {l, b} "hop" in the ficticious {4 ⇒ 3, 3 ⇒ 2} back-steps.)
Aside: although it may not look like it above, in practice eg 10x15 is easier to type/read/parse than {10,15}.

Default: 0 (no margin).
Returns: a sequence of length 1..4, or an atom, 0 if not set.
There is also a read-only MARGINS version which always returns a sequence of length 4, effectively defaulted to {0,0,0,0}.

Affects: All elements except gDialog, gSplit, and gMenu.
Actually, I tell a bit of a lie, MARGINS (only/read-only) can be used on a gDialog to obtain the window decoration sizes, as influenced by FULLSCREEN, HDR, MNZ, RST, and CLX, as well of course as potentially being different on different backends, and not that many programs will or should need it.
Instead of trying to set margins on a dialog, what you really want and in fact must do is set them on its sole child.

See also: GAP, EXPAND, SPACE, Layout Management.
Javascript also has an elem.style.margin, in case you were looking for that.
MINSIZE,
MAXSIZE 		(non inheritable) Specifies the element maximum/minimum size in pixels during the layout process.

See Layout Management for more details on sizes.

Value; "widthxheight", where width and height are integer values corresponding to the horizontal and vertical size, respectively, in pixels.

You can also set only one of the parameters by removing the other one and maintaining the separator "x", but this is equivalent of setting the other value to 0. For example: "x40" (height only = "0x40") or "40x" (width only = "40x0").

Affects: All, except menus.

On a gDialog also limits the interactive resize of the dialog, but does not override any explicit SIZE settings.

Aside: 
A bug in the MAXSIZE handling on Windows was fixed (way back in 2013) in 2.24.23, but I'm using 2.24.10.
Works fine on non-dialogs and in my newer GTK3.dlls, and should never have affected any Linux version.

See Also: SIZE
TIP (non inheritable, write-only) Text to be shown when the mouse is hovered over the element.

Value: Text.
TITLE The textual title/description of the interface element, often used to modify otherwise static text, such as that on a gButton, gCheckbox, gDialog, gFrame, gLabel, and maybe more. In most cases (except gDialog) the natural size is made large enough to include all the text in the selected font, even using multiple lines, plus things like the button/frame borders or the check box, if any.

There is a completely unrelated GTITLE attribute on a gGraph, similar things are handled quite differently on gTable and gTabs (see links for details), and the static text on menus is handled by gMenuSetAttribute() rather than this routine.

Value: Text.

Default: ""

Notes
The '\n' character is usually accepted for line change (except for menus, but they're not handled here anyway).
The "&" character can be used to define a mnemonic, use "&&" to show the "&" character itself.
If a mnemonic is defined such as "&Close" then the 'C' is underlined and pressing <Alt C> activates the control.
In some cases/platforms/backends the underline only appears while the Alt key is being held down.
You certainly shouldn't even be attempting to put mnemonics or line breaks on (say) a gDialog or gFrame title.

An automatic mapping of "&&","&","_" to "&","_","__" occurs under GTK, since that does mnemonics differently.

Mnemonics are not formally supported under pwa/p2js: I've done what I can, &<x> is replaced with <u><x></u> and elem.accesskey is set, && is replaced with &amp;, and \n is replaced with <br>. Beyond that, it is down to your operating system and browser, and I can tell you they will all be different. Alt (aka Options) and key, sometimes plus Ctrl, or Shift, or both, normally activates the control, or at least sets focus on it. Should your browser treat (say) Alt P as "start printing", I am not about to, and in fact it would be extremely foolish of me to even try and stop it, system-wide, but you might have some success with a KEY handler.

Be warned there is no guarantee of consistency of retrieved titles on different backends - WinAPI may return "&post", GTK "_post", and p2js "post", but at least match("post",title) w/should always be non-zero. Replacing a title with a modifed retrieved one is a whole different kettle of fish, and I'm not even going there.

Aside: I am not sure what this ever meant, and I've not deliberately put anything like this in, but I shall reproduce the following quote from the original IUP documentation: “For the benefit of GTK, should you define a mnemonic using "&" and the string has an underscore, then make sure that the mnemonic comes before the underscore”. I suspect but cannot be sure it is an outdated remark, from before "_"->"__", or perhaps some subtle buglette of IUP that I have hopefully not accidentally managed to replicate.

Note that gRedraw() may be required after changing the title for all characters to be shown.

Affects: All elements with an associated text.

See Also: FONT. Note that JavaScript's elem.title implements TIP.
USER_DATA A general storage slot: anything and everything which might be useful, especially in a handler routine, should and can safely be stored here. Can be set directly via the optional fourth parameter of a gTimer() on creation, as for other controls it should be set independently. Fairly obviously a leading '=' invocation of [inline] gSetAttributes() would be needed to set a non-string value. One word of caution: to avoid p2js violations an explicit nullification after fetch may be rqd/save a deep_copy(), and there is [as yet] no facility to update, say, USER_DATA[7] in-situ. Default: NULL.
USER_SIZE (implicit) Most applications should just do everything via SIZE, and pretend that USER_SIZE does not even exist.
[There is indeed a valid argument this should not be made public, then, but that would probably confuse things even more.]

Setting the SIZE attribute actually sets this instead, whereas reading that returns the actual size, though obviously reading this gets whatever was explicitly set (via SIZE or USER_SIZE), or {0,0} if not ever set, irrespective of whatever real size it might end up with.

Setting the size of an element to say 150x75 then placing it in a container with a size of 175x50 leads to formally undefined results, which may even differ between different backends. The layout manager has to deal with those sort of conflicts all the time, in particular the natural size of most containers is tiny or even zero, and it does not raise any special error when a conflict has arisen from user-specified sizes rather than natural sizes, instead it simply overrides them as best it can/as it sees fit. It tries hardest of all to honour the dialog size, except when outside any MINSIZE/MAXSIZE, since that is the one thing every end user can, unless explicitly disabled, physically drag about on-screen at run-time or effectively manage the same by maximising or restoring the window. Such actions implicitly stomp all over (the dialog's) USER_SIZE, except for retaining the restore size.

Returns: integer {width,height}. For more details see Layout Management.

See Also: SIZE
VALUE The current content or state of a control, see individual entries for details.

Affects: gCheckbox, gDatePick, gDropDown, gProgressBar, gSlider, gSpin, gText.

Should you be looking for the value() builtin, obviously click on that link.
VISIBLE (not inherited, but parent-dependant) Shows or hides the element.

Value: true/"YES" (visible), false/"NO" (hidden). Default: true.

Notes: An interface element is only visible when all parents up to and including the dialog are also visible. Setting this on a container does not (actually) override the setting of any children, in order for everything to be restored as expected, should said container later be made visible again, however fetching it performs the appropriate parent-scan for you. Be warned this may return true when the containing dialog is MINIMISED, so that need checking that as well, to determine whether a control is actually on-screen.

Affects: All controls that have visual representation.
Technicalia While pGUI had all manner of convoluted restrictions on attributes, such as v being atom_string or the dire distinction between IupSetAttribute() and IupSetStrAttribute(), largely because C and C-based languages simply cannot distinguish between an integer and a string (aka char*) 1, this one routine (gSetAttribute) covers and replaces at least 5 and probably more pGUI routines, even technically things such as IupSetInt() - while there is a gSetInt() as noted above, you can pass the exact same parameters direct to gSetAttribute().

As well as not replicating that particular distinction, theGUI either actions things immediately or makes a (reference counted) copy, so (unlike pGUI) there is no danger here of a value "vanishing" before it can be used.

Appropriate care may still sometimes be required with some values, such as a raw memory address, which must fairly obviously remain valid for the entire lifetime of the interface element, or at least until all copies/references to it are replaced.

1 Of course Phix manages that via a combination of limiting integers to 31/63 bits, storing 32/64 bit "integers" in an atom, so it can do some bit-twiddling to distinguish an integer from a shifted ref address, where it can find another byte to distinguish atom/string/dword_sequence, and not having or depending on a pointer type, rather you store pointers in an atom and it becomes your sole responsibility to use them as such, for instance the language itself would not raise an eyebrow should you ask for the square root of a pointer, or for that matter use the pointer itself when you should be using whatever it points at. One Phix would catch is char* vs string, which are of course literally the exact same thing in C. Swings and roundabouts, I guess, the simple point being made is just that C and Phix have some rather fundamental differences, which certainly bleed through a bit here.

In practice id is defined as an integer, but verified to be the declared gdx type above once any gImage/gPixbuf have been dealt with.
