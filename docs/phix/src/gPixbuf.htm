gPixbuf

Definition: include theGUI.e

gdp pid = gPixbuf(integer w, h, atom bgclr=TG_WHITE)
-- or --
 gdp pixbuf = gPixbuf_from_Image(gdm img) -- [NB not fast, aim for once-only]
Description: Creates a platform independent pixel buffer that can later be converted to a gImage.

w: width in pixels.
h: height in pixels.
bgclr: the initial colour of all pixels.
img: if created via gImage_from_XPM(), bPixbuf must have been true. The img is automatically destroyed.

Similar to gImage, a gPixbuf is not an interface element and cannot be added to a layout hierarchy.
A gCanvas is regularly blanked, and usually resizeable (by the end user), whereas similar would only ever happen to a gPixbuf when deliberate and explicit (in the application source code).

To get images/text onto a pixbuf, you must (first convert it to an image, then) draw them on an image and convert that (back) to a pixbuf. See the feature summary of gCanvas/gImage/gPixbuf below for more details. As noted there, you should expect such conversions to take about ⅛s, or at most four image⇔pixbuf round trips per second.
pwa/p2js: Supported, but note from/take pixels is rather slow, ~⅛s vs. <1e-6s on desktop/Phix.
See Also: gCanvas, gImage, gImage_from_XPM, gImage_from_Pixbuf

The file demo\theGUI\gPixbuf.exw is pretty much same as gCanvas.exw but reworked to use gPixbuf, and test a few things on gImages, plus a rack of timing tests, and no ugly bresenham edges on WinAPI.

There are separate cases where I would recommend a gCanvas, being simple/static, or a gPixbuf for deep analysis or any pixel-level work, or OpenGL or similar, for fast/full-on gaming.

Notes: There is no particular advantage of gPixbuf(w,h,clr,false) over repeat(repeat(clr,w),h) except for a clearer and more obvious/intuitive statement of intent (whereas bOwned:=true returns a -ve integer index).

A gImage passed to gPixbuf_from_Image() must be one created by gImage_from_XPM() with bPixbuf set to true, or any created via gImage_from_Pixbuf() or gImage(), ie on those latter two that flag is effectively automatically true.
Deprecated: While very fast on desktop/Phix, a serious bottleneck in theGUI.js has made me question their merit.
However there is no point actually removing them until all rosettacode examples have been converted to use theGUI.
They permit the use of gDrawLine..gDrawRect on a pixbuf, which quite frankly isn't what a pixbuf is for, and besides you can create a gImage, do those (and gDrawText) on that, then convert it to a pixbuf, which if nothing else gives you the same performance characteristics on desktop/Phix and pwa/p2js.

gdp/*(/int)*/ pid = gPixbuf_from_Pixels(gdp/*(/seq)*/ pixels) -- (pid & pixels usually same var)
-- or --
 gdp/*(/seq)*/ pixels = gPixbuf_take_Pixels(gdp/*(/int)*/ pid) -- (ditto)

pixels: as per a potentially modified deep_copy(repeat(repeat(bgclr,w),h)).
pid: (as param) take ownership of the pixels and destroy/recycle the internal pid entry.
Technically both gPixbuf() and gPixbuf_from_Image() also have bool bOwned=false as their final parameter: by default a 2D sequence (app-owned) is returned, when true gdp result is -ve int (theGUI-owned) appropriate for gDrawLine(), etc.

gPixbuf_from_Pixels() transfers ownership of a 2D pixel sequence to theGUI (stores it in an internal list).
gPixbuf_take_Pixels() transfers ownership from theGUI to the application (deletes it from an internal list).
Note that gPixbuf_from_Pixels() must be passed a sequence with a reference count of one: should it be a constant then [an inline] deep_copy() is needed, should you have constructed it yourself the best bet is to return it from an inline function call. One other trick you might use is p = gPixbuf_from_Pixels(p) whereby the [local] input p is your hand-built 2d sequence of pixels and the output p (ie same var) will be an even -ve gdp, which uses automatic pbr to deal with that pesky reference count (luck willing), and in fact the gdp type is deliberately designed to support precisely that. You need not worry about ownership (or reference counts) at all when passing a sequence to gImage_from_Pixbuf().
The pixels must however be exclusively owned by theGUI before gDrawLine() (etc) can be used, and gDrawImage() and gDrawText() only work on a gCanvas or gImage, not a gPixbuf, though (only if you really have to) you could do them on a gImage (which may be created from a gPixbuf, as next) and convert that (back) into a gPixbuf.
Attributes: (also deprecated)
FGCLR default: TG_BLACK,
LINESTYLE default: TG_CONTINUOUS,
LINEWIDTH default: 1,
USER_DATA default: NULL,
SIZE (read-only).

Note that (mainly because of ownership issues) SIZE is read-only and like a gImage there is no USER_SIZE.
Technically you can also set BGCLR, but it is not actually used (or sensibly defaulted) anywhere.
While NULL (===TG_BLACK) effectively resets FGCLR/USER_DATA, others may need their defaults deliberately restoring.
Lastly note that gSetAttributes (the with-an-s one) is not applicable to a gPixbuf; instead do them one-at-a-time.
Technicalia Negative widths and heights trigger an error as expected, however a height of zero returns {{}}, the same as w=0, h=1 does, which means that length(pixbuf[1]) is always a valid method to check the width and that way simplifies the internals a bit. To be crystal clear, width is ignored when height is zero.

The question must arise: Why do we have gCanvas, gImage, and gPixbuf?

Essentially a gCanvas is the thing you can actually see on screen, at the very far end of the GPU pipeline.
When the provided primitives suffice, or you can leverage (say) opengl to take full advantage of all that lovely fast GPU hardware, then you certainly should. However direct access is helpful/essential for more detailed image analysis and processing such as image sharpening or canny edge detection, and then performance becomes an issue. The GPU pipeline is (probably) quite a bit slower than you might think: it is nice and wide so you can tip lorry-loads of instructions in that run in parallel, but certainly not optimised for retrieval, especially should it need flushing first. [At the risk of confusing you, since switching to emulated controls even a gCanvas is emulated with a gImage, and the only real "canvas" is now the whole client area of a (headerless) dialog, or conversely you could say everything except a gDialog is technically a gCanvas, albeit in an associated image sense only.]

A gPixbuf is an application-side version of the pixels that is much more suitable for detailed analysis and direct manipulation, but in a format the hardware and backend do not understand at all (though theGUI itself does).

A gImage is half way between, in a backend-specific format. Theoretically pixel-level operations could be performed on them, but slower than and no real advantage over using a pixbuf. You would certainly not want to have to convert a gPixbuf or XPM string to (the equivalent of) a gImage every time, say, you displayed an icon on a menu item - as would almost certainly be needed did the gdm type not exist. A gImage also offers a way to get text and images into a gPixbuf, albeit with the obvious performance penalties such conversions entail.

Every platform (even GTK!) has highly optimised native methods to display or transfer a gImage onto a gCanvas.
The slowest part is going to be converting between images and pixbufs, so focus on doing that as few times as possible.

T indicates a canvas is target-only, whereas an image can be source or target (and pixbuf neither).
You can convert images to pixbufs and vice versa to work around the N on those lines.
Pixel-level working has to occur in a pixbuf, then -> image -> canvas to appear on-screen,
with any text and/or images only added at the image (or canvas) stage, and ideally with the
conversion of pixbuf->image performed just the once only.

A key point here is that a gCanvas supports all the "plain" features necessary for (say) gGraph, gList, and gTable, including things like the fake scrollbars.
