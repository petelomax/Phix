gDatePick

Definition: include theGUI.e

gdx id = gDatePick([rtn value_changed=NULL,] [object ymd=NULL,] string attributes="", dword_seq args={})
Description: Creates a date editing interface element, which can drop-down a calendar for selecting a date.

value_changed: (optional) a procedure invoked when the contents change, see VALUE_CHANGED.
ymd: (optional) allows the initial date to be specified, as integer {year,month,day[,...]} [any rest ignored].
For more information on the attributes and args parameters see gSetAttributes().
This is a paranormalised function. (see technicalia)

Returns the identifier of the created element.
pwa/p2js: Supported.
Example: 
-- demo\theGUI\gDatePick.exw (less GTK hacks)
include theGUI.e

gdx dp = gHbox({gDatePick()},"MARGIN=13x13"),
   dlg = gDialog(dp,`gDatePick`,`SIZE=240x80`)
gSetAttribute(dp,`RANGE`,{{2025,11,5},{2025,11,15}})
gMainLoop(dlg)

Notes: DEV: fixme:
There is no fancy editing as yet: a native Windows control has three mini-spin controls within it.
Attributes:
RANGE Specify the permitted date range, default {{1800,1,1},{9999,12,31}} (see technicalia). If the start/end are integer they are treated as day offsets to the current date, so for example {-180,0} is the past 180 days (~6 months).

A perhaps too-subtle point is the range determines what is hoverable, and that in turn determines what is clickable. You shouldn't see any hover effects or be able to click on anything (wholly) outside the specified range, and of course any mouse navigation limits apply exactly the same to keyboard navigation (specifically left/right and [page] up/down).
VALUE The current date, timedate-compatible, eg {2025,12,31,0,0,0}, but does not quibble when set with a length-3 sequence, and returns a length-6 sequence with hms of 0s so that, (eg) format_timedate() doesn't either. Should you have a string such as "31/12/2025" you are expected to parse_date_string() it yourself: apart from such strings potentially being in umpteen different formats, when a gDatePick gets a dword-sequence it updates both the always-visible gText and the embedded gCalendar drop-down, when it gets a string it assumes it is from the embedded gCalendar, strictly in DD/MM/YYYY format that needs no further working, and only updates the gText. Hence trying to set a string on a gCalendar is bluntly ignored, and/or raises an error, that is rather than having counter-intuitive control-type-and-string-format-dependent conditionals or needing two separate handlers.
As just suggested, a gDatePick (currently) only deals with/displays/supports DD/MM/YYYY format strings.
also ACTIVE, CANFOCUS, ? DAYMODE, TIP, VISIBLE.
Handlers: Set using eg gSetHandler(datepick,"VALUE_CHANGED",value_changed), or specify value_changed at creation time
VALUE_CHANGED Called after the value was interactively changed by the user.

procedure value_changed([gdx id,] timedate dt)
id: (optional) identifier of the element that activated the event.
dt: equivalent to timedate dt = gGetAttribute(id,"VALUE") as the first line.
dt will be a length-6 sequence with the ymd followed by hms of 0s.
also KEY: All common handlers are supported.
Technicalia As this is a paranormalised function, and with no expectation that you should memorise this lot, except maybe "rdaa", you can assume that all of the following are potentially perfectly valid, with the fairly obvious restrictions that any parameters must be in strict rdaa-order, and no args without attributes: 

                        gDatePick()
                        gDatePick(attributes)
                        gDatePick(attributes, args)
                        gDatePick(ymd)
                        gDatePick(ymd, attributes)
                        gDatePick(ymd, attributes, args)
                        gDatePick(rid)
                        gDatePick(rid, attributes)
                        gDatePick(rid, attributes, args)
                        gDatePick(rid, ymd)
                        gDatePick(rid, ymd, attributes)
                        gDatePick(rid, ymd, attributes, args)   -- (the full set)

Note that in practice rid and ymd are actually defined as object, and attributes as a sequence rather than a string, but verified to be of the documented types, after being repositioned via theGUI/paranormalise_rdaa().

Technically a gDatePick/gCalendar does not formally support dates prior to 1752, the introduction of the Gregorian calendar, however that limitation is not particularly strictly applied. Adoption of the Gregorian calendar varies by country, from 1582 to 2016, and I've done nothing about that. The following notes really date from prior to/during the implementation of the RANGE attribute and are more concerned with pedantically documenting some current and somewhat obscure behaviour than imposing any "hard" restrictions. Dates prior to 0 CE (aka 0 AD) broke some internal maths, and were therefore prohibited, plus all the '-' were not exactly pretty anyway. All months prior to 1752 are generally and somewhat flakily deemed to have 31 days, also format_timedate() produces "Anyday" when asked for a weekday of 31/12/1751 and earlier, and since Anyday is 8, (usually) show up as 25..31 with the 1st always on a "Monday", which kinda messes up the neater scrolling of >= 1752. In particular, the last few days of the previous month may be inconsistently displayed, and clicking on them lead to inconsistent results (pre-1752).

The bottom line is the start of RANGE can be as far back as 0/1/1, but not without some glitches, which given the limited practicality of such dates are left as an exercise for someone else to solve, as are -ve years, and that in Great Britain and its colonies, in 1752 the 2nd September was immediately followed by the 14th September.

Apparently there is also some small confusion over an "extra missing day" in 1800, due to it being a leap year in the Julian calendar but not in the Gregorian calendar, which I don't quite fully grasp (as in “why not 1900 too, then?”), hence the slightly strange start year of 1800 for RANGE.

Be advised so that scrolling months can resurrect Jan 31st, the control may internally hold the 31st of February, and in fact (as above) deliberately/visibly does so for any year prior to 1752, likewise it may sometimes internally hold the 31st July even when the upper range limit is 15th July, but otherwise that should not display/leak, nor, again, does it throw a hissy fit should you specify such things. Even immediately after setting 31/2 (for any year >=1752) you can expect to get the 28/29th back (at least, that is, as long as nothing in your application or anything it or the datepick invokes decides to suddenly throw an error).

Also, be aware that I have tried my level best to make navigation (esp keyboard) as natural as possible, but certain RANGE settings can introduce little niggles that are almost impossible to document fully, aka if you run into a wall (ie range limit), don't expect zero dents.
