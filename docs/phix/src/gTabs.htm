gTabs

Definition: include theGUI.e

gdx id = gTabs(object tabitems, [rtn tabchanged=NULL,] string attributes="", dword_seq args={})
Description: Creates a single line tab control with titles and/or images. Also known as a notebook in some systems.

tabitems: a static sequence of titles|images[,colours,style] for each tab, or a TABITEMS function.
tabchanged: an optional procedure invoked whenever the user changes the selected tab, see TABCHANGED below.
For more information on the attributes and args parameters see gSetAttributes().
This is a paranormalised function (see technicalia).

Returns: the identifier of the created element.
pwa/p2js: Supported. (Out of date, but should be a fairly easy chore.)
See Also: gImage, gCanvas
Example: 
-- demo\theGUI\gTabs.exw
include theGUI.e
constant xpm = ...(code omitted for brevity)
gdx tabs = gTabs({{"First",xpm},"Second","Third","Fourth"}),
    dlg = gDialog(gVbox({tabs}),"gTabs","SIZE=240x80")
gMainLoop(dlg)

Notes: A gTabs is not a container, the application is responsible for managing visibility, size, etc of any associated controls.
theEditor uses a single gList below a gTabs and replaces the content of that when switched. Other applications might use a gHbox or gVbox (assuming only one immediate child would ever be VISIBLE at the toplevel it matters not which) with different contents for each tab item, but that is down to the application, rather than the tab control.

There is no natural width on a gTabs (apart from the 1-char canvas thing, typically 15px). Either it is a fixed width, which you should explicitly provide, or the dialog should be much more dependent on whatever you have put underneath it, with the gTabs expanding to match/cover that. I might even say that if non-static tabs are not allowed to expand full width, or shrink way below some mythical natural size, your'e missing out on half the fun. The natural height is 24.

Each element of tabitems, or the returns from a TABITEMS function, can be a plain string or
a sequence of the form {string[,image][,bgclr[,fgclr[,style]]]}.

A string title must always be provided, even if it is "" (the empty string),
images can be any width and are centred vertically based on a 16px height model (ie +8-floor(h/2)),
a background colour (eg TG_RED, "RED", or #FF0000) applies around and between images/text,
a foreground colour applies to the title text only, as does
style, which can be TG_NORMAL, TG_BOLD, TG_ITALIC, TG_BOLDITALIC, or a full FONT string.

Note that changing any of these on the fly or for the currently selected tab constitutes a dynamic tabslist, which mandates the use of a TABITEMS function.

Although you should be able to mimic the appearance and behaviour of such, there is no formal way to disable individual tabitems. There is no way, and never will be, to change the visibility of individual tabitems, instead you are expected to physically remove and re-insert entries yourself. You can mimic the appearance of mnemonics using the standard & prefix, however you would have to write a special custom KEY handler yourself to actually switch to the appropriate tabs (or not if your'e pretending they're disabled), which would be a function that has an alt parameter/all 5, and lets any other mnemonics though unhindered.
Attributes:
EXPAND (non inheritable) The default value is "HORIZONTAL" (or 2 from gGetInt), and vertical expansion is not supported.
POSN (non inheritable) Retrieves/changes the active tab by its position, starting at 1. Default: 1.
The legacy gCbox also utilises a POSN attribute, in case you were looking for that.
SHOWCLOSE (non inheritable) display a close button on each tab. Default value: "NO"/false.
An error occurs when setting this to true if tabitems is defined as a static sequence, rather than a TABITEMS function (since closing an entry in a static list means it's not static), or if neither a TABCLICK nor a TABCLOSED handler has been defined.
TABCOUNT (read-only, non inheritable) returns the number of tabs, ie the length of a static tabitems sequence or matching the return of tabitems(0). Included primarily to make the internal code more self-documenting/debuggable.
also ACTIVE, FONT, MINSIZE, MAXSIZE, SIZE, TIP, VISIBLE.
Handlers: Set using eg gSetHandler(tabs,"TABITEMS",tabitems), or 2 out of 5 can be specified at creation time
TABITEMS (same as first param on creation, when that's a function) Required whenever the set of tabs is not static.

sequence tabslist = {"one","two","three"} -- assume this gets modified

function tabitems([gdx id,] integer item)
    return iff(item=0 ? length(tabslist) : tabslist[item])
end function

id: (optional) identifier of the element that activated the event.
item: index of the desired item, or 0 to get the total number of tabs.

Return must be an integer when item is 0, otherwise a plain string or
sequence of the form {string[,image][,bgclr[,fgclr[,style]]]}.
TABCLICK Note the standard CLICK handler is used internally by this control and should not be overidden, however it will forward all such events to a TABCLICK handler first, if one is defined, and as a bonus indicates which tab (if any) it occurred on, with -2 for none, -1 for left overflow and 0 for right, before POSN is updated, so that attribute can still be used within this handler to determine the current/prior one, and whether a close button has been clicked.

function tabclick(gdx id, sequence status, integer x, y, posn, bool bClose)
See CLICK for more details, note however this must be a function with all six parameters. If status does not begin with "LS" (ie a left single click) the internal handler will do nothing (nor invoke any other handlers bar this).

Returns of TG_IGNORE prevent any further processing (which is merely to update POSN as required and invoke the appropriate TABCHANGED and/or TABCLOSED when they exist and redraw) and hence effectively disable the control (or just the close action), TG_CONTINUE and TG_DEFAULT behave as if this handler did not exist, with all other values triggering an error, including TG_CLOSE.

A single TABCLICK handler can replace/do all the work needed for both TABCHANGED and TABCLOSED handlers, but of course when you have no need to prohibit things or don't need both, the latter are a tiny smidge simpler.
TABCHANGED (same as second param on creation, when provided as such) Handler called when(/after) the user changes the active tab.

procedure tabchanged([gdx id,] integer old_posn, new_posn)
id: (optional) identifier of the element that activated the event.
old_posn, new_posn: integers 1..TABCOUNT

It is entirely the application's responsibility to hide/show/update any associated controls.
Note this is decidedly not a "before change with the option to prohibit" event: you would have to mimic such yourself, and perhaps explicitly [gSetAttribute(tabs,`>a href="#POSN">POSN`,?)] switch back to some incomplete tab.
TABCLOSED Handler called when the user clicks on the close button. Called only when SHOWCLOSE is true.

procedure tabclosed([gdx id,] integer posn)
id: (optional) identifier of the element that activated the event.
posn: the tab position

It is entirely the application's responsibility to maintain the dynamic tabitems list, which it owns.
TABMOUSE In a similar fashion to CLICK/TABCLICK the standard MOUSEMOVE handler is also used internally by this control and should not be overidden (otherwise hover effects may/will stop working), however it will forward all such events to a TABMOUSE handler first, if one is defined, and as a bonus indicate which tab (if any) the mouse cursor is over, with -2 for none, -1 for left overflow and 0 for right, and whether it is on a close button.

procedure tabmouse(gdx id, integer x, y, posn, bool bClose[, left,middle,right[, ctrl,shift,alt]])
See MOUSEMOVE for more details, note however this must be a procedure with five, eight, or all eleven parameters. You might need this, for instance, to change the tooltip on each tab.
Technicalia As this is a paranormalised function, and with no expectation that you should attempt to memorise this lot, except maybe "traa", you can assume that all of the following are potentially perfectly valid, with the fairly obvious restrictions that any parameters must be in strict traa-order, and no args without attributes: 

                        gTabs(tabitems)
                        gTabs(tabitems, attributes)
                        gTabs(tabitems, attributes, args)
                        gTabs(tabitems, tabchanged)
                        gTabs(tabitems, tabchanged, attributes)
                        gTabs(tabitems, tabchanged, attributes, args)  -- (the full set)

Note that in practice tabchanged is actually defined as object, and attributes as sequence rather than string, but manually verified to be of the documented types, after being repositioned via theGUI/paranormalise_raa().

which immediately raised issues with figuring out the native sizes, and had lots of little behavioural quirks between backends, and then I looked at adding an optional close button on each tab, and immediately decided it would be much simpler to re-implement the whole thing myself, so behind the scenes it is just a gCanvas, of height 24 that expands to fill the width of the parent container, with all drawing and mouse interaction directly implemented in hll code. I'll toot my own horn here and recommend that anyone remotely interested in such matters take a good gander at tg_redraw_tabs() in theGUI.e: it packs a serious punch in a mere 120 lines of pretty elegantly written code, imnsho! Note the standard sole child exception applies, and especially when building a prototype you may notice some faint strange blotchy squares being drawn below a gTabs, as it dithers between that rule and "height=24", unless it is in an appropriate gHbox or gVbox that absorbs any excess space.

Current limitations (all fairly minor) include:
Windows (11) only look-and-feel, but it should be pretty straighforward to knock up something which better marries in with other backends (I'd need some screenshots, ideally with and without overflow[but see below] and close buttons).
I didn't bother to implement any of the very subtle gradients in the tab edges (well done if you can see them above).
I should perhaps also mention that WinAPI makes some weird and subtle adjustments to the results from (the equivalent of) gGetTextExtent() that I never figured out, so it uses a slightly different and simpler spacing model. Also, (as I used it) WinAPI uses an imagelist, forcing all images to be the same wxh, but the replacement does not.
No focus or keyboard handling, as yet, and tbh I am not sure that I ever want to bother with those (except mnemonics).
Single line tab only, not that I think it would be spectacularly difficult to do multiline.
There is no '+' button but it should be easy to add one [only when the right overflow button is not showing?], and presumably have it simply invoke tabitems(-1), and possibly bump left-overflow when needed. [PLUSBTN?]
I reinvented the left/right overflow buttons so that (when needed) left is always on the left and right is always on the right, since I think that is more intuitive, looks smarter, and was in fact easier to implement.
In accordance with every version of this control I have ever seen, should a single pixel overlap the edge (or the overflow button) it displays no part of the rightmost tab, though I'm not entirely convinced that is ideal.
On the plus side, it drags in any left-hidden tabs much better/earlier when [none on the right and] resizing the containing window, at least on WinAPI (I gave up with the native controls before I could make the GTK versions "shrinkable"). However it does not [as yet] hide more on the left(/right) to ensure the current/active tab stays visible, when shrinking. Then again you can, as you probably should be able to, dribble said off-screen, either way, using the overflow buttons.
